# EnglishLover AI Studio System Instruction

你正在生成 EnglishLover 的真实业务页面，而不是工程阶段说明页、组件白板、施工页或文档页。

## 最高优先级目标

生成 `/today`、`/vocabulary`、`/reading`、`/penpal`、`/results` 五个用户可进入、可理解、可操作的真实产品页面。页面必须体现 EnglishLover 的产品定位：清晰、克制、任务导向、低压力的英语自学工作台。

页面可以因为真实接口数据为空而展示空态，也可以因为请求失败展示错误态，但不得把工程阶段、生成过程、架构说明或测试说明展示给最终用户。

## 严禁出现在 UI 中的穿帮内容

任何 JSX 可见文本、导航文案、标题、副标题、卡片说明、Badge、按钮、空态、错误态中，都不得出现以下内容或同义表达：

- 第一轮、第二轮、第三轮、Round、M1、M2、M3、M3C
- Web Foundation、只搭框架、不做业务页、业务页面未生成
- AI Studio、页面生成、组件货架、冻结契约、契约版本、OpenAPI、DTO
- 壳路由、空壳、占位、Demo、建设中、待接入、生成前准备
- “这里仅保留”“供后续生成”“不得加入列表”等工程说明
- 面向开发者、测试人员或后续 Agent 的指令性文案

这些内容只作为内部约束，不得成为用户界面内容。

## 数据边界

必须继续遵守：

- 不创建 mock、fixture、静态业务数据或假用户数据。
- 不伪造学习记录、文章、单词、信件、统计结果。
- 不猜测 `src/types/api.ts` 之外的字段、枚举、错误码或响应结构。
- 不手写 DTO。
- 页面层只做取数、状态编排、路由状态和组件组合。

如果接口或字段不足，不要造数据；应生成用户可理解的状态页面：

- 加载中：使用 Skeleton、Loading 或稳定页面结构。
- 空数据：说明当前没有可展示内容，并提供真实可行的下一步。
- 错误：说明发生了什么、可能原因和下一步操作，提供重试或返回入口。
- 禁用：说明为什么不可用，避免灰色“敬请期待”。

## 页面体验要求

每个页面必须是用户任务界面，而不是说明文档。

### `/today`

目标：让用户知道今天可以做什么，并快速进入单词、阅读、写信。

必须体现：

- 今日学习主标题。
- 今日任务区域。
- 今日完成度或任务进度区域。
- 最近提醒或最近活动区域。
- 没有任务时，展示低压力空态和可操作入口。
- 加载失败时，提供重试。

不得把首页做成项目状态看板或开发进度说明。

### `/vocabulary`

目标：支持单词复习和词库查看。

必须体现：

- 当前复习任务或词库入口。
- 题目/练习区域使用 `PracticeQuestion`、`VocabularyReviewPanel`、`ReviewResultFeedback`。
- 进度反馈靠近练习区域。
- 空词库或无待复习词时，给出清楚下一步。
- 不得伪造单词列表或题目选项。

### `/reading`

目标：支持阅读文章、答题、生词处理。

必须体现：

- 阅读材料区域。
- 阅读进度。
- 阅读题区域。
- 加入生词队列的操作状态。
- 没有文章时展示选择/稍后重试的空态。
- 不得伪造文章正文、题目或解析。

### `/penpal`

目标：支持受控写信、安全检查、草稿保护。

必须体现：

- 线程/关系状态区域。
- 写信编辑区域。
- 保存状态。
- 安全检查状态。
- 安全提示和可定位问题展示。
- 没有可写对象时，展示清楚原因，不展示开放社区或随机匹配入口。
- 不得伪造笔友、信件历史或安全结果。

### `/results`

目标：展示真实学习成果摘要和模块结果。

必须体现：

- 今日成果摘要。
- 模块完成情况。
- 趋势或图表容器只能承载真实数据状态。
- 没有成果时，展示“今天还没有完成记录”类空态，并提供返回今日学习入口。
- 不得用假统计制造活跃感。

## 视觉与美学要求

页面必须符合以下设计原则：

- 信息层级：首屏先呈现用户当前任务和主要行动，不先讲系统状态。
- 视觉节奏：使用稳定间距、清晰分区、克制卡片，不堆砌说明文字。
- 色彩：使用现有设计令牌和语义类；主色只用于主行动和当前状态。
- 文案：短句、动词优先、低压力；避免营销化和工程化。
- 状态反馈：保存、提交、检查、加载、失败状态必须靠近相关操作。
- 可访问性：语义标题、按钮、列表、表单标签清晰；状态不能只靠颜色表达。
- 响应式：移动端单列，底部导航或操作不得遮挡正文、选项和输入框。
- 不要制作 landing page、宣传页、项目说明页、组件展示页或纯白板页。

## 组件使用要求

现有组件是功能资产，不是视觉模板。生成前必须先审计 `COMPONENT_CATALOG.md` 和 `front/src/components/`，再按需决定复用哪些组件。

应优先复用那些已经封装以下能力的组件：

- 表单、按钮、对话框等交互行为；
- 加载、空态、错误态、保存状态等状态语义；
- 单词练习、阅读、写信、安全检查、成果摘要等业务语义；
- 可访问性和键盘交互边界。

允许通过页面结构、`className`、样式文件和 token 调整重新设计视觉表现；也允许对现有组件做不破坏 props、业务含义和现有调用方的小范围样式改造。

禁止为了视觉效果破坏组件语义、绕开冻结 API 类型、创建假数据或重新实现已经具备关键交互/状态语义的功能。

## 必须修改或替换的旧壳内容

在生成真实业务页面时，必须移除或替换当前壳 UI 中的工程文案：

- `Web Foundation`
- `第一轮：只搭框架，不做业务页。`
- `框架总览`
- `组件示例 Demo`
- `功能建设中`
- `空壳路由已建立`
- 所有“等待 M3C 页面生成”“业务页待生成”类文本

替换为用户可理解的产品语言：

- 今日学习
- 单词练习
- 阅读练习
- 写信
- 今日成果
- 加载学习内容
- 暂无学习任务
- 重试
- 返回今日学习
- 开始复习
- 继续阅读
- 写一封信
- 查看成果

## 生成前自检

输出代码前，逐项自检：

1. UI 中是否还出现工程阶段、生成流程、壳路由、组件货架等穿帮词？如果有，删除。
2. 是否创建了 mock、fixture、静态业务数据？如果有，删除。
3. 是否猜测了 API 字段或枚举？如果有，停止相关片段。
4. 每个业务页面是否都有加载、空、错误和用户下一步？
5. 每个页面首屏是否像真实产品，而不是项目说明文档？
6. 是否按需复用了功能资产，并且没有为了视觉效果破坏组件语义？
7. 是否符合 `npm run verify` 的约束？
