## 2. 项目立项背景

英语学习通常包括词汇积累、阅读理解和书面表达等环节。对于普通学习者而言，学习过程中容易遇到以下问题：

- 单词记忆缺少持续复习，容易出现学完就忘的情况。
- 阅读练习材料分散，用户难以形成稳定训练流程。
- 写作练习缺少具体场景，用户不知道写什么、如何持续练。
- 学习记录不清晰，用户难以判断自己是否有进步。
- 部分学习工具功能较复杂，用户进入后需要较长时间适应。

因此，本项目拟建设一个完全免费的在线英语学习平台 Web 端，将单词记忆、阅读练习和笔友写信练习整合到同一平台中，为用户提供清晰、连续、低门槛的英语学习体验。

该项目的建设重点不在于堆叠大量功能，而是围绕用户真实学习过程，帮助用户完成：

```text
开始学习 -> 完成练习 -> 获得反馈 -> 保存记录 -> 持续复习
```

## 3. 项目定位

### 3.1 产品定位

EnglishLover 是一个面向英语自主学习者的免费 Web 端学习平台，重点提供单词记忆、阅读练习和笔友写信练习功能，帮助用户完成词汇输入、语境理解和语言输出训练。

### 3.2 平台性质

本平台具有以下特点：

- **完全免费**：不设计付费课程、会员体系或商城功能。
- **Web 端优先**：用户通过浏览器访问，无需安装客户端。
- **学习任务导向**：围绕单词、阅读、写作三个核心任务组织页面。
- **轻量化使用**：降低注册、配置和操作门槛。
- **可持续扩展**：后续可扩展学习统计、复习计划、更多内容类型等功能。

### 3.3 项目非目标

为了控制范围，第一阶段明确以下内容不作为首版目标：

- 不建设付费课程系统。
- 不建设电商或会员支付系统。
- 不建设开放广场式大型社区；首版仅验证受控的一对一笔友写信闭环，可先使用小范围用户池或受控测试账号，不等同于开放社区上线。
- 不实现复杂实时聊天。
- 不在首版中引入过度复杂的推荐算法。
- 不以公开社交、排行榜或娱乐化装饰作为核心卖点。

## 4. 用户分析

### 4.1 目标用户

平台主要面向有英语学习需求的普通学习者，包括：

| 用户类型 | 用户特点 | 主要需求 |
|---|---|---|
| 基础词汇学习者 | 词汇量不足，容易遗忘 | 需要清晰的单词学习和复习流程 |
| 阅读能力提升者 | 希望通过文章理解英语 | 需要分级阅读材料和理解练习 |
| 写作练习用户 | 英语表达机会较少，缺少真实交流对象 | 需要安全的笔友通信、写作支架和历史记录 |
| 自主学习用户 | 希望长期坚持学习 | 需要学习进度反馈和持续学习入口 |

### 4.2 用户痛点

结合英语学习过程，可以归纳出以下用户痛点：

1. **学习入口分散**
   单词、阅读、写作往往分布在不同工具或资料中，用户需要频繁切换。

2. **学习过程缺少连续性**
   用户完成一次练习后，平台如果没有记录和反馈，下次学习时很难接着进行。

3. **复习机制不足**
   单词学习如果只停留在“看一遍”，很难形成长期记忆。

4. **写作缺少真实交流对象**
   很多学习者想练习英文写作，但缺少安全、稳定、低压力的真实交流对象，写完后也难以获得回应和持续动力。

5. **学习成果不直观**
   如果没有已学单词数、阅读完成数和写作记录等反馈，用户难以获得坚持学习的动力。

### 4.3 用户使用场景

典型使用场景包括：

#### 场景一：日常单词学习

用户进入平台后，选择单词记忆模块，学习当天需要掌握的单词，查看释义、例句和记忆状态，并完成简单复习。

#### 场景二：阅读理解练习

用户进入阅读练习模块，选择一篇适合自己水平的文章，阅读正文，查看重点词汇，并完成基础阅读题目。

#### 场景三：笔友写信练习

用户进入笔友模块，创建受保护的学习者档案，通过系统推荐、受控测试关系或基础筛选找到合适笔友。建立关系后，用户在收件箱中阅读来信，使用写作助手完成回信，并在安全检查后发送。若首版尚未开放真实大规模用户匹配，应在文档和界面中明确说明测试范围。

#### 场景四：查看学习进度

用户完成学习后，进入学习记录页面，查看已学习单词数量、阅读完成数量和写作练习次数，从而了解自己的学习情况。

## 5. 核心业务目标

本项目的核心业务目标包括：

1. **帮助用户稳定积累词汇**
   通过单词学习和复习功能，让用户能够持续接触、记忆和巩固单词。

2. **帮助用户在语境中理解英语**
   通过阅读练习，让用户在文章中理解词汇、句子和篇章含义。

3. **帮助用户进行英文输出训练**
   通过受控的一对一笔友通信，让用户在真实交流中完成英文表达，并通过写作支架和同伴互动持续练习。

4. **帮助用户获得学习反馈**
   通过学习记录和进度展示，让用户知道自己完成了哪些学习任务。

5. **保证平台简单、稳定、易扩展**
   通过模块化设计、组件复用和标准化流程，降低后续维护成本。

## 6. 功能需求分析

### 6.1 单词记忆模块

#### 6.1.1 功能目标

单词记忆模块不应停留在“查看单词详情、用户手动标记掌握状态”的信息展示层面，而应形成一个基于主动回忆和间隔重复的记忆闭环。该模块的目标是帮助用户在学习新词后，通过测验、反馈、自动排程和复习记录逐步巩固记忆。

因此，单词模块应从“静态单词本”升级为“学习—测验—反馈—排程—复习—诊断”的完整流程。

#### 6.1.2 设计原则

1. **主动回忆优先**
   用户不只是看释义，而应在看词回忆释义、看中文回忆英文、选择题、拼写题或听写中完成提取练习。

2. **间隔重复驱动复习**
   系统根据用户答题结果、复习次数、上次复习时间和困难度自动计算下次复习时间，减少用户手动管理复习计划的负担。

3. **状态数据细粒度化**
   不再只依赖“未学习/学习中/已掌握”三种粗粒度状态，而应记录掌握度、熟练度、困难度、连续正确次数、错误次数、复习历史等信息。

4. **错词与复习深度整合**
   错词不应只是单独列表，而应自动进入更高优先级复习队列，并在后续练习中以更高频率出现。

5. **反馈及时明确**
   用户每次作答后，应看到正确答案、错误原因或记忆提示，避免低效重复浏览。

6. **保留扩展空间**
   数据结构应预留学习行为日志，为后续自适应学习、多端同步、离线缓存和个性化推荐提供基础。

#### 6.1.3 基础功能

| 功能点 | 说明 | 优先级 |
|---|---|---|
| 单词列表 | 展示待学习、待复习、困难词等分类入口，并支持基础搜索、筛选和排序 | 必须有 |
| 单词详情 | 展示释义、音标、例句、发音入口和记忆提示 | 必须有 |
| 主动回忆练习 | 支持看词回忆、选择题、拼写题或听写等至少一种测验形式 | 必须有 |
| 自动复习排程 | 根据学习结果生成下次复习时间和待复习队列 | 必须有 |
| 细粒度掌握数据 | 记录掌握度、困难度、复习次数、正确/错误次数、最近复习时间 | 必须有 |
| 答题反馈 | 用户作答后显示正确结果、错误提示和下一步操作 | 必须有 |
| 错词强化 | 错误单词自动进入困难词或高优先级复习队列 | 应该有 |
| 学习行为日志 | 记录每次学习、作答、复习的时间戳、结果和反应时间 | 应该有 |
| 多模态学习 | 支持单词发音、例句朗读；后续可扩展图片联想、跟读评测和听力辨别练习 | 应该有 |
| 诊断统计 | 展示今日目标、待复习数、连续学习天数、困难词分布等 | 应该有 |
| 自定义词库 | 支持用户后续导入或收藏单词 | 可以有 |
| 多端同步与离线支持 | 后续支持跨设备同步和离线学习后的冲突处理 | 可以有 |

#### 6.1.4 记忆状态与数据维度

单词学习状态应由多维数据共同描述，而不是只依赖一个简单标签。建议基础数据包括：

| 数据维度 | 说明 |
|---|---|
| learningStatus | 当前学习阶段，如未学习、初学、复习中、稳定掌握 |
| masteryScore | 掌握度评分，可用于展示学习进展 |
| difficultyLevel | 单词对当前用户的困难程度 |
| reviewCount | 累计复习次数 |
| correctCount | 累计答对次数 |
| wrongCount | 累计答错次数 |
| lastReviewedAt | 最近一次复习时间 |
| nextReviewAt | 系统计算的下次复习时间 |
| reviewIntervalDays | 当前复习间隔天数，用于解释和迁移排程结果 |
| algorithmVersion | 当前使用的复习算法或规则版本，便于后续平滑升级 |
| consecutiveCorrect | 连续答对次数 |
| lastAnswerResult | 最近一次作答结果 |
| nearMissCount | 接近正确次数，用于区分轻微拼写错误和完全错误 |
| correctedAfterNearMiss | 接近正确后是否完成修正输入，用于判断反馈闭环是否完成 |
| nearMissCorrectionAttempts | 接近正确后的修正尝试次数，用于避免反复修正造成挫败 |
| nearMissCorrectionType | 接近正确后的修正方式，如重新输入、补全字母、选择正确拼写或确认差异 |
| lastScheduleReason | 最近一次进入复习队列的原因，如到期复习、错词强化或长期未复查 |
| deferredUntil | 稍后再练的返回时间或返回位置 |
| deferReason | 暂缓练习原因，如连续错误、用户主动跳过或会话保护 |
| userOverrideStatus | 用户手动调整状态，如手动标记已掌握、重置进度或暂不复习 |
| lastUserOverrideAt | 用户最近一次手动调整学习状态的时间 |
| userOverrideCountInWindow | 一段时间内的手动调整次数，用于触发轻量冷却或二次确认 |
| selfAssessmentGap | 用户自主标记与客观练习、阅读表现之间的偏差指标，可为空 |
| maxIntervalDays | 最大复习间隔，避免高掌握词长期消失 |
| lastSeenAt | 最近一次展示或预习时间，用于处理已查看但未练习的新词 |
| previewedAt | 新词预习时间，区分已看过但未完成练习的状态 |
| responseTimeMs | 最近一次作答耗时，可用于判断熟练度 |
| averageResponseTimeMs | 平均作答耗时，用于后续诊断分析 |
| partialSessionResult | 未完成会话中已产生的有效学习成果摘要 |
| easinessFactor | 单词对当前用户的容易程度，可兼容 SM-2 类算法 |
| stability | 记忆稳定性参数，可为 FSRS 等算法预留空间 |
| currentExerciseLevel | 当前适合该单词的练习难度等级 |
| distractorUsedCount | 作为选择题干扰项出现的次数，可用于后续干扰项轮换 |
| lastUsedAsDistractorAt | 最近一次作为干扰项出现的时间，可用于后续降低重复抽取 |
| confusableResolvedStreak | 易混淆词对连续正确区分次数，可用于后续降级提醒 |
| availableExerciseTypes | 自定义或低质量词条当前适合生成的题型列表 |
| pronunciationScore | 后续跟读评测得分，可为空 |
| pronunciationAttemptCount | 跟读练习次数，可为空 |
| lastPronunciationAt | 最近一次跟读时间，可为空 |
| updatedAt | 最近一次学习状态更新时间，用于同步冲突判断 |
| deviceId | 最近一次更新来源设备，用于后续多端同步排查 |
| syncVersion | 同步版本号，用于避免旧数据覆盖新数据 |
| confusableWith | 易混淆词列表，可由人工标注或后续算法挖掘 |

这些字段可以为间隔重复算法、困难词识别、学习统计和后续个性化推荐提供数据基础。其中 `easinessFactor` 和 `stability` 初期可以为空或使用默认值，但应与 `nextReviewAt` 的计算逻辑一起封装，方便后续从简单规则升级到 SM-2、FSRS 或其他间隔重复算法。

为避免后续算法替换时出现复习安排断层，首版简单规则也应保留可迁移的数据含义。MVP 阶段可采用“结果分级 + 间隔递增”的基础规则：完全正确时提高掌握度并延长复习间隔；接近正确时记录为独立结果，不直接进入长间隔；错误时降低掌握度、缩短复习间隔并进入错词强化队列。系统不应只保存最终的 `nextReviewAt`，还应保存作答结果、间隔天数、连续正确次数和算法版本，使后续升级到 SM-2、FSRS 等算法时能够根据历史日志重新估算容易度、稳定性和复习计划。

#### 6.1.5 推荐典型流程

```text
进入单词模块
  -> 查看今日任务：新词数量、待复习数量、困难词数量
  -> 学习新词：查看发音、释义、例句和记忆提示
  -> 完成新词预习后，提示进入主动回忆练习或加入稍后练习
  -> 进入主动回忆练习：选择题/拼写题/听写/卡片翻转
  -> 系统判断正误并给出即时反馈
  -> 逐题实时提交作答结果
  -> 更新掌握度、困难度、复习次数、反应时间和行为日志
  -> 自动计算下次复习时间
  -> 展示简短排程说明，例如下次复习时间和本次调整原因
  -> 错误单词进入困难词或高优先级复习队列
  -> 用户查看今日完成度和薄弱词提示
```

该流程比“查看详情后手动标记”更符合记忆规律，也更容易形成有效学习闭环。

#### 6.1.6 复习机制需求

复习机制应明确回答以下问题：

| 问题 | 需求说明 |
|---|---|
| 复习什么 | 优先复习到期单词、错误单词、低掌握度单词 |
| 何时复习 | 系统根据上次复习结果自动生成下次复习时间 |
| 如何复习 | 采用主动回忆练习，如选择题、拼写题、听写或卡片翻转 |
| 如何判断效果 | 根据用户作答正误、反应时间、题型难度和复习次数更新掌握度 |
| 错词如何处理 | 错词自动进入高优先级复习队列，并增加后续出现频率 |
| 为什么今天复习 | 在用户端用简短标签说明排程原因，如“上次答错”“今天到期”“长期未复查”“接近正确需巩固” |

自动排程不应完全以黑盒形式呈现。用户在今日任务、单词卡片或复习结束页中，应能看到简短、可理解的解释，例如“这个词上次拼写错误，所以今天再次练习”“该词已 14 天未复查，系统安排一次保鲜复习”。解释内容不需要暴露复杂算法参数，但应让用户知道复习安排与自己的学习行为有关，从而提高对系统排程的信任。

排程解释也不应过度打扰。MVP 阶段可采用“默认标签 + 按需展开”的方式：列表中只显示简短标签，如“到期”“错词”“保鲜”，用户点击或悬停后再查看完整说明；同一会话中重复出现的原因可在顶部汇总，例如“今日复习主要包括到期词 12 个、错词 5 个”。这样既保留透明度，也避免每张卡片都显示长句造成视觉疲劳。

#### 6.1.7 复习优先级与队列调度约束

复习队列不能简单地把“到期单词、错误单词、低掌握度单词”全部混在一起，否则可能出现错误词挤占正常到期复习、低掌握度词长期排不上队等问题。因此，系统应设计明确的复习优先级策略。

建议采用“分层筛选 + 配额混合 + 上限保护”的方式：

| 队列类型 | 来源 | 调度原则 |
|---|---|---|
| 到期复习队列 | `nextReviewAt <= 当前日期` 的单词 | 作为每日复习主体，避免正常间隔被延误 |
| 长间隔兜底队列 | 高掌握度但超过最大间隔或长期未出现的单词 | 低频抽样复查，避免假性掌握 |
| 错词强化队列 | 最近答错或连续错误的单词 | 提高优先级，但不应完全挤占到期复习 |
| 低掌握度队列 | 掌握度低、困难度高的单词 | 按比例混入，避免长期遗忘 |
| 新词队列 | 尚未学习的新单词 | 受每日新词上限控制，避免增加过多记忆负担 |

MVP 阶段可以先采用简单规则：优先处理到期复习词，再按比例插入错词和低掌握度词；当复习任务过多时，限制单日任务总量，把超出部分顺延到后续复习，而不是无限增加当天负担。这个比例不宜固定写死，可根据用户近 7 天正确率或最近若干次练习表现动态调整：正确率较高时适当提高新词和正常到期词比例；正确率较低时减少新词、提高错词和低掌握度词比例；错词占比应设置上限，避免用户在单次会话中被错误词反复打断。用于调节配额的正确率应有明确口径，建议优先统计到期复习和错词复习中的主动回忆题，不把新词首次预习计入；选择题、卡片回忆、拼写题可按题型难度设置权重；“接近正确”不计为完全正确，可按部分掌握或单独指标记录。这样可以避免低门槛题型或新词首练扭曲复习强度。对于高掌握度单词，应设置 `maxIntervalDays` 或低频抽样复查机制，避免系统把单词长期隐藏后形成“假性掌握”。

#### 6.1.8 最大复习间隔与假性掌握保护

间隔重复算法不能无限拉长复习间隔。对于高掌握度单词，如果复习间隔被拉到数月甚至更长，用户可能已经遗忘，但系统仍把该词视为稳定掌握，形成“假性掌握”。

因此，系统应预留以下保护机制：

- 设置最大复习间隔 `maxIntervalDays`，超过上限后进入低频兜底复习。
- 对高掌握度词进行少量抽样复查，验证记忆是否仍然稳定。
- 当抽样复查失败时，立即降低掌握度并缩短后续复习间隔。
- 在统计面板中展示长期未复查词数量，提醒用户进行维护性复习。
- 从用户视角将兜底复习包装为“记忆保鲜”“温故知新”或“快速抽查”，避免让用户只看到额外负担。
- 对兜底复习结果给出积极解释，例如“这些词很久没出现了，本次答对说明记忆保持较稳定”。

MVP 阶段可采用固定最大间隔或简单兜底队列，后续再根据算法升级动态调整。界面上不宜只显示“长期未复查词数量”，还应提供明确入口，例如“开始 5 个记忆保鲜词”，让用户理解这是维护长期记忆的轻量任务，而不是突然增加的复习压力。

如果用户连续跳过记忆保鲜任务，系统不应让待保鲜列表无限堆积。可采用小批量滚动策略：每次只展示少量高优先级保鲜词，连续跳过时降低提醒频率或合并为“本周保鲜建议”，而不是每天累加待办数量。长期跳过的词仍保留在兜底池中，但不应挤占正常到期复习和错词复习。

#### 6.1.9 用户自主控制边界

自动排程可以降低用户管理复习计划的负担，但不应完全剥夺用户的控制权。单词模块应提供有限、可追踪的手动调整能力，并明确哪些操作会影响算法判断，哪些操作只影响当前会话。

建议控制边界如下：

| 用户操作 | 处理原则 |
|---|---|
| 手动标记“已掌握” | 允许操作，但应记录为用户手动调整，不直接等同于多次主动回忆成功；后续仍可安排低频抽样复查 |
| 手动标记“仍不熟” | 可降低掌握度或缩短下次复习间隔，并进入低掌握度队列 |
| 跳过本次复习 | 默认只影响当前会话，不直接增加错误次数；若连续多次跳过，可作为低信心信号进入短期提醒 |
| 重置某词记忆进度 | 作为明确操作提供二次确认，重置后回到新词或初学状态 |
| 暂不复习某词 | 可作为扩展能力，需区别于“已掌握”，避免系统误判为稳定掌握 |

所有手动调整都应写入学习行为日志，并在界面上提示影响范围。例如，用户跳过一个单词时，系统可提示“本次跳过不会算作答错，但稍后仍会安排复习”。这样既保留自动排程的稳定性，也给用户必要的掌控感。

为避免用户在情绪波动时频繁批量重置或反复手动标记同一批单词，系统应设置轻量安全阀。例如，同一单词在短时间内多次手动调整时，要求二次确认；批量重置超过一定数量时，提示“建议先完成一次复习再批量调整”；短时间内频繁操作可进入冷却状态，只允许继续学习，不继续叠加算法权重。冷却机制不应阻止用户学习，只用于保护学习日志质量和排程稳定性。

更深层的设计问题是“系统智能”和“用户自主权”的平衡。若用户频繁手动标记“已掌握”，但后续在阅读中遇到这些词时查词频繁、答题正确率下降或阅读速度明显变慢，系统不应简单否定用户判断，也不应完全信任主观标记。建议在后续设计中引入轻量的“认知偏差”指标，用于观察用户自评掌握度与客观表现之间的长期差异。

认知偏差指标只用于温和引导，不用于惩罚用户。例如，当系统发现用户标记了较多已掌握词，但近期阅读中仍频繁卡在这些词上时，可以在阅读诊断页或单词复习入口提示：“你标记了不少已掌握词汇，但最近阅读时遇到它们还有些卡顿，要不要安排一次轻量的‘词汇保鲜’挑战？”用户可以接受、稍后再说或关闭该建议。这样既尊重用户自主权，也让系统客观数据发挥校准作用。

#### 6.1.10 错词强化与挫败保护

错词强化虽然能提高复习针对性，但如果用户连续错误，尤其是在拼写题、听写题等高难度题型中，容易产生挫败感。因此，系统应避免把错词强化设计成单纯的“越错越频繁”。

建议增加以下保护机制：

- 连续错误时自动降低题型难度，例如从听写题降级为选择题或卡片回忆。
- 降级后连续答对 2～3 次时，自动尝试恢复或提升一级题型难度，重新检验真实掌握度。
- 连续错误达到阈值后，先展示释义、例句、词根或记忆提示，再继续练习。
- 对同一单词设置单次学习会话内的出现次数上限，避免反复打击用户。
- 对困难词使用“稍后再练”机制，暂时移出当前会话，在后续复习中重新出现。
- “稍后再练”不应简单等同于推迟到第二天。轻度错误可在间隔若干个题目后回到当前会话；连续错误或用户主动选择暂缓时，可移入本次会话末尾或下一次短间隔复习；单次会话达到错误次数上限后，应优先引导用户查看释义、例句或记忆提示，而不是继续强制答题。
- 升级失败时允许回退到较低难度，避免用户从低难度直接跳回高难度造成挫败。
- 反馈文案应鼓励用户继续学习，避免使用负面评价。

#### 6.1.11 拼写题与听写题答案校验策略

拼写题和听写题如果采用完全严格的字符串匹配，容易因为大小写、空格、标点或输入法差异导致用户被误判错误，影响学习体验。因此，系统应在保证学习准确性的前提下设计答案标准化和容错策略。

建议采用以下规则：

| 校验项 | 处理方式 |
|---|---|
| 大小写 | 默认不区分大小写，如 `Apple` 与 `apple` 视为同一答案 |
| 首尾空格 | 自动去除首尾空格 |
| 连续空格 | 多个连续空格合并为一个空格 |
| 标点差异 | 听写句子可忽略句末句号、问号等非关键标点 |
| 撇号和连字符 | 统一常见英文撇号格式，如智能引号与普通引号 |
| 英美拼写 | 对确认为合法变体的拼写，可通过 `acceptedSpellings` 字段配置 |
| 近似拼写错误 | 不直接判为正确，可标记为“接近正确”并提示用户修改 |

建议将结果分为三类：

1. **正确**：标准化后与标准答案或合法变体一致。
2. **接近正确**：只有轻微拼写差异，系统提示修改，但不直接按完全掌握处理。
3. **错误**：与答案差异明显，进入正常错误反馈流程。

“接近正确”应作为独立作答结果写入行为日志，不宜简单等同于正确或错误。建议处理规则为：给予轻量正向反馈，提示具体差异；不增加连续正确次数；不直接进入长间隔复习；可小幅提高掌握度或保持不变，并安排短间隔再次练习。这样既能承认用户已经接近掌握，又能避免错误拼写被系统强化。

在交互闭环上，系统不应只展示正确答案后直接进入下一题。对于接近正确的拼写或听写结果，建议要求用户完成一次轻量修正，例如重新输入完整正确拼写、补全错误字母或点击确认差异。修正成功后再进入下一题，并记录 `correctedAfterNearMiss` 等行为标记。这样可以让用户真正完成一次正确拼写输出，而不是只被动看见差异。

“确认差异”本身应具有微学习效果，而不是单纯点击“知道了”。例如，系统可高亮错误字母与正确字母，让用户点击差异位置、补全缺失字母，或从两个相近拼写中选择正确形式。若用户选择点击确认，也应至少要求其看见差异高亮和正确拼写，再进入下一题。这样可以把确认动作转化为一次短暂的辨别练习。

不同修正路径的学习深度应略作区分，但不增加用户负担。重新输入完整拼写或补全错误字母属于输出型修正，可记录为较强的修正证据；从两个拼写中选择正确形式属于辨认型修正，只能说明用户完成了差异辨别，不应等同于完整拼写输出。系统可以统一给出通过反馈，但在后续排程中对辨认型修正保持更短复习间隔，避免低门槛修正被误判为完全掌握。

接近正确的修正也应设置上限，避免用户反复输入相近错误形成挫败循环。建议一次题目中最多要求 1～2 次修正；若再次接近正确或仍然错误，系统应展示正确拼写并让用户点击确认差异，随后将该词安排到短间隔复习或降级题型，而不是无限要求重新输入。这样既保留纠错闭环，也避免小错误被放大成负面体验。

若 MVP 阶段选择拼写题或听写题作为主动回忆题型，则至少应实现大小写忽略、首尾空格清理、连续空格合并、基础标点处理、“接近正确”的独立记录和一次轻量修正确认，并设置修正次数上限。若首版只采用选择题、卡片翻转等非输入型题型，上述答案校验能力可作为后续题型扩展要求。英美拼写变体和近似拼写提示可以通过配置字段逐步完善，避免过度宽松导致用户把错误拼写当成正确记忆。


#### 6.1.12 交互与反馈需求

单词模块应补充微观交互和即时反馈：

- 卡片翻转：先显示英文或中文，用户回忆后再查看答案。
- 选择题：根据释义选择正确单词，降低初期练习门槛。
- 拼写题：根据中文释义或发音输入英文单词。
- 听写题：播放单词或例句发音，用户输入听到的内容。
- 新词练习引导：用户查看一组新词后，页面应提供“立即练习”“加入今日复习”“稍后提醒”等入口，避免新词长期停留在只预习未练习状态。
- 最小配对辨别：针对近音词设计听力辨别题，例如区分 `ship` 与 `sheep`。
- 语境听力选义：播放包含目标词的短句，让用户根据语境选择正确释义。
- 答案容错：拼写题和听写题应先进行基础标准化处理，再判断正误，避免因大小写、多余空格或标点差异造成不必要的错误判定。
- 即时反馈：答错后显示正确答案、例句和记忆提示。
- 微观进展反馈：除完成今日任务、连续答对、掌握困难词、错词减少或掌握度提升外，系统还可提示“本次回忆更稳定”“这个易混词已连续 3 次答对”“该词从高风险降为中风险”等细粒度进步，让用户在重复练习中感知小幅改善。
- 速度类反馈应谨慎使用。若提示“本次回忆比上次更快”，应作为可选的积极补充，而不是核心评价标准；对于拼写、听写或用户明显耗时思考后答对的情况，应优先强调准确性和稳定性，避免让用户为了追求速度而牺牲理解。
- 今日收获反馈：练习结束页除展示数字外，还可用更具象的方式呈现学习成果，例如“今日稳固了 8 个旧词、找回 3 个易忘词、解锁 1 组易混词对”，或用轻量“记忆地图/今日收获卡”展示本次学习覆盖的主题和掌握变化。MVP 阶段不必制作复杂分享图，但应让用户看到一次练习带来的具体成果，而不是只看到完成数量。
- 未完成场景也应给出正向承接。如果用户只完成了一部分练习就退出，恢复页或阶段性结束页可以展示“已完成部分仍已保存”“刚刚稳固了 4 个词，剩余 6 个可稍后继续”等提示，避免用户觉得半途退出等于白学。未完成收获不应夸大为完整任务完成，只需强调已保存、可继续和已有小成果。
- 排程解释提示：当某个词进入今日任务时，卡片或详情页应显示简短原因，如“今天到期”“上次接近正确”“长期未复查”，避免用户只看到数量而不了解安排逻辑。
- 错误纠正：对连续错误单词，降低后续练习难度或增加提示。
- 恢复性升级：当某个单词在降级题型下连续答对 2～3 次后，系统应尝试提升一级题型难度，例如从选择题升至卡片回忆，再升至拼写题或听写。
- 升级保护：如果升级后再次失败，应允许回退到较低难度并继续巩固，形成“降级—巩固—试探升级”的动态循环。
- 键盘操作：练习交互应支持 Tab、Enter、Space 和数字键选择选项等标准键盘操作，提高练习效率。
- 无障碍标签：卡片翻转、选项按钮、发音按钮和反馈状态应提供清晰的 ARIA 标签，保证屏幕阅读器能够理解当前状态和操作结果。

#### 6.1.13 统计与诊断需求

单词学习统计不应只展示“已学/已掌握”数量，还应提供一定诊断价值。建议包括：

- 今日新学单词数量。
- 今日待复习单词数量。
- 今日复习完成度。
- 连续学习天数。
- 掌握度分布。
- 困难词数量和分类。
- 最近一周学习趋势。
- 错误率较高的单词或词类。
- 近 7 天或 30 天正确率、复习完成率和记忆保持率变化。
- 遗忘风险较高的单词列表，以及遗忘较快的词类、话题或难度分布。
- 易混淆词组，例如形近词、近义词或用户频繁交替出错的词对。
- 即将到期复习数量。
- 长间隔兜底复习数量，提示高掌握但长时间未出现的单词。

MVP 阶段可以先实现今日任务、待复习数量、完成度和困难词数量；最近一周正确率趋势、遗忘曲线、保持率变化和不同词类的衰退速度可视化可作为后续增强。统计展示应避免给出过度精确的算法结论，可以采用“遗忘风险较高”“近期保持较稳定”等解释性描述，帮助用户理解系统排程依据。

#### 6.1.14 词库元数据需求

若系统要统计错误率较高的词类、话题或难度分布，单词数据必须具备足够的元数据。否则，诊断统计只能停留在单词数量层面，无法分析用户薄弱点。

建议单词数据预留以下字段：

| 字段 | 说明 |
|---|---|
| partOfSpeech | 词性，如名词、动词、形容词等 |
| difficulty | 难度等级 |
| topicTags | 话题标签，如校园、旅行、科技、商务等 |
| examTags | 考试标签，如中考、高考、CET、考研等，是否启用需后续确认 |
| phonetic | 音标 |
| examples | 例句列表 |
| synonyms / antonyms | 近义词和反义词，后续可选 |
| source | 单词来源，便于质量追踪 |
| version | 数据版本，便于后续更新和兼容 |
| audioSource | 发音或例句音频来源，便于版权和质量追踪 |
| audioLicense | 音频授权信息，避免使用来源不明资源 |
| confusableWith | 易混淆词列表，用于后续对比练习和诊断展示 |

MVP 阶段至少应保证词性、难度、话题标签、音标、例句和易混淆词字段可扩展，避免后续为了统计、测验生成和易混词诊断重新改造数据结构。

#### 6.1.15 选择题干扰项质量控制

选择题虽然能降低初学者练习门槛，但如果干扰项质量过低，用户可能通过排除法答对，而不是完成真正的词义提取。因此，选择题干扰项不应只随机抽取，也不应只满足数量要求。

建议干扰项生成遵循以下原则：

| 维度 | 要求 |
|---|---|
| 词性一致 | 优先选择与正确答案相同词性的词，避免选项过于明显 |
| 难度接近 | 干扰项难度不应与正确答案差距过大 |
| 主题相关 | 同一主题或相近语境的词更适合作为干扰项 |
| 长度适中 | 避免一个选项明显比其他选项短或长，造成形式提示 |
| 语义可辨 | 干扰项应有一定迷惑性，但不能与正确答案完全等价 |
| 错误来源可解释 | 用户选错后，系统应能说明混淆点，如近义、形近、音近或语境误解 |

干扰项还应考虑用户个人学习状态。用户已经稳定掌握的词可以少量作为干扰项，但不宜频繁用于容易混淆的题目，避免用户因反复被熟词干扰而动摇信心；如果某个已掌握词多次被错选，应将其标记为易混淆词对，而不是简单降低两个词的掌握度。对于尚未学习的词，也不宜大量作为干扰项，否则用户可能因为陌生选项过多而依赖排除法。

当用户错选了一个已稳定掌握的干扰项时，系统应立即给出温和对比，而不是只进入下一题。例如展示“你把 `affect` 和 `effect` 混淆了，本题考查的是动词含义”，并提供一个 10～20 秒的轻量对比卡片或后续专项入口。对比卡片应包含一个极简微动作，例如让用户点选“哪个是动词”、补全一句短例句，或点击确认两词差异，而不是只被动关闭。该反馈不应直接判定熟词退步，而是先沉淀为易混淆对；若后续多次混淆，再安排专项对比练习。

干扰项还需要考虑“新鲜度”。随着用户词汇量增长，如果同一批候选词反复作为干扰项出现，用户可能形成“这个选项总是错的”的机械记忆。后续迭代可记录 `distractorUsedCount` 和 `lastUsedAsDistractorAt`，当某个词对同一用户近期频繁作为干扰项出现时，降低其再次抽取优先级，优先轮换其他候选词。该能力属于运营期算法微调，不纳入 MVP。

易混淆词对也应有“确认后管理”。如果同一组易混淆词在后续复习中被用户连续正确区分多次，可将其从高频专项辨析降级为“已克服”，只在低频兜底复习中偶尔抽查，避免用户觉得系统反复提醒已经掌握的内容。后续可通过 `confusableResolvedStreak` 记录连续正确区分次数，达到阈值后降低提醒频率。该能力同样作为后续迭代预留。

MVP 阶段可先采用“同词性 + 同难度 + 同主题优先”的基础规则，并对干扰项不足的情况提供降级策略，例如改用卡片回忆题或从高质量公共词库中补足，而不是生成明显无效的选项。后续可根据用户错选记录沉淀高质量易混词对，用于更有针对性的选择题和专项复习。

#### 6.1.16 学习负荷与排程配置

不同用户的学习时间和记忆能力不同，系统不能只提供固定复习强度。若自动生成的复习任务过多，用户容易放弃；如果任务过少，又会影响学习效果。因此，单词模块应允许用户配置基础学习负荷。

建议提供以下配置项：

- 每日新词上限。
- 每日复习上限。
- 每次练习题目数量。
- 复习强度模式，例如轻量、标准、强化。
- 是否优先处理错词。
- 是否启用听写或拼写等高难度题型。

MVP 阶段可以先提供每日新词上限和每日复习上限两个配置项，其他配置可作为后续扩展。

#### 6.1.17 单词列表搜索、筛选与排序

随着词库规模扩大，用户需要快速定位和管理单词。单词列表不应只展示静态列表，还应具备基础检索能力。

建议支持：

- 按英文单词或中文释义搜索。
- 按学习状态筛选，如新词、待复习、困难词、已掌握。
- 按掌握度、困难度、最近复习时间、下次复习时间排序。
- 按词性、话题、难度标签筛选。
- 自定义词库导入后，支持按来源筛选。

MVP 阶段至少应支持关键词搜索和按学习状态筛选；大规模列表应考虑分页、虚拟滚动或分批加载，避免页面卡顿。

#### 6.1.18 多媒体与多模态需求

语言学习需要声音输入，因此单词模块应优先考虑发音能力：

- 单词发音播放。
- 例句朗读。
- 发音加载失败时提供文字兜底。
- 后续可扩展图片联想、跟读录音、发音评测等能力。

在首版中，发音和例句朗读可以作为“应该有”功能；发音评测和图像联想属于后续扩展功能。

#### 6.1.19 多媒体能力的 MVP 边界与降级策略

多媒体功能需要明确排期边界。为避免开发摇摆，建议将单词发音和例句朗读划分为“首版增强功能”，即：如果技术条件允许，应在首版实现；如果受限，应保留明确接口和降级体验。

降级策略包括：

- 发音资源加载失败时，仍显示音标、释义和例句。
- 网络较差时，不阻塞单词学习主流程。
- 发音按钮应有加载中和失败提示。
- 后续如支持离线缓存，应标记缓存状态，避免用户误以为资源可用。
- 发音评测、跟读打分、图片联想不纳入 MVP，作为后续增强功能。
- 如果后续支持跟读评测，应预留发音评分字段，但默认不保存原始录音；确需上传或保存录音时，应明确告知用户并获得授权。授权触发时机应放在用户首次点击跟读、录音或发音评测入口时，而不是用户刚进入单词模块时；授权粒度可采用“首次使用授权 + 设置页可随时关闭”的方式，若涉及云端上传或长期保存，应在上传前单独说明用途、保存周期和删除方式。

#### 6.1.20 听力辨别与跟读接口预留

听力训练不应只局限于单词或例句听写。后续可以逐步加入更细的听力辨别能力：

- 最小配对辨别：训练用户区分近音词。
- 语境听力选义：在短句中识别目标词含义。
- 跟读练习：用户朗读单词或例句，系统给出发音反馈。

若后续加入跟读评测，应预留以下设计：

- `pronunciationScore`：发音评分，可为空。
- `pronunciationAttemptCount`：跟读次数。
- `lastPronunciationAt`：最近跟读时间。
- 录音默认不长期保存；如需云端处理或保存，应明确提示用途和保存周期。
- 音频材料应记录来源和授权信息，避免版权风险。

MVP 阶段可以先实现发音播放或例句朗读，最小配对辨别、语境听力选义和跟读评测作为后续扩展。

#### 6.1.21 自定义词库导入与质量控制

自定义词库属于扩展功能，但需要提前考虑数据质量。如果用户导入的词表格式混乱，可能影响释义展示、测验生成和统计分析。因此，自定义词库应具备基本校验规则。

建议规则包括：

- 导入前检查必填字段，如单词、释义、词性或难度。
- 对音标、例句、标签等字段提供统一格式要求。
- 对重复单词提供合并、覆盖或跳过选项。
- 对来源不明或字段缺失的单词标记为“待完善”。
- 自定义词不应直接污染系统内置词库，应与系统词库分层存储。
- 选择题干扰项生成应优先使用同难度、同词性或同主题的高质量词条。

自定义词条还应考虑题型适配。用户导入的词条可能只有单词和释义，缺少音标、例句、词性或音频资源，直接用于听写题、语境选义题或高质量选择题可能造成误判。后续可根据字段完整度生成 `availableExerciseTypes`：只有单词和释义的词条先用于卡片回忆和基础选择题；具备词性和难度时再参与高质量干扰项生成；具备音标、发音或例句时再开放听写、发音和语境类题型。

MVP 阶段可以暂不实现导入功能，但数据模型应预留 `source`、`isCustom`、`qualityStatus`、`availableExerciseTypes` 等字段。若后续导入功能上线，应优先根据 `qualityStatus` 做题型降级，而不是强行让字段不完整的词条参与所有题型。

#### 6.1.22 长周期学习动机维持

轻量提示只能提供短期反馈，难以支撑长期学习。平台后续应考虑更稳定的动机机制，但仍要避免偏离学习目标。

可规划的动机机制包括：

- 长期目标设置，例如考试日期、目标词汇量、每日学习计划。
- 阶段性里程碑，例如连续学习 7 天、掌握 100 个困难词。
- 学习趋势反馈，例如最近一周完成度和复习稳定性。
- 轻量排行榜或小组学习，但不作为 MVP 核心功能。
- 冲刺模式，例如考试前按截止日期调整每日任务量。

MVP 阶段应优先实现今日目标、连续学习天数和阶段性完成提示；社交型激励可后续再做。

连续学习天数应明确跨日和时区口径，避免用户因深夜学习被误判中断。建议以用户本地时区或账户设置时区为准，并设置“学习会话归属日”规则：若用户在深夜开始学习并跨到次日凌晨，可将该连续会话归入开始日期，或在产品上采用 3～4 小时宽限窗口。无论采用哪种规则，都应在统计说明中保持一致，避免前端和后端口径不一致导致连续记录异常。

#### 6.1.23 中断恢复与会话连续性

用户可能在练习中途退出、网络断开或页面崩溃。如果系统不保存中间状态，用户需要重复完成已做题目，会造成明显挫败。因此，单词模块应定义会话恢复机制。

建议记录以下会话状态：

- 当前练习模式。
- 当前题目列表。
- 已完成题目数量。
- 每题作答结果。
- 当前学习进度。
- 已反馈题目是否已经提交到排程算法。
- 会话开始时间和最近保存时间。
- 会话是否已完成。

恢复策略包括：

- 用户中途退出后，再次进入可选择继续上次练习或重新开始。
- 新词只完成查看但未完成主动练习时，应保留为“已预习未练习”状态，下次优先回到新词/未完成队列，而不是按已掌握词排程。
- 采用逐题实时提交策略：每题作答并反馈后，立即更新该单词的学习状态、排程时间和行为日志。
- 已完成题目的结果应及时保存，避免整组练习失败后全部丢失。
- 网络失败时先保存本地状态，待恢复后再同步。
- 如果会话过期，应保留已完成记录，并重新生成剩余任务。
- 未完成的新词学习会话应保留预习状态和未练习词列表，避免用户重复浏览已看内容，也避免遗漏主动回忆。
- 会话恢复时只拉取未完成题目列表，不应对已生效的作答结果重复提交。

#### 6.1.24 技术与数据扩展需求

为避免未来重构，单词模块应在需求阶段预留以下能力：

- 学习行为日志：记录每次查看、作答、复习、跳过和完成行为，并预留 `responseTimeMs` 字段记录从题目展示到作答提交的耗时。
- 复习调度字段：保存下次复习时间、间隔、困难度、掌握度、容易度系数和记忆稳定性参数。
- 自定义词库扩展：支持后续收藏、导入或创建个人词库。
- 多端同步扩展：预留用户标识、`updatedAt`、`deviceId`、`syncVersion` 等同步字段。
- 离线学习扩展：预留本地缓存、同步状态和冲突处理思路；复习行为日志应优先采用追加合并，避免直接覆盖。
- 列表管理扩展：预留按掌握度、困难度、标签、最近复习时间筛选和排序的能力。
- 算法替换扩展：复习算法应封装，便于后续从简单规则升级为 SM-2、FSRS 或其他更复杂的间隔重复策略。
- 易混淆词扩展：预留 `confusableWith`、`confusableResolvedStreak` 字段和易混词行为分析能力，支持后续专项对比练习及已克服词对降级提醒。
- 干扰项轮换扩展：预留 `distractorUsedCount`、`lastUsedAsDistractorAt` 等字段，支持后续降低重复干扰项抽取频率。

#### 6.1.25 多端同步与冲突处理原则

多端同步虽然可以放在后续版本实现，但需求阶段应先约定基础原则，避免后期出现本地进度覆盖云端进度、复习日志丢失或下次复习时间异常等问题。单词模块中的数据可分为“行为日志”和“汇总状态”两类处理：

| 数据类型 | 冲突处理原则 |
|---|---|
| 学习、作答、复习日志 | 采用追加合并，不因另一端同步而直接删除 |
| 掌握度、困难度、下次复习时间 | 作为系统计算字段，优先根据合并后的日志重新计算 |
| 用户主动编辑内容 | 如自定义释义、收藏、标签等，可按 `updatedAt` 采用最后修改优先 |
| 本地离线会话 | 恢复网络后按题目级记录合并，避免整组会话覆盖 |

为支持上述规则，数据结构应预留 `updatedAt`、`deviceId`、`syncVersion` 和本地临时记录标识。MVP 阶段即使不实现完整多端同步，也应避免把学习进度设计成只能整体覆盖的单一状态字段。

#### 6.1.26 单词模块专项非功能需求

单词模块的数据量和交互频率较高，需要补充专项非功能需求：

| 类型 | 要求 |
|---|---|
| 性能 | 大量单词列表应支持分页、虚拟滚动或分批加载；搜索和筛选不应造成明显卡顿 |
| 隐私 | 学习行为日志、答题记录、反应时间和发音评分属于个人学习数据，默认不公开展示 |
| 录音安全 | 跟读录音默认不长期保存；确需上传或保存时，应在录音或上传前告知用户用途、保存周期和删除方式，并允许用户在设置中关闭相关能力 |
| 音频版权 | 单词发音、例句朗读等音频资源应记录来源和授权信息 |
| 同步安全 | 复习行为日志优先追加合并，掌握度和下次复习时间等汇总字段可由日志重新计算，避免多端同步时直接覆盖学习记录 |
| 浏览器兼容 | 核心学习流程应兼容主流现代浏览器；音频不可用时不应阻断文字学习 |
| 可访问性 | 键盘操作、ARIA 标签和可读反馈应作为基础交互规范，不应等到后期返工 |


#### 6.1.27 验收标准

单词模块验收应区分 MVP 最低标准和后续增强标准。若首版只选择一种主动回忆题型，则只验收该题型相关能力；拼写题、听写、发音等未进入首版的能力，应保留设计和降级说明，但不作为 MVP 阻断项。

**MVP 最低验收：**

- 用户可以正常查看今日新词、待复习词和困难词入口。
- 用户可以完成至少一种主动回忆练习，而不是只能查看单词详情。
- 用户每次练习后能获得正确或错误反馈，并能看到下一步操作。
- 系统能根据练习结果更新基础掌握度、复习次数和下次复习时间。
- 用户能看到简短的复习安排原因，例如到期复习、上次答错或长期未复查；排程解释可采用标签化或按需展开方式，避免界面信息过载。
- 系统能按复习队列优先级生成任务，避免错词完全挤占正常到期复习。
- 用户可以配置或使用默认的每日新词上限、每日复习上限。
- 单词列表应支持基础关键词搜索和按学习状态筛选；大量数据时应预留分页、虚拟滚动或分批加载方案。
- 练习中途退出、网络异常或页面刷新后，已完成题目结果不应全部丢失；已反馈题目应采用逐题保存或等效的可靠保存策略。
- 新词若只完成查看或预习但未进入主动回忆练习，应标记为“已预习未练习”，不应直接进入长间隔复习。
- 页面在无数据、加载失败或保存失败时有明确提示。
- 单词数据模型至少预留词性、难度、话题标签、来源、质量状态、复习算法版本和会话恢复相关字段。
- 如首版暂不实现发音或例句朗读，应在待办中标注，并提供音频不可用时不阻断文字学习的降级方案。

**后续增强验收：**

- 错误单词能进入困难词或高优先级复习范围，并具备连续错误后的难度降级、稍后再练和连续答对后的恢复性升级机制。
- 用户可以在明确边界内跳过、暂缓、手动调整或重置单词学习状态，且这些操作应写入学习日志。
- 频繁手动调整或批量重置应有二次确认、冷却或提示机制，避免破坏排程稳定性。
- 当用户自主标记与客观练习、阅读表现长期背离时，系统应以温和建议提示复查，不应强制覆盖用户选择。
- 复习队列动态配额的正确率口径应明确区分题型、新词首练、复习题和“接近正确”。
- 选择题干扰项应优先满足同词性、难度接近、主题相关和用户学习状态合理，避免明显无效选项或过度干扰已稳定掌握词。
- 错选已掌握干扰项时，应给出带有极简微动作的温和对比反馈或沉淀为易混淆词对。
- 自定义词库应预留题型适配字段，字段不完整的词条后续应能降级到适合的练习题型。
- 拼写题或听写题上线后，应支持大小写、空格和基础标点的标准化处理；出现“接近正确”时，系统应提示差异，并提供一次轻量修正或确认，修正次数应有上限。
- 练习结束页可提供“今日收获”类具象反馈，展示本次稳固、找回或改善的学习成果；未完成会话也应展示已保存的阶段性小成果。
- 卡片翻转、选择题、拼写题、听写和发音按钮应支持基础键盘操作和必要的无障碍语义。

### 6.2 阅读练习模块

#### 6.2.1 功能目标

阅读练习模块不应只是“文章 + 重点词 + 题目”的静态组合，而应围绕真实语境理解、阅读策略训练和学习反馈形成闭环。该模块的目标是帮助用户在有主题、有语境、有任务的文章中理解信息，并通过词汇辅助、长难句辅助、分层题型、答案解析和阅读记录，把一次阅读转化为可积累的能力训练。

阅读模块应形成以下学习路径：

```text
分级选材 -> 带辅助工具阅读 -> 完成多类型理解任务 -> 查看解析与原文依据 -> 沉淀生词和阅读记录 -> 根据表现调整后续材料
```

#### 6.2.2 设计原则

1. **真实语境优先**
   阅读材料应有明确主题和使用场景，例如校园、旅行、科技、文化、考试阅读等，而不是孤立句子堆叠。

2. **辅助但不打断阅读**
   生词、长难句、段落定位、音频等辅助工具应按需展开，避免把阅读页面变成信息噪声。

3. **题型覆盖阅读能力**
   阅读题不应只有简单选择题，应逐步覆盖主旨、细节、推断、词义猜测、结构理解等能力。

4. **反馈必须可解释**
   答题反馈不能只显示对错，还应说明原因、定位原文依据，并给出必要提示。

5. **阅读记录具备诊断价值**
   除完成状态外，还应记录阅读时长、正确率、生词数量、阅读速度和重读情况。

6. **与单词和写作模块联动**
   阅读中遇到的生词应可加入单词本，阅读主题和表达可为写信练习提供素材。

#### 6.2.3 基础功能

| 功能点 | 说明 | 优先级 |
|---|---|---|
| 阅读材料列表 | 展示可练习文章，并支持按难度、主题、标签、完成状态筛选 | 必须有 |
| 冷启动选材 | 新用户通过自评水平、学习目标、主题偏好或短篇诊断题匹配首篇阅读材料 | 必须有 |
| 阅读搜索与排序 | 支持按标题、主题搜索，按难度、推荐度、最近阅读时间排序 | 应该有 |
| 分级阅读材料 | 文章具备难度、主题、预计阅读时间、目标词等元数据 | 必须有 |
| 文章详情 | 展示标题、正文、难度、主题、段落结构和阅读任务 | 必须有 |
| 阅读辅助工具 | 支持重点词释义、音标、发音、例句、段落定位、长难句提示，并保存已展开辅助状态 | 应该有 |
| 生词本联动 | 用户可将阅读中的生词加入单词记忆模块 | 应该有 |
| 高亮标记 | 用户可对好句、疑问点或论证句进行轻量高亮标记 | 应该有 |
| 阅读音频 | 支持文章或段落朗读；音频不可用时不阻断文字阅读 | 可以有 |
| 阅读理解题 | 完整版本覆盖主旨、细节、推断、词义猜测、结构理解等题型；MVP 至少包含主旨、细节、词义猜测三类 | 必须有 |
| 答案解析 | 对每道题提供正确答案、简要解析和原文依据，并提供轻量内容纠错入口；错误选项详细分析可后续增强 | 必须有 |
| 阅读记录 | MVP 记录完成状态、阅读位置、基础阅读时长、正确率和生词量；总停留/有效时长区分、重读次数等可后续增强 | 必须有 |
| 断点续读 | 中途退出后保留阅读位置和已完成题目 | 应该有 |
| 泛读模式 | 允许用户只阅读不答题或跳过部分题目，但记录为泛读 | 可以有 |
| 泛读转精读 | 用户泛读后可转为精读并补做题目、查看解析 | 可以有 |
| 阅读策略提示 | 显性提示略读、扫读、预测、总结等阅读策略 | 应该有 |

#### 6.2.4 阅读材料元数据需求

为了支持筛选、推荐、统计和模块联动，阅读材料应预留以下字段：

| 字段 | 说明 |
|---|---|
| title | 文章标题 |
| topic | 主题分类，如校园、旅行、科技、文化等 |
| difficulty | 难度等级 |
| estimatedReadingTime | 预计阅读时间，元数据中保存基于通用平均速度的静态基准值；前端展示时可结合用户历史有效阅读速度动态校准 |
| wordCount | 单词数量 |
| targetWords | 目标词或重点词列表 |
| source | 文章来源或生成方式 |
| license | 内容授权信息 |
| audioSource | 音频来源，可为空 |
| audioAlignment | 句子级或词级音频时间戳，用于听读同步高亮，可为空 |
| paragraphs | 段落结构，便于定位原文依据 |
| questionTypes | 题型标签 |
| strategyFocus | 阅读策略标签，如略读、扫读、预测、总结 |
| topicTags | 主题标签，用于兴趣偏好和推荐匹配 |
| generatedBy | 内容生成方式，如人工、AI 辅助、导入材料 |
| reviewStatus | 内容审核状态，如待审核、已审核、需修订 |
| createdAt / updatedAt | 内容创建和更新时间 |

这些字段可以避免阅读材料成为孤立文本，并为后续内容筛选、自适应推荐和版权追踪提供基础。

其中 `estimatedReadingTime` 不应只理解为固定展示值。初期可按文章词数和通用平均阅读速度估算；当用户已有足够阅读记录后，页面展示的预计时间可以结合该用户最近一段时间的 `activeReadingDuration` 和个人平均阅读速度动态校准。元数据中的静态值作为基准参考，个性化展示值作为体验优化，不改变文章本身难度标记。

#### 6.2.5 阅读辅助需求

阅读过程中，用户遇到生词、长难句或文章结构不清时，容易产生挫败感。阅读辅助工具应帮助用户继续阅读，但不能替代用户思考。

建议支持：

- 点击重点词查看释义、音标、发音和例句。
- 一键加入生词本，并同步到单词记忆模块的待学习队列。
- 长难句提示，说明句子主干、从句结构或关键表达。
- 对明显较长、从句较多或包含复杂结构的句子提供轻量标识，例如下划线、侧边小图标或“句子较长”提示，引导用户在读不顺时主动查看帮助。
- 长难句标识应克制显示，不应把整篇文章标成困难文本；用户已多次忽略或文章难度较低时，可减少主动提示，避免干扰阅读。
- 段落定位，答题解析可以跳转到相关原文段落。
- 阅读进度提示，例如当前段落、已读百分比。
- 阅读策略锚点：在文章中段可用轻量提示标记与策略相关的位置，例如“先看这一段首句判断主旨”“这里适合扫读数字信息”，帮助用户把阅读前策略提示落实到具体文本。
- 可选文章朗读或段落朗读，帮助用户结合听力输入。
- 听读同步高亮：播放音频时可高亮当前句子或词，并支持跟随滚动；环境不支持时降级为纯音频。
- 对辅助内容采用按需展开方式，避免过度打断阅读。

辅助工具还需要兼顾“沉浸式体验”和“功能可发现性”。策略模式、听读同步、长难句提示等功能如果长期隐藏，用户可能不知道它们存在；如果持续显示，又会打断阅读心流。因此，阅读模块应采用“适时教育”方式：在用户首次遇到明确场景时给出非模态、可忽略、低频出现的提示。例如，用户第一次在长难句段落停留超过一定时间，可在句子侧边出现轻量气泡：“读起来有点绕？试试点击左侧放大镜查看结构。”用户忽略后气泡自动消失，并在一段时间内不重复出现。

适时教育提示应遵循以下原则：只在高相关场景出现；不遮挡正文和题目；可关闭或忽略；同类提示有频率上限；用户已主动使用过相关功能后减少提示。这样既能让用户发现功能，又能保持专注阅读。

#### 6.2.6 阅读题型需求

阅读理解题应覆盖不同层次的阅读能力，而不是只检查用户是否记住某个细节。

| 题型 | 训练目标 |
|---|---|
| 主旨题 | 判断文章中心思想和作者意图 |
| 细节题 | 定位和理解文章中的明确信息 |
| 推断题 | 根据上下文推断隐含含义 |
| 词义猜测题 | 根据语境判断单词或短语含义 |
| 结构题 | 理解段落关系、转折、因果和论证结构 |
| 观点态度题 | 判断作者态度或文章语气 |

MVP 阶段可以先实现主旨题、细节题和词义猜测题三类，后续再扩展推断题、结构题和观点态度题。

#### 6.2.7 答题反馈与解析需求

答题反馈应成为学习环节，而不是简单显示“正确/错误”。每道题建议包含：

- 用户选择结果。
- 正确答案。
- 简要解析。
- 原文依据或段落定位。
- 错误选项为什么不合适。
- 相关重点词或句型提示。

反馈可以分层展示：先给出结果和正确答案，再允许用户展开详细解析，避免页面过于拥挤。

#### 6.2.8 阅读记录与诊断需求

阅读记录应具备诊断价值。建议记录：

| 数据 | 说明 |
|---|---|
| completedAt | 完成时间 |
| grossDuration | 总停留时长，包含用户离开或长时间无操作的时间 |
| activeReadingDuration | 有效阅读时长，排除页面不可见和长时间无操作时段 |
| readingSpeedWpm | 阅读速度，优先由词数和有效阅读时长估算 |
| accuracy | 答题正确率 |
| questionTypeAccuracy | 各题型正确率 |
| questionTimeSpent | 每道题答题用时 |
| optionChanges | 每道题选项修改次数，用于判断犹豫度和猜测风险 |
| newWordsCount | 新加入生词数量 |
| lookedUpWordsCount | 查看释义次数 |
| expandedAuxiliaries | 已展开的辅助内容，如段落、词汇、长难句提示 |
| highlights | 用户高亮标记对象数组，至少包含 highlightId、selectedText、tag、startOffset、endOffset、lastModified，便于同步合并 |
| rereadCount | 重读次数或重新打开次数 |
| lastPosition | 中断时的阅读位置 |
| mode | 精读、泛读或复习阅读 |
| estimatedLevel | 本次阅读后系统重新估算的阅读能力等级，可采用内部等级或后续接入标准化指标；实际更新时应考虑平滑处理 |
| selfDifficultyRating | 用户自评文章难度，如简单、适中、较难，可为空 |
| selfUnderstandingRating | 用户自评理解程度，如完全理解、大部分理解、部分不理解，可为空 |
| strategyModeMetrics | 策略模式耗时、目标完成数和历史对比数据，可为空 |
| strategyAnchorEvents | 阅读过程中策略锚点触发或点击记录，可为空 |
| difficultSentenceHintsOpened | 用户查看长难句提示的句子 ID 列表，可为空 |
| educationHintsShown | 已展示的适时教育提示类型和次数，用于控制提示频率 |
| educationHintsDismissed | 用户关闭或忽略的提示记录，用于减少重复打扰 |
| extensiveSelfCheck | 泛读完成后的快速自检结果，可为空 |
| highlightToWritingPrompted | 高亮内容是否已提示转为写作素材，可为空 |

这些数据可以用于展示阅读趋势、薄弱题型和材料推荐，也可以与单词模块联动分析“阅读中高频查词”。其中阅读速度应优先基于 `activeReadingDuration` 计算，避免用户离开页面或长时间无操作导致速度失真。

为避免高亮数据在多端同步时格式模糊，`highlights` 不建议只保存文本或 ID 列表，MVP 可采用对象数组的最小结构：

```text
highlights: [
  {
    highlightId: string,
    selectedText: string,
    tag: string,
    startOffset: number,
    endOffset: number,
    lastModified: string
  }
]
```

其中 `highlightId` 用于识别同一条高亮，`startOffset` 和 `endOffset` 用于回跳原文位置，`lastModified` 用于同步冲突判断。后续如支持删除和修改标签，可继续扩展 `operationType`、`version`、`sourceArticleId` 等字段。

#### 6.2.9 推荐典型流程

```text
进入阅读模块
  -> 按难度、主题或推荐列表选择文章
  -> 查看文章目标、预计阅读时间和重点词
  -> 阅读正文，按需查看生词、长难句、策略模式和段落朗读
  -> 在阅读中通过轻量锚点提醒略读、扫读、预测或总结策略
  -> 将重要生词加入单词本，并可高亮好句或疑问点
  -> 完成主旨、细节、词义猜测等阅读题
  -> 查看答题结果、解析和原文依据
  -> 保存阅读时长、正确率、生词量和阅读位置
  -> 系统更新阅读统计，并为后续材料推荐提供依据
```

#### 6.2.10 中断、重读与泛读场景

真实阅读不一定一次完成，因此应考虑：

- 中途退出后保留阅读位置、已完成题目、已加入生词、已展开辅助内容和高亮标记。
- 用户再次进入文章时，可以选择继续阅读、重新阅读或查看解析。
- 用户可以跳过题目进入泛读模式，但系统应将记录标记为泛读，不与精读正确率混淆。
- 重读文章时应记录为复习阅读，避免重复计入首次完成数量。
- 网络异常时应尽量保留本地阅读位置、已答题结果、辅助展开状态和高亮标记。

#### 6.2.11 模块联动与扩展需求

阅读模块不应成为数据孤岛，建议预留以下联动能力：

- 与单词模块联动：阅读中加入的生词进入单词记忆模块，阅读中的目标词掌握情况可反向影响词汇复习。
- 与写作模块联动：阅读主题、好句、高亮片段和表达可以成为后续写信练习素材。
- 与阅读笔记联动：用户高亮内容可汇总到“我的高亮笔记”，支持按标签筛选、回跳原文，后续可制作为复习卡片或句子仿写素材。
- 与写作素材引导联动：当用户累计高亮若干个“好句”或“表达”标签时，系统可在阅读完成页或写作入口轻量提示“你已收集 N 个好句，可尝试在写信练习中使用”，帮助用户发现阅读到写作的迁移路径。
- 与学习记录模块联动：阅读时长、正确率、阅读速度和生词数量进入学习统计。
- 与推荐能力联动：后续可根据用户难度表现、题型正确率、阅读速度、查词行为和主题偏好推荐文章。
- 内容生态扩展：后续可支持管理员维护文章、导入公开授权材料或生成时效性主题阅读，但必须处理来源和版权。长期运营中应注意常青内容与时效性内容的比例，建议以常青内容为主体，时效性内容作为补充和引流，避免素材库快速老化、维护成本过高。
- 与激励模块联动：阅读完成、连续阅读、题型提升、策略模式进步和泛读转精读可触发轻量成就或进度反馈。
- 与能力画像联动：阅读模块首页可用技能树或能力雷达展示主旨、细节、推断、词义猜测、速度等维度。

#### 6.2.12 个性化推荐与动态难度

阅读推荐不能只停留在“后续联动”的抽象表述，应至少明确冷启动和动态调整思路。

建议采用分阶段策略：

| 场景 | 推荐策略 |
|---|---|
| 新用户冷启动 | 通过自评水平、学习目标、考试方向、主题偏好或 1～2 篇短篇诊断阅读确定初始难度；诊断可跳过 |
| 无历史数据 | 默认推荐中低难度（如 CEFR A2-B1 或内部等级初级）、主题通用或用户感兴趣主题、预计阅读时间较短的文章，降低首次挫败感 |
| 有阅读记录 | 根据正确率、阅读速度、查词次数和题型表现调整后续文章难度 |
| 表现过强 | 连续高正确率且查词少时，逐步提高文章难度或引入推断题、结构题 |
| 表现吃力 | 正确率低、查词多或退出频繁时，降低难度、缩短篇幅或增加辅助提示 |

MVP 阶段可以先实现“自评水平 + 主题偏好 + 默认推荐 + 基于正确率的简单难度调整”，后续再扩展更精细的自适应推荐模型。诊断用短篇阅读应采用轻量设计，例如 2～3 段短文和 2～3 道核心题，允许用户跳过；用户跳过时，系统根据自评、主题偏好或默认难度进行模糊匹配，不强制完成诊断。若用户没有任何历史数据，且同时跳过诊断、自评和主题偏好选择，系统应以“中低难度（如 CEFR A2-B1 或内部等级初级）+ 通用兴趣主题 + 短篇材料”作为首次推荐兜底，避免首篇材料过难或过于随机。

开发阶段可将 `estimatedLevel` 作为可追踪的能力估算指标，但不建议单次阅读后直接大幅调整。更稳妥的做法是结合最近多次阅读表现，并通过平滑算法逐步更新。能力估算应以客观答题数据、有效阅读速度、查词次数和完成稳定性为主要依据；用户自评可以作为信心指数或认知偏差的诊断信号，用于触发学习策略提示和温和反馈，但不应直接覆盖客观数据或单独决定难度推荐。

当用户自评与客观表现长期不一致时，系统应优先做解释和支持，而不是机械调整难度。例如，用户多次自评“较难”但正确率、阅读速度和查词次数表现尚可，系统不应立即降低难度，而可推荐同等级中篇幅更短、主题更熟悉或辅助更充分的文章，并提示“你的正确率不错，可以继续保持当前难度”。如果用户自评“简单”但正确率持续偏低，则应优先引导回看解析、补充词汇或降低题型难度。这样可以把自评作为信心和体验信号，而不是替代客观能力判断。

#### 6.2.13 阅读激励与动机维持

阅读模块除记录数据外，还需要提供轻量正向反馈，帮助用户感知进步。激励机制应服务学习，不应喧宾夺主。

建议包括：

- 今日阅读完成提示。
- 连续阅读天数，并可通过运营配置支持周末宽限、补签卡等轻量容错机制；补签卡应通过真实学习行为获得，设置有效期和持有上限，避免滥用。
- 阅读总词数或总时长。
- 首次完成某类题型的徽章。
- 正确率提升提示。
- 阅读速度稳定提升提示。
- 生词复习回流提示，例如“本篇加入的 5 个生词已进入单词复习”。
- 阶段性目标，例如完成 10 篇校园主题阅读。

MVP 阶段至少可实现完成提示、阅读进度条和连续阅读天数；徽章、主题挑战和补签机制可作为后续扩展。连续记录规则应避免过度惩罚偶发中断，防止用户因一天中断而放弃长期学习。补签机制如后续启用，应限定获取条件，并明确为满足以下任一条件即可获得 1 张补签卡：连续 3 天完成阅读，或完成 1 次精读；同时设置最大持有数和过期时间。

长期使用中，激励机制还应关注学习心流和情绪保护，避免把用户只当作数据指标推动：

- 当系统检测到用户连续答错、频繁查词或阅读过程中多次停顿时，可优先提供一次“策略提示卡片”，例如“要不要先快速扫读一遍，再仔细阅读？”，而不是只简单降低文章难度。
- 可预留“挑战模式”作为用户主动开启的增强功能，例如限时重读、变速扫读或同篇文章复盘挑战。挑战模式不应默认开启，避免给基础较弱用户造成额外压力。
- 当用户中断 7 天以上再次进入时，不宜直接强调连续记录中断，可显示“欢迎回来，继续你的探索之旅吧！”并提供“一键回顾上次阅读内容”。
- 当用户重新打开此前正确率低于 50% 的文章时，可在文章开头给予轻量鼓励，例如“重新挑战，你现在的理解一定更深了”，减少羞愧感和焦虑感。

中断回归还可延伸为沉默用户唤醒策略。后续如果系统支持通知或邮件提醒，可以结合用户历史阅读偏好生成低频、个性化的召回内容，例如“你之前喜欢的科技主题有新文章了”。这类唤醒应严格控制频率，并提供关闭入口，避免把学习提醒变成打扰。MVP 阶段只需在站内回归页保留温和欢迎和上次内容回顾。

#### 6.2.14 显性阅读策略训练

阅读辅助解决的是“看不懂怎么办”，但阅读能力提升还需要教用户“如何阅读”。因此，模块应在合适位置显性提示阅读策略。

建议策略包括：

| 策略 | 使用场景 |
|---|---|
| 略读 Skimming | 阅读前快速把握标题、首尾段和文章主旨 |
| 扫读 Scanning | 查找日期、数字、人名、地点等细节信息 |
| 预测 Predicting | 根据标题、图片或首段预测文章内容 |
| 推断 Inferring | 根据上下文理解隐含信息 |
| 总结 Summarizing | 阅读后用一句话概括段落或全文 |
| 猜词 Guessing from Context | 根据上下文判断生词含义 |

MVP 阶段可以先在文章开始前给出 1 条阅读策略提示；阅读过程中的 1～2 个轻量策略锚点、解析中标注题目对应策略等能力可作为后续增强，避免首版阅读流程过重。策略锚点应与具体文本位置绑定，例如段首、转折句、数字信息或结论句，而不是只在文章开头显示一句抽象建议。为避免打断阅读，策略锚点默认不自动展开，建议以小图标或轻量下划线形式出现，用户 hover 或点击后再显示 tooltip/浮层说明。

#### 6.2.15 策略模式与主题偏好推荐

阅读策略提示如果只是文字，用户可能不知道如何执行。可将部分策略做成轻量交互模式：

- 略读模式：高亮标题、首尾段和每段首句，帮助用户快速抓主旨。
- 扫读模式：显示查找任务卡片，例如查找数字、人名、地点或时间。
- 策略模式可随时退出，不强制完成。
- 策略模式结束时提供轻反馈，例如扫读耗时、找到目标数量、是否比上次更快，并提供“再次练习”入口。
- 策略模式结束反馈应包含简短改进建议，而不只是展示数字。例如扫读耗时较长时提示“下次先找数字和大写专有名词”，略读主旨偏差时提示“先看标题、首段和每段首句再判断主题”。
- 解析中说明题目使用了哪种策略。

推荐系统还应融合主题偏好，避免只按难度推荐。

建议规则：

- 冷启动自评中增加“感兴趣主题”多选。
- 推荐分可综合难度匹配度、主题兴趣度和少量探索随机因子。
- 可提供可选“主题口味测试”：用 2～3 个短文片段让用户快速选择“有兴趣/无感”，将结果平滑合并到主题偏好模型。
- 用户主动搜索、完成、收藏某主题可作为正向信号。
- 用户频繁跳过某主题可作为弱负向信号。
- 保留少量跨主题探索，避免形成过滤气泡。

#### 6.2.16 主题口味测试与策略成效反馈

主题偏好如果只依赖用户一开始勾选，可能不准确，但也不应让新用户经历过长流程。主题口味测试应尽量融合进诊断、自评或首次阅读完成页，而不是作为强制独立步骤。默认实现路径建议放在诊断短篇阅读完成后的结果卡片底部，以可折叠问题形式询问“想看到更多这类主题吗？[是 / 否]”；如果用户跳过诊断，再在推荐首页提供“发现更多主题”轻量入口。也可以在第一篇阅读完成后提供补充测试：

- 展示 2～3 个极短片段，每个代表一个主题；如果用户已完成诊断短篇，可直接询问“想看到更多这类主题吗？”。
- 用户选择“有兴趣 / 无感”。
- 结果平滑合并到主题偏好，不直接覆盖原偏好。
- 保留少量探索推荐，避免用户陷入固定主题。

策略模式也应提供即时成效反馈：

- 扫读模式结束时显示耗时、目标定位数量和历史对比。MVP 阶段优先采用“比较条 + 箭头 + 文字说明”的最轻量形式，例如“本次比上次快 5 秒 ↑”“连续 3 次提升”；后续再扩展微型趋势图或进度环。
- 略读模式结束时提示是否抓住主旨。
- 根据本次表现给出下一次可执行建议，例如“先定位时间词再读整句”“遇到转折词后重点看后半句”“先用一句话概括段落再看选项”。
- 提供“再次练习”入口。
- 策略成效可进入学习记录，但不应过度排名化。历史数据不足 3 次时，只展示本次结果和鼓励文案，避免过度解读。

#### 6.2.17 泛读与精读转换路径

泛读和精读不应是单向割裂的流程。用户先泛读一篇文章后，如果觉得内容有价值，应能转为精读。

建议流程：

```text
泛读文章
  -> 标记感兴趣
  -> 完成 1～2 个快速自检问题或一句话理解确认
  -> 转为精读
  -> 补做阅读题
  -> 查看解析和原文依据
  -> 添加生词、高亮好句和疑问点
  -> 更新为精读记录
```

转换规则：

- 泛读记录保留阅读时长和进度。
- 转为精读后补充题目完成情况和正确率。
- 同一篇文章不重复计入首次阅读数量，但可更新阅读模式。
- 如果用户在精读中途退出且只完成部分题目，应记录为“精读未完成”，并保留已完成题目和阅读位置。
- 泛读完成页应提供低负担理解自检，例如“用一句话概括本文”“选择文章主旨”或“标记一个仍不确定的问题”。自检结果不计入正式正确率，主要用于帮助用户感知是否真的读懂。其中“标记一个仍不确定的问题”不建议单独开发新文本框，MVP 可复用高亮能力：用户选中相关句子或词组后自动添加“疑问”标签的高亮记录。
- 如果用户自检不确定、概括失败或标记疑问较多，系统可温和提示“可以转为精读看看解析”，而不是只放置一个静态转化按钮。
- 如果用户自检较好，也可以提示“你已经抓住大意，可选择做 3 道题验证细节”，让泛读到精读的转化更有意义。

MVP 阶段可以先在泛读完成页提供“继续做题转为精读”入口，并增加 1 个主旨选择或理解自评问题，避免泛读记录只停留在“读完即走”。

#### 6.2.18 有效阅读时长与过程数据

阅读速度如果只用文章词数除以总停留时长，容易受到用户离开页面、分屏、长时间无操作等因素影响，导致能力估算和推荐降级失真。因此，阅读模块应同时记录总停留时长和有效阅读时长。

建议规则：

- 使用 Page Visibility API 识别页面是否可见，页面不可见时暂停有效阅读计时。
- 当用户超过 2 分钟无滚动、点击、选词、答题等交互时，暂停有效阅读计时。
- 暂停时可显示轻量提示：“阅读已暂停，点击继续”。
- 最终同时记录 `grossDuration` 和 `activeReadingDuration`。
- 阅读速度 `readingSpeedWpm` 优先使用 `activeReadingDuration` 计算。
- 答题过程记录每题 `timeSpent` 和 `optionChanges`，用于识别快速准确、犹豫后正确或疑似猜测。

如果 MVP 阶段展示有效阅读时长，则至少应区分页面可见和不可见时间；若首版只展示基础阅读完成记录和正确率，有效阅读时长、无操作暂停和每题过程数据可作为后续增强。

#### 6.2.19 阅读辅助状态与主动标记

断点续读不应只保存文章位置，还应保存阅读辅助状态，否则用户重新进入文章时需要重复展开词义、长难句或段落定位，破坏连续体验。

建议在阅读进度对象中增加：

```text
expandedAuxiliaries: {
  paragraphIds: [],
  wordIds: [],
  sentenceIds: []
}
```

同时，阅读模块应支持轻量主动标记：

- 用户可选中文本并高亮。
- 高亮可添加简单标签，例如“好句”“论证”“疑问”。
- MVP 阶段优先支持高亮和标签，暂不强制开放自由笔记输入，降低实现复杂度。
- 高亮内容进入阅读笔记列表，并可作为写作模块素材引用。
- 当用户完成阅读且本篇有高亮好句时，完成页可提示“本篇已收集 N 条表达”，并提供“查看素材”或“用于写信练习”的轻量入口。若 MVP 阶段暂不实现高亮能力，可先在数据模型中预留字段；若实现高亮，入口可先跳转到高亮列表，不强制立即生成写作任务。为减少实现歧义，触发写作素材提示的初始阈值建议设为累计高亮好句或表达不少于 3 条，后续可根据运营数据或 A/B 测试调整。
- “我的高亮笔记”聚合视图应展示片段、标签、来源文章和上下文，支持按标签筛选和点击跳回原文；MVP 可先实现列表展示和原文跳转。
- 高亮数据本地优先保存，登录后可同步服务端。

#### 6.2.20 阅读后自我评估与能力可视化

阅读解析结束后，可以增加极简自我评估环节，帮助用户反思自己的理解状态，并为系统校准推荐提供额外信号。

建议设计：

- 自评文章难度：比预期简单、适中、较难。
- 自评理解程度：完全理解、大部分理解、部分不理解。
- 自评不强制填写，不影响完成记录。
- 当系统表现和用户自评差异明显时，例如正确率较低但用户认为完全理解，后续可用温和提示引导用户回看相关段落。
- 自评数据可辅助解释用户状态，但在 `estimatedLevel` 计算中权重应低于客观答题数据、有效阅读速度和查词次数；更适合作为“信心指数”或“认知偏差”维度，用于触发回看建议、策略提示和鼓励反馈。
- 自评提交后应给出即时微反馈，例如正确率低于 60% 但自评“完全理解”时提示“这篇似乎比预想的更有挑战？看看解析能帮你巩固关键点”；正确率较高但自评“部分不理解”时提示“你的作答很准确，语感可能比自我感觉更扎实”，帮助用户校准自我判断。
- 若用户连续自评“较难”但客观表现稳定，系统可优先提供鼓励和策略建议，例如“你觉得吃力，但细节题表现不错，下篇继续保持同级难度并开启段落提示”；若用户自评轻松但客观表现下降，则提示补看解析或选择更短文章。

阅读模块还可在个人页展示轻量技能树或能力雷达，例如主旨、细节、推断、词义猜测、速度五个维度。能力展示不必使用复杂数字，可采用“待加强 / 良好 / 优秀”等模糊档位，并解释系统为什么推荐下一篇文章。后续可进一步形成“能力叙事”，即根据能力雷达、题型表现和近期趋势生成一段自然语言学习总结，例如“你的细节定位能力进步很快，建议多尝试推断题来提升深层理解”。这类总结应保持鼓励和可行动，不宜用绝对化评价给用户贴标签。

#### 6.2.21 听读同步与高亮复用

阅读音频如果只播放声音，用户注意力容易游离。后续可增强为听读同步：

- 播放整篇或段落音频时，高亮当前朗读句子或词。
- 音频暂停、拖动时，文本高亮同步跳转，文章可使用 `scrollIntoView({ block: 'nearest' })` 自然滚动到对应句子。
- `audioAlignment` 建议采用句子级时间戳数组，例如 `[{ s: 0, st: 0.0, et: 2.5 }]`，分别表示句子序号、开始时间和结束时间；词级对齐可作为后续增强。
- 真人音频可通过字幕文件或句子级时间戳对齐；浏览器朗读可由 Web Speech API 或其他方案驱动。
- 时间戳可由 AI 辅助生成，但发布前应抽查准确性。
- 环境不支持时降级为纯音频，不阻断阅读。高亮句子应具备合适的可访问性提示，例如不抢夺用户焦点；可在文章容器外放置视觉隐藏但屏幕阅读器可访问的 `aria-live="polite"` 区域，当高亮句子变化时更新为当前句子的纯文本。

高亮笔记不应只停留在文章内。后续应提供“我的高亮笔记”聚合视图，支持按标签筛选、跳回原文，并可一键转为句子复习卡片或写作仿写素材。MVP 阶段若实现高亮，可先支持高亮列表和原文跳转；若暂不实现高亮，应在数据结构中预留最小联动字段：`cardGenerated`（是否已生成卡片）、`generatedCardId`（生成后的卡片 ID，可空）、`sourceArticleId`（来源文章 ID）、`writingPrompted`（是否已提示用于写作）。界面上如预留“生成复习卡”或“生成写作素材”入口但暂未实现，应置灰并提示“即将开放”。当用户累计一定数量好句或完成相关主题写作时，可提示“从你的阅读素材中挑一句试着仿写”，但提示频率应受控，避免打断阅读主流程。

#### 6.2.22 题目纠错与内容质量闭环

阅读题目、答案解析和原文依据可能由人工编辑或 AI 辅助生成，存在错误风险。为了形成内容质量闭环，应为用户提供低打扰纠错入口。

建议设计：

- 在题目解析区域底部提供“不显眼但可见”的“内容有误？”文字入口。
- 用户可勾选问题类型，如答案错误、解析不清、原文依据不符、题目表述有歧义。
- 提交时记录用户标识、题目 ID、文章 ID、时间和问题类型。
- 后台可按纠错频次、题目重要度和内容来源排序安排人工复核。
- 对 AI 辅助生成内容，纠错日志可作为后续审核和提示词改进依据。

除用户主动纠错外，还应预留基于行为数据的主动质量预警机制。后台可以统计同类题型下的异常指标，例如某道题的 `optionChanges` 明显偏高、错误率异常高、解析展开后仍频繁被反馈，或某篇文章的未完成退出率异常偏高。达到阈值后，系统可自动生成内容复核任务，提醒人工检查题目是否存在歧义、解析是否不清或文章难度标注是否失准。

为支持该机制，内容侧可预留 `qualityAlertReason`、`qualityAlertScore`、`reviewTaskStatus` 等字段。MVP 阶段可以先记录行为数据和用户纠错日志，后续再启用自动预警规则。

长期运营中，内容质量不应只依赖单条复核任务，还可为运营或教研人员提供内容健康度仪表盘。仪表盘可聚合展示被纠错频次最高的题目、异常退出率最高的文章、难度标注争议最大的材料、解析展开后仍高错率的题目等，帮助内容维护从“被动响应”转向“主动治理”。该能力属于后台运营增强，不应打断前台学习流程。

MVP 阶段可以先实现问题类型勾选，不要求用户填写长文本说明，避免增加负担。

#### 6.2.23 诊断结果引导与情绪安抚

冷启动诊断后，用户不应只看到冰冷等级或低分。诊断结果页应降低挫败感，并明确下一步路径。

建议诊断结果卡片包含：

- 当前估计水平，采用非标签化描述，例如“基础提升期”“稳步进阶期”。
- 推荐阅读主题和预计阅读时长。
- 简短鼓励语，说明系统会随练习动态调整推荐。
- 主行动按钮，例如“看看第一篇文章”。
- 可选入口，例如“重新选择兴趣主题”或“跳过诊断”。

避免直接强调“低级别”“错误很多”等负面表述，尤其是新用户首次进入时。

#### 6.2.24 内容供给与 AI 辅助边界

长难句提示、高质量题目和解析如果完全依赖人工生产，后期内容扩展速度会受到限制。可以考虑 AI 辅助生成，但必须明确边界。

建议原则：

- AI 可辅助生成文章初稿、题目草稿、长难句解释和答案解析。
- AI 生成内容应标记 `generatedBy` 和 `reviewStatus`。
- 面向用户发布前，应经过人工抽查或审核流程，尤其是题目答案、解析和版权来源。
- AI 不应生成来源不明的版权文本，也不应伪装成真实新闻或真实作者材料。
- 对低风险练习材料可采用“AI 生成 + 规则校验 + 人工抽查”；对考试材料或正式内容应采用更严格审核。
- 用户侧应优先看到学习内容本身，不必暴露复杂生成过程，但后台需保留内容来源和审核记录。

MVP 阶段可以先使用人工维护或明确授权材料；AI 辅助内容生成作为扩展能力预留字段和流程。

#### 6.2.25 隐私合规与数据治理

阅读行为数据能够反映用户学习能力和习惯，应谨慎处理。

建议要求：

- 用户阅读记录、查词记录、答题记录默认仅本人可见。
- 用户应能删除自己的阅读记录或清空学习数据；删除后的记录应标记 `deletedAt`，并从后续个性化推荐计算中排除。
- 如果学习数据用于推荐或算法优化，应尽量匿名化、聚合化处理；已删除记录不再用于个人画像，但历史匿名聚合统计可能无法完全回溯删除，应在隐私说明中明确。
- 如涉及多地区用户，应在后续部署阶段确认数据存储地域和适用法规。
- 对外展示排行榜或统计时，不应暴露用户具体阅读内容和错误明细。
- 日志中避免记录不必要的敏感信息。

- 用户长期积累的学习档案应具备可携带性。后续可规划“导出我的学习数据”功能，提供机器可读的 JSON 或 CSV 格式，导出范围可包括阅读记录、词汇库、答题记录、能力评估、高亮笔记和写作素材索引。
- 能力指标应预留外部标准映射字段，例如 `estimatedLevel` 后续可扩展 `cefrLevel`、`lexileEstimate`、`levelMappingVersion` 等可空字段。首版不必承诺精确对标外部证书，但数据结构应允许后续与 CEFR、Lexile 或学校、考试机构能力体系建立映射。
- 数据导出应由用户主动触发，并在导出前说明包含的数据范围。涉及笔友通信、AI 反馈或隐私设置的数据，应遵循最小必要原则，避免把他人信息一并导出。

MVP 阶段至少应做到不公开个人阅读记录，并在数据结构中区分用户可见记录和系统诊断日志；数据导出和外部标准映射可作为后续增强，但字段预留应在数据模型设计阶段考虑。

#### 6.2.26 多端同步与离线阅读

阅读场景常发生在网络不稳定或多设备切换中，因此需要提前规划多端和离线体验。

建议设计：

- 登录用户的阅读位置、答题结果、生词添加记录、辅助展开状态和高亮标记可同步到服务端。
- 未登录用户可先保存在本地，登录后再选择是否合并。
- 支持短期离线阅读缓存，至少缓存已打开文章的正文、题目和已展开辅助状态。
- 离线状态下允许继续阅读和本地保存答题，网络恢复后再同步。
- 同一文章在多端同时修改时，优先采用智能合并：阅读位置保留更靠后的进度，生词本、高亮标记和辅助展开状态合并新增数据；仅当同一道题在多端提交不同答案等无法自动合并的情况，才降级为用户确认或保留服务端最新。
- 对高亮、辅助展开等可增删状态，应记录实体 ID、`version` 或 `lastModified`、操作类型 `add/update/remove`；删除应作为状态更新同步，而不是简单物理删除，避免离线删除后又被其他端“合并新增”复活。
- 对同一实体的操作合并建议采用操作型日志：两端都删除则保留删除；一端修改一端删除时按时间戳或版本最后者生效；新增条目直接合并；同一高亮标签被多端同时修改等少数无法智能合并的情况，可采用最后写入获胜，并在同步历史中标注“部分状态可能未同步”。
- 音频资源可不强制离线缓存，音频不可用时不影响正文阅读。

MVP 阶段可以先实现本地阅读位置保存，后续再扩展账号同步和离线提交。

#### 6.2.27 专项非功能需求

| 类型 | 要求 |
|---|---|
| 性能 | 首屏目标加载时间建议控制在 1.5 秒以内；长文章应采用分段懒加载渲染，并在加载过程中使用骨架屏占位，避免页面跳动，段落切换应避免明显卡顿 |
| 移动端体验 | 移动端正文字号建议不低于 16px，行高不低于 1.6，主要按钮点击区域不小于 44px；相邻可点击元素间距建议不小于 8px，避免生词、朗读、高亮等按钮误触 |
| 异常处理 | 文章加载失败、题目加载失败、保存失败时应有明确提示和重试入口 |
| 空状态 | 所有可能为空的列表或区域都应提供友好引导，例如搜索无结果、筛选无匹配文章、生词本为空、高亮笔记为空时，应提示用户放宽条件或说明下一步操作 |
| 内容版权 | 阅读材料、图片和音频应记录来源和授权信息 |
| 可访问性 | 正文、题目、选项、解析应支持键盘访问和屏幕阅读器语义；高亮、辅助提示和状态变化不应只依赖颜色，应配合下划线、图标、标签或文本说明；后续可支持自定义字号、行距和背景色，如护眼色或夜间模式 |
| 专注阅读 | 阅读页应尽量减少导航栏、通知气泡和非核心按钮干扰；后续可提供沉浸阅读模式，一键隐藏非核心 UI，仅保留正文和少量可呼出的阅读工具 |
| 适时教育 | 策略模式、听读同步、长难句提示等功能应通过低频、非模态、场景相关的提示被发现；用户忽略或关闭后应降低出现频率 |
| 数据隐私 | 阅读时长、查词记录、答题记录、高亮标记属于个人学习数据，默认不公开；用于推荐或算法优化前应尽量匿名化或聚合化 |
| 浏览器兼容 | 核心阅读和答题流程应兼容主流现代浏览器，音频不可用时不影响阅读 |

#### 6.2.28 验收标准

阅读模块验收也应区分 MVP 最低标准和后续增强标准。MVP 应先保证“能选材、能阅读、能答题、能看解析、能保存记录”，策略模式、高亮素材、听读同步和泛读精读转换等能力可在后续迭代中逐步上线。

**MVP 最低验收：**

- 新用户可以通过默认推荐、自评水平或诊断入口获得第一篇阅读材料。
- 用户可以按难度、主题或关键词查找阅读材料；搜索无结果或筛选无匹配时，应有友好说明和下一步引导。
- 用户可以查看文章标题、正文、难度、主题、预计阅读时间、主题标签和重点词。
- 用户可以在阅读过程中查看重点词释义，并在需要时将生词加入单词本或待学习队列。
- 用户可以完成至少三类阅读题：主旨题、细节题、词义猜测题。
- 系统能展示答题结果、正确答案、简要解析和原文依据，并提供低打扰内容纠错入口。
- 阅读记录至少保存完成状态、阅读位置、基础阅读时长、正确率、新增生词数量和可选自评结果。
- 用户中途退出后，再次进入能够继续阅读或重新开始，并恢复已保存的阅读位置和已完成题目。
- 移动端阅读页面具备可读字号和合理行距，主要按钮点击区域应适合触控；重点词、辅助提示和状态变化不应只依赖颜色。
- 文章加载失败、保存失败或题目加载失败时有明确提示，不应让页面长期空白或让用户误以为结果已经保存。
- 阅读材料应有来源或版权字段规划；若首版未接入音频，音频能力不应阻断文字阅读流程。

**后续增强验收：**

- 用户可以看到至少一条阅读策略提示，并可在阅读过程中看到与文本位置相关的轻量策略锚点；策略锚点默认以小图标或下划线提示，hover 或点击后展开说明。
- 用户可使用至少一种轻量策略模式，策略结束后能看到耗时、目标完成情况和下一次可执行改进建议。
- 长难句或复杂结构句应具备轻量提示入口，用户可以主动展开查看句子主干或结构说明。
- 阅读辅助功能可具备适时教育机制，让用户在相关场景发现策略模式、听读同步或长难句提示，同时不打断阅读主流程。
- 用户在泛读完成页可完成低负担理解自检，并可选择转为精读补做题目；系统应将后续题目结果合并到同一篇文章的精读记录中。
- 用户在精读中途退出且只完成部分题目时，系统应记录为“精读未完成”，并保留阅读位置、已完成题目和已保存辅助状态。
- 用户可以对文本进行高亮标记，标记内容可进入阅读笔记或写作素材，并支持回跳原文。
- 高亮条目可预留 `cardGenerated`、`generatedCardId`、`sourceArticleId`、`writingPrompted` 等生成复习卡或写作素材的接口字段；如果界面预留入口但暂未实现，应置灰并说明。
- 当用户累计高亮好句或表达达到初始阈值时，阅读完成页或写作入口可有轻量提示，引导用户查看素材或用于写作练习。
- 阅读页面应控制视觉干扰，后续沉浸阅读模式应有明确入口规划。
- 听读同步时间戳作为后续增强字段预留，并应明确句子级时间戳格式；环境不支持时降级为纯文本或纯音频。

### 6.3 笔友写信练习模块

#### 6.3.1 功能目标

笔友写信练习模块用于帮助用户在真实但受保护的社区环境中进行英文书面交流训练。该模块不应再以系统预设的虚拟笔友作为主流程，而应围绕真实学习者之间的收信、写信、回信和互助反馈建立学习闭环。

该模块的核心价值不是把信件当成普通作文题，也不是让 AI 替代真实交流，而是为用户提供安全的笔友身份、合适的匹配机制、可靠的通信规则、可选的写作辅助和积极的同伴互动。用户在模块中既能练习英语表达，也能通过真实回应获得持续写作动力。

考虑到真实用户通信涉及账号、权限、安全审核和运营责任，MVP 阶段应先完成“受控一对一写信闭环”的验证：可以采用小范围用户池、教师/测试账号或受控邀请关系，不直接开放大规模社区。完整公开推荐、搜索发现、主题公开信、同伴互评和成长统计可作为后续增强。

#### 6.3.2 核心学习路径

笔友写信练习模块应围绕真实笔友关系设计学习路径：

1. **建立安全身份**：用户创建昵称、英语水平、兴趣标签、学习目标和隐私可见范围。
2. **发现或匹配笔友**：用户通过系统推荐或自主搜索找到水平、兴趣和通信频率较合适的学习者。
3. **双向确认关系**：双方确认后建立笔友关系，未确认前不能直接发送私人信件。
4. **阅读真实来信**：用户在收件箱中查看来自笔友的新信件，并了解双方通信上下文。
5. **使用写作支架**：用户在写信时可打开写作助手，查看思路启发、常用表达、段落结构和友好沟通检查清单。
6. **发送前自愿检查**：用户可选择使用拼写、语法、用词或润色检查，但系统反馈只作为私有辅助，不替代用户表达。
7. **安全发送信件**：发送前系统进行风险词、联系方式和骚扰内容检查，通过后进入真实通信。
8. **收到真实回信与互助反馈**：笔友可回信、点赞、感谢，也可在用户请求时提供友好的语言建议。
9. **沉淀成长记录**：系统记录通信历史、写作数量、收到信件数量、常见错误趋势、优秀表达收藏和同伴积极反馈。
10. **管理关系生命周期**：关系可持续通信、休眠、重新激活或友好结束，降低真实社交中的尴尬和流失。

#### 6.3.3 基础功能

| 功能点 | 说明 | 优先级 |
|---|---|---|
| 用户档案 | 用户以昵称、年龄组、英语水平、兴趣标签、学习目标等形成受保护的笔友身份 | 必须有 |
| 隐私设置 | 用户可设置哪些档案信息对笔友可见；隐身、仅接受邀请等细分模式可后续完善 | 必须有 |
| 笔友推荐 | 根据英语水平、兴趣标签、学习目标和通信频率意愿推荐笔友；MVP 可先使用受控测试关系或基础推荐 | 应该有 |
| 搜索与筛选 | 支持按水平、兴趣、地区或通信语言等条件发现笔友；地区字段是否开放需后续确认 | 可以有 |
| 主题公开信 | 作为一对一匹配的低压补充，用户可围绕平台话题发布受控公开信，其他用户以回信方式互动 | 可以有 |
| 双向确认 | 双方确认成为笔友后才能开始通信，任一方可解除关系 | 必须有 |
| 收件箱 | 按笔友分组展示新信件、未读状态和通信历史 | 必须有 |
| 信件编辑器 | 支持用户撰写英文信件，显示字数、草稿保存状态和收信对象 | 必须有 |
| 草稿保存 | 用户中途退出时自动保存未完成内容 | 必须有 |
| 写作助手 | 提供思路启发、常用表达、段落结构和友好沟通检查清单 | 应该有 |
| 发送前检查 | 用户可自愿使用拼写、语法、用词和润色检查，反馈不公开、不强制 | 应该有 |
| 内容安全审核 | 对敏感词、联系方式、骚扰内容和垃圾信息进行拦截或提示 | 必须有 |
| 举报与解除关系 | 每封信可举报，用户可屏蔽或解除笔友关系 | 必须有 |
| 同伴互助反馈 | 用户可在学习风格中设置是否欢迎友好语言建议，收信人可通过模板化方式提供少量帮助、点赞或感谢 | 可以有 |
| 首信破冰 | 双向确认后提供共同起点、话题灵感和空白信纸，降低谁先开口的压力 | 应该有 |
| 通信历史 | 展示与不同笔友的往来信件、关系状态和最近互动时间 | 必须有 |
| 成长统计 | 展示写作数量、收到信件数量、错误趋势、优秀表达收藏、被点赞/感谢次数等 | 可以有 |
| 社区规范 | 首次通信前展示友好通信指南和违规处理规则 | 必须有 |
| 关系生命周期管理 | 支持关系阶段提示、话题推荐、休眠归档和友好结束通信；MVP 至少支持 active/ended/blocked 等基础状态 | 应该有 |

#### 6.3.4 用户身份与档案系统

真实社区场景下，用户需要以安全的学习者身份出现，而不是面对系统生成的虚拟人物。用户档案既要帮助匹配笔友，也要避免暴露过多个人信息。

用户注册或首次进入笔友模块时，应填写或选择以下信息：

| 字段 | 说明 | 是否公开 |
|---|---|---|
| userId | 用户唯一标识，系统内部使用 | 不公开 |
| nickname | 昵称，不要求使用真实姓名 | 对笔友公开 |
| ageGroup | 年龄组，如初中、高中、大学、成人等，不展示具体年龄 | 可选公开 |
| englishLevel | 英语水平，可由自评或简单测试得到 | 对笔友公开 |
| interests | 兴趣标签，如旅行、音乐、电影、运动、阅读等 | 对笔友公开 |
| learningGoal | 学习目标，如练习写作、准备考试、日常交流等 | 可选公开 |
| communicationLanguage | 希望使用的通信语言或主要练习语言 | 对笔友公开 |
| preferredFrequency | 期望通信频率，如每周 1 次、每周 2-3 次等 | 对笔友公开 |
| learningStyle | 学习风格偏好，如自然交流型、开放学习型 | 对笔友公开或半公开 |
| shareableTopics | 愿意在信中交流的话题范围 | 对笔友公开 |
| avatar | 卡通头像或系统插图库头像，不强制上传真实照片 | 对笔友公开 |
| visibilityMode | 可见范围，如公开推荐、仅接受邀请、隐身 | 不完全公开 |

系统应在档案页明确说明“哪些信息会被其他笔友看到”。用户应能随时修改兴趣、学习目标、通信频率、学习风格和隐私设置。若项目面向未成年人，需在后续设计中单独确认监护人同意、隐私展示范围和通信限制等合规要求。

为降低用户每次写信前勾选“请求帮助”的心理压力，语言帮助偏好可以放在账户级学习档案中表达。例如，用户可以选择“自然交流型：不主动请求纠错，只希望自然聊天”或“开放学习型：欢迎友好的语言小建议”。这种设置比每封信临时勾选更稳定，也更像用户学习风格的一部分。发送单封信时仍可允许临时关闭或开启，但默认应尊重用户档案中的偏好。

#### 6.3.5 笔友匹配机制

笔友匹配应服务于学习效果和交流体验。若双方英语水平差距过大、兴趣完全不同或通信频率不一致，容易造成挫败或中断。因此，系统应同时提供“系统智能推荐”和“自主搜索发现”两种路径。

MVP 阶段不要求一次性完成完整匹配系统。为验证核心写信闭环，可以先使用受控测试账号、小范围邀请关系或基础规则推荐，只要能保证双方关系确认、通信权限和安全审核闭环成立即可。完整搜索筛选、复杂推荐排序和大规模冷启动运营应放到后续阶段。

智能推荐可参考以下条件：

- 英语水平相近，避免一方完全看不懂对方来信。
- 至少有 2 个以上共同兴趣标签，便于开始话题。
- 学习目标不冲突，例如都希望练习日常交流或写作表达。
- 通信频率意愿接近，减少一方长期等待的情况。
- 用户隐私模式允许被推荐。

自主搜索可支持按英语水平、兴趣、地区、通信语言、学习目标等筛选，但不作为 MVP 必须项。地区信息是否开放应谨慎处理，默认可采用宽泛地区或文化背景描述，不鼓励展示具体地址。

匹配成功后，系统应提供可选的破冰话题或第一封信模板，但不能限制用户自由表达。正式通信前应采用双向确认机制：一方发出笔友邀请，另一方同意后才建立关系。任何一方都可以随时解除关系，解除后对方不能继续发送信件。

双向确认后容易出现“首信沉默”，即双方都等待对方先开口。系统应在关系建立后自动向双方发送轻量通知，例如“你们已经是笔友了，可以从下面 3 个话题开始第一封信”。同时可生成一张带话题提示的空白信纸，展示共同兴趣和可选开场句，形成共同起点，降低谁先主动的社交尴尬。


#### 6.3.6 匹配质量与低压公开信补充

高质量匹配与社区活跃度之间存在天然矛盾。水平相近、兴趣相合、频率接近的匹配更容易形成长期关系，但也可能让用户只和少数笔友深度通信；同时，新用户或兴趣较窄的用户如果暂时找不到理想笔友，可能在建立关系前就流失。

因此，笔友模块可在一对一通信之外预留一个轻量的“主题公开信”机制，作为匹配不足时的低压补充。该机制不应发展成开放广场式大型社区，而应限定在平台设置的学习话题下运行：用户围绕一个主题写一封公开信，其他用户可以用“回信”的方式互动。公开信可以帮助用户先参与写作，也能为后续一对一匹配提供更多判断线索。

主题公开信应遵循以下约束：

- 只围绕平台提供的英语学习话题或安全生活话题展开，不开放任意动态流。
- 发布前同样经过敏感词、联系方式和不当内容检查。
- 用户可选择是否允许他人基于公开信发起笔友邀请。
- 公开信默认展示昵称、英语水平和兴趣标签等有限信息，不展示过多个人档案。
- 如果用户不希望公开参与，应仍能只使用一对一笔友匹配。
- MVP 阶段可先不实现，若匹配冷启动困难明显，再作为受控补充功能加入。


#### 6.3.7 真实收信与回信流程

笔友模块的核心循环应从系统生成任务改为真实收发信。用户进入模块后，首先看到收件箱，而不是作文题列表。收件箱按笔友分组展示未读信件、最近来信时间、关系状态和是否有待回复内容。

用户打开一封信后，可以查看上下文历史，并点击“回信”。写信界面顶部应显示“正在给 [笔友昵称] 回信”，帮助用户明确真实交流对象。写作助手作为侧边栏或折叠面板存在，提供通用句型、情感表达建议和话题灵感，而不是系统强制规定的写作要点。

发送前应有二次确认，例如“确定要发送这封信吗？发送后无法撤回”。确认前系统应进行内容安全检查，重点识别联系方式、敏感词、骚扰内容和明显垃圾信息。送达或已读状态可以作为可选功能；若启用已读状态，应允许用户关闭，避免给双方造成回复压力。

如果笔友长时间未回复，例如超过 7 天，系统可以轻量提示：“你的笔友可能正在忙，要不要先去发现新的笔友？”提示应避免催促或指责对方。

#### 6.3.8 写作支架与沟通检查清单

原有写作支架系统应保留，但需要从“完成系统任务”调整为“支持真实沟通”。由于真实信件内容无法预知，系统不应再设置由平台定义的必须回应点，而应提供通用的友好沟通检查清单。

写作支架建议包括：

| 支架类型 | 示例说明 |
|---|---|
| 思路启发 | 当用户不知道写什么时，提示可以回应近况、分享经历、表达感受、提出问题 |
| 常用表达 | 提供问候、感谢、赞同、安慰、邀请、道歉、结尾祝福等常用英文表达 |
| 段落结构 | 建议使用问候、回应对方、分享自己的内容、提出新问题、友好结尾的结构 |
| 情感表达 | 帮助用户表达开心、惊讶、遗憾、鼓励等常见情绪 |
| 话题灵感 | 基于双方共同兴趣推荐可聊话题，如最近看的电影、喜欢的音乐、学习计划 |
| 第一封信模板 | 为新关系提供开场模板，但允许用户自由修改 |
| 友好沟通检查清单 | 提醒用户检查是否回应对方问题、是否主动问了新问题、语气是否友好、是否避免过度纠错 |

友好沟通检查清单可以包含：

- 我是否回应了对方提出的问题？
- 我是否分享了一点自己的真实想法或经历？
- 我是否主动提出了一个新问题，方便对方继续回信？
- 我的语气是否友好、尊重、不过度冒犯？
- 如果我指出语言问题，是否足够温和，并且不超过对方请求的范围？

用户打开写作支架的行为不应被视为能力不足。支架只是辅助表达，不能替用户自动生成整封信。

写作支架还需要平衡“鼓励输出”和“避免模板化垃圾内容”。第一封信模板和常用表达可以降低启动难度，但如果用户几乎整封信都直接复制模板，会降低收信人的真实交流体验，也不利于用户形成自己的表达。系统可对模板使用比例做轻量检测，例如当一封信中模板化内容占比明显过高时，在发送前给出温和提示：“这封信看起来很像模板，笔友可能更期待听到你自己的故事。要不要再花一分钟，加上几句你的真实想法？”该提示不应强制拦截，只作为质量提醒。

#### 6.3.9 发送前语言反馈与润色辅助

即时语言反馈应保留，但定位需要调整。AI 或自动检查工具只能作为发送前的私有写作辅助，不能替代真实语言交流，也不能强制用户接受修改。最终发送的内容应由用户确认，并保持用户自己的表达。

可提供以下可选工具：

| 工具 | 说明 | 展示原则 |
|---|---|---|
| 拼写检查 | 标记明显拼写错误并给出候选修改 | 默认私有，不公开给笔友 |
| 基础语法检查 | 提示主谓一致、时态、冠词、单复数等常见问题 | 用户自行决定是否采纳 |
| 用词建议 | 提醒重复、过于生硬或不自然的表达 | 不自动替换原文 |
| 润色建议 | 给出更自然的表达版本 | 必须由用户手动确认 |
| 语气检查 | 提醒语气是否可能过于生硬或不礼貌 | 只作为发送前提醒 |

反馈结果不应强制公开，也不应在笔友端展示“这封信经过 AI 修改”等标签。若反馈生成失败，系统应保留草稿并允许用户继续发送或稍后重试。若使用外部 AI 能力，应在隐私说明中告知用户信件内容可能被用于生成检查结果，并提供不开启检查的选择。

#### 6.3.10 同伴互评与学习支持

真实笔友关系不应被设计成互相批改作文，但社区可以提供可选、友好、低压力的同伴学习支持。

语言帮助请求不宜只放在每次发送前临时勾选，因为这可能让基础用户感觉自己在暴露弱点。更合适的方式是将其设计为账户级学习风格偏好，并允许单封信临时调整。例如：

- 自然交流型：不主动请求纠错，只希望自然聊天。
- 开放学习型：欢迎友好的语言小建议。
- 语法关注型：希望对方指出 1-2 个明显语法问题。
- 表达提升型：欢迎更地道的表达建议。
- 单封信临时设置：本次暂时不需要语言建议，或本次希望得到一点帮助。

收信人看到对方欢迎语言建议后，可以点击“提供帮助”，但系统不应直接提供一个无限制的大文本框。更合适的方式是提供少量友好反馈模板，引导反馈保持正向、聚焦和低压力。例如：“我特别喜欢你这句……”、“关于……，有个更自然的说法是……”、“如果只改一个小地方，我建议……”。系统也可以自动标出疑似拼写或语法问题，收信人选择认可后再加入回信，降低同伴反馈门槛。

同伴反馈模板还需要避免“去人性化”。如果用户只需点击按钮就发送系统生成的建议，接收方可能会感觉对方没有真正阅读自己的信。因此，模板化反馈应保留一个很小的人工输入环节。例如，系统提供“我特别喜欢你这句：_____”或“关于 _____，我觉得可以这样说：_____”，其中至少一个空格需要用户手动填写。这个动作不会显著增加负担，却能让反馈更像真实交流，而不是机械批改。

为鼓励积极互动，系统可提供点赞和感谢机制。收信人可以给某个表达点赞，也可以发送“谢谢你的信，它让我很开心”这类轻量感谢。点赞、感谢和有帮助的反馈可进入用户成长统计，但不应形成公开排名，避免把社区变成竞争场。

#### 6.3.11 安全通信与内容审核

安全是笔友社区能否长期运行的基础。模块应在产品流程中建立多层安全保障，至少包括内容过滤、举报处理、通信限制、隐私控制和未成年人保护预案。

| 安全能力 | 说明 | 优先级 |
|---|---|---|
| 敏感词过滤 | 对辱骂、色情、暴力、歧视等不当内容进行拦截或提示 | 必须有 |
| 灰度讨论提醒 | 对语气激烈但未明显违规的争议话题，优先给出尊重沟通提醒，而不是直接拦截 | 应该有 |
| 联系方式检测 | 检测邮箱、电话、社交账号等联系方式，避免引导站外联系 | 必须有 |
| 举报机制 | 每封信提供举报入口，原因包括骚扰、不当内容、身份冒充、垃圾信息等 | 必须有 |
| 屏蔽与解除关系 | 用户可屏蔽某个笔友或解除关系，解除后对方不能继续发送信件 | 必须有 |
| 通信频率限制 | 新用户或异常用户每日发送信件数量应有限制，防止骚扰和垃圾信息 | 应该有 |
| 审核队列 | 被系统拦截或用户举报的信件进入审核流程 | 应该有 |
| 误判申诉 | 用户可对误拦截内容提交申诉或修改后快速重试，系统记录处理结果 | 应该有 |
| 内容权限控制 | 信件内容默认仅收发双方可见；如设置管理员或审核角色，其仅在举报、安全审查或合规需要时查看 | 必须有 |
| 未成年人保护 | 如果服务对象包含未成年人，应设置更严格的隐私、匹配和通信限制 | 条件必需 |

安全提示应尽量清晰，并注意语气从“警告/处罚”转向“保护/提醒”，避免用户感觉自己在和审查系统对抗。例如，与其提示“发送失败，信件包含禁止内容”，不如提示：“为了保护你的隐私安全，我们暂时帮你拦截了这封信。请检查一下，是否不小心包含了邮箱、电话或社交账号？修改后可以再发送。”

同时，内容审核应预留误判处理机制。用户认为系统误判时，可以提交申诉或请求复核；系统应记录拦截原因、规则类型和处理结果，便于后续修正规则。对于低风险误判，应允许用户修改后快速重试，不应让一次误判打断整封信的写作流程。违规处理规则应在社区规范中提前说明。

对于有争议但不明显违规的内容，系统应避免简单使用敏感词拦截。用户可能会讨论环境保护、学习压力、文化差异等较复杂的话题，表达可能比较强烈，但这不一定等同于不安全内容。此类灰度场景更适合采用“关系维护提醒”，例如在编辑器顶部轻量提示：“你们的讨论很有深度。记得保持对彼此观点的尊重，这会让交流更长久。” 这类提示不评价观点对错，只提醒沟通方式，目标是保护长期笔友关系，而不是压制表达。

#### 6.3.12 社区规范与新手引导

笔友社区需要清晰的行为边界。用户首次通信前，应看到简短、可视化的友好通信指南，并确认理解基本规则。

友好通信指南可以包括：

- 使用尊重、友善的语气。
- 真诚回应对方内容，不只发送模板化句子。
- 主动提出问题，帮助对话继续。
- 不索要真实姓名、电话、邮箱、社交账号、具体住址等隐私信息。
- 不发送骚扰、歧视、暴力、色情或广告内容。
- 不在对方未请求时过度纠正语法。

系统还应提供第一封信模板、友好结尾模板和破冰话题建议，帮助新用户降低开场压力。违规处理应透明，包括提醒、警告、限制通信、封禁等层级，具体规则可在后续产品规范中细化。

#### 6.3.13 真实关系的生命周期管理

真实笔友关系会经历开始、熟悉、暂停和结束。系统应尊重这种节奏，而不是只记录一次写信行为。

建议支持以下关系状态：

| 状态 | 说明 |
|---|---|
| invited | 已发送邀请，等待对方确认 |
| active | 双方已成为笔友，可以正常通信 |
| waitingReply | 一方已发送信件，等待对方回复 |
| dormant | 长时间未通信，关系进入休眠 |
| ended | 任一方主动结束关系 |
| blocked | 因用户屏蔽或安全原因无法继续通信 |

关系页面可展示轻量提示，例如“这是你们的第 3 封信”“你们已经通信 2 周啦”。如果用户不知道接下来聊什么，可以点击“话题灵感”，系统根据双方共同兴趣推荐话题。

当用户想结束通信时，系统应提供“结束通信”功能，并可给出一封友好的结束信模板。超过一定天数未回复的关系可自动进入休眠状态，用户可选择重新激活或结束关系。这样可以减少用户因内疚、尴尬或长期等待而直接流失。

#### 6.3.14 数据记录需求

为支持真实通信、匹配、安全和成长统计，笔友模块应记录以下几类数据。具体字段可在详细设计阶段继续拆分。

| 数据类别 | 关键字段或说明 |
|---|---|
| 用户档案 | userId、nickname、ageGroup、englishLevel、interests、learningGoal、preferredFrequency、learningStyle、visibilityMode、avatar |
| 匹配关系 | relationId、userAId、userBId、matchReason、status、createdAt、lastInteractionAt、endedAt |
| 信件记录 | letterId、relationId、senderId、receiverId、content、wordCount、draftStatus、sentAt、readAt、safetyStatus |
| 写作辅助记录 | checkUsed、suggestionTypes、acceptedSuggestionCount、polishUsed、templateUsageRatio、templateQualityPromptShown、userConfirmedFinalContent |
| 同伴反馈 | feedbackId、letterId、helperId、feedbackType、feedbackTemplateId、feedbackManualInput、feedbackContent、thanksCount、likedExpressionIds |
| 公开信记录 | publicLetterId、topicId、authorId、content、replyCount、safetyStatus、allowInvite、createdAt |
| 安全记录 | reportId、letterId、reportReason、reviewStatus、actionTaken、grayDiscussionSignal、relationshipNudgeShown、createdAt |
| 成长统计 | sentLetterCount、receivedLetterCount、activePenPalCount、likedCount、thanksCount、collectedExpressions、commonIssueTrend |

信件内容属于敏感学习和交流数据，默认不公开展示。历史记录只对通信双方可见；举报、审核或合规场景下的访问权限应单独控制并留下操作记录。若后续接入云端同步或 AI 检查，应在隐私说明中解释数据用途、保存范围和用户可关闭的选项。

#### 6.3.15 典型流程

MVP 阶段可先采用受控关系建立流程：

```text
首次进入笔友模块
  -> 创建安全笔友档案并设置基础隐私范围
  -> 查看友好通信指南
  -> 通过受控测试关系、基础推荐或邀请建立笔友关系
  -> 查看收件箱或给已确认笔友写第一封信
  -> 按需打开写作助手和发送前检查
  -> 自动保存草稿
  -> 发送前进行安全检查与二次确认
  -> 信件发送成功并进入通信历史
  -> 用户可举报、屏蔽或解除关系
```

完整版本可扩展为：

```text
首次进入笔友模块
  -> 创建安全笔友档案并设置隐私范围
  -> 查看友好通信指南
  -> 通过系统推荐或自主搜索发现笔友
  -> 发送笔友邀请
  -> 对方确认后建立笔友关系
  -> 系统发送关系建立通知、共同兴趣和首信破冰话题
  -> 查看收件箱或给新笔友写第一封信
  -> 按需打开写作助手和发送前检查
  -> 发送前进行安全检查与二次确认
  -> 信件发送成功
  -> 对方阅读、回信、点赞、感谢或提供可选语言帮助
  -> 用户在通信历史和成长统计中查看往来记录
  -> 关系持续、休眠或友好结束
```

#### 6.3.16 异常与空状态

- 用户未完成档案时，应提示先补充昵称、英语水平、兴趣标签和隐私设置。
- 暂无推荐笔友时，应提示调整兴趣、通信频率或稍后再试，并可引导用户尝试受控的主题公开信。
- 笔友邀请被拒绝或超时未处理时，应给出平和提示，并推荐继续发现其他笔友。
- 用户未建立笔友关系时，收件箱应展示“还没有笔友，可以先发送一个友好的邀请”。
- 信件为空或字数过少时，应提示补充正文内容。
- 信件包含联系方式、敏感词或疑似骚扰内容时，应阻止发送或进入审核，并说明原因。
- 信件语气较激烈但未明显违规时，应优先展示尊重沟通提醒，不应直接按违规内容处理。
- 信件模板化内容比例明显过高时，应在发送前给出温和提醒，引导用户补充个人想法，但不直接禁止发送。
- 自动保存失败时，应提示用户手动保存，并避免覆盖本地已输入内容。
- 发送前语言检查失败时，不影响用户保存草稿；是否继续发送应由用户确认。
- 举报提交后，应提示用户可先屏蔽对方，系统会继续处理举报。
- 双向确认后双方长时间都未发送首信时，应提示破冰话题、第一封信模板或共同兴趣，不直接催促某一方。
- 关系休眠时，应提供重新激活、结束关系或发现新笔友的入口。

#### 6.3.17 验收标准

- 用户可以创建不暴露真实姓名的笔友档案，并清楚看到哪些信息会对其他笔友可见。
- 用户可以设置基础隐私范围；公开推荐、仅接受邀请或隐身等细分模式可作为后续增强。
- MVP 阶段可以通过受控测试关系、基础推荐或邀请建立笔友关系；完整公开推荐和搜索筛选不作为首版必须验收项。
- 当高质量匹配不足时，系统后续可提供受控主题公开信作为低压写作补充，但不扩展为开放广场式社区。
- 未建立关系的用户之间不能直接发送私人信件。
- 用户可以在收件箱中按笔友查看未读信件和历史通信。
- 用户可以在写信页面使用思路启发、常用表达、段落结构和友好沟通检查清单。
- 写作支架不再包含由系统预设的必须回应点，而是引导用户自行判断是否回应对方问题。
- 当信件过度依赖模板时，系统应在发送前温和提醒用户补充真实想法，避免社区出现大量低质量模板信。
- 用户可以在发送前自愿使用拼写、语法、用词或润色检查，反馈结果默认不公开、不强制采纳。
- 用户发送信件前会经过内容安全检查和二次确认；内容安全审核能拦截明显违规内容，例如联系方式和明显敏感词。
- 安全误判时，用户已写内容不应丢失，应能修改后重新提交；申诉和人工复核可作为后续增强流程。
- 对争议话题或激烈表达，系统后续应优先采用非评判性的关系维护提醒，只有明显违规时才拦截或进入审核。
- 每封信都应提供举报入口，用户可以屏蔽或解除笔友关系。
- 用户解除或屏蔽笔友关系后，对方不能继续向其发送信件，通信权限应随关系状态及时生效。
- 用户可以在账户级学习风格中设置是否欢迎友好语言建议，并可在单封信中临时调整；MVP 可先保留偏好字段，后续再完善同伴反馈流程。
- 收信人后续可以选择是否提供友好的同伴反馈，反馈入口应优先使用正向模板或少量聚焦建议，避免长篇批改。
- 同伴反馈模板应至少保留一个需要用户手动填写的内容点，避免完全机械化发送系统建议；该能力不作为 MVP 必须项。
- 用户可以查看与不同笔友的通信历史；收到的信件数量、被点赞/感谢次数、优秀表达收藏和常见错误趋势可后续进入成长统计。
- 双向确认后系统可以提供关系建立通知、共同兴趣和首信破冰提示，降低首信沉默概率。
- 长时间未回复的关系后续可以进入休眠状态，用户可以重新激活或友好结束通信；MVP 至少应支持正常通信、解除关系和屏蔽。
- 当匹配、发送、保存、检查或举报流程失败时，系统应给出明确提示，并保证用户已写内容不丢失。

### 6.4 学习记录模块

#### 6.4.1 功能目标

学习记录模块不应只展示“学了多少”的静态数字，而应帮助用户从数量、质量、趋势、目标差距和学习行为几个角度理解自己的学习进度。用户进入该模块后，应能回答以下问题：今天是否完成计划、最近是否在持续进步、哪些内容掌握较弱、时间投入是否稳定，以及下一步应该优先学习或复习什么。

因此，学习记录模块的目标是形成“记录学习行为 -> 汇总学习结果 -> 展示进度变化 -> 给出行动提示”的反馈闭环。MVP 阶段前端应优先展示今日任务完成度、连续学习天数和模块完成数量三个核心信号；学习时长、目标完成对比、本周趋势、跨模块关联、脱敏历史和日志清理策略等可先作为后台统计口径、数据模型或接口预留，不要求首版全部可视化，避免学习记录页拖慢核心功能上线。

#### 6.4.2 设计原则

1. **数量与质量并重**
   统计不只展示已学单词数、阅读数量和写作次数，还应体现正确率、掌握度、完成率、学习时长等质量和效率指标。

2. **体现时间趋势**
   除今日数据外，系统后续应支持本周维度的学习趋势，并预留近 90 天日统计和近 12 个月月度汇总的查询能力。MVP 阶段可以先保证后台统计口径和数据结构可扩展，不强制在前端展示完整趋势图。

3. **目标对比明确**
   学习记录应与每日或每周目标关联，展示已完成、未完成和超额完成情况，避免用户只看到孤立数字。

4. **统计口径可解释**
   页面应说明“已掌握”“待复习”“完成阅读”等指标的判断标准。关键阈值不应分散写死在页面中，而应作为统一配置维护，便于后续调整。

5. **从记录到建议**
   当待复习数量过多、正确率下降、连续学习中断或某类任务长期缺失时，系统应给出轻量提示和下一步入口，但建议应结合用户当前目标，避免固定规则长期打扰用户。

6. **活动类型可扩展**
   学习记录应基于统一的学习行为日志统计，不应把数据结构固定为单词、阅读和写作三类字段，后续可通过 `activityType` 扩展听力、口语、课程学习等类型。

7. **跨模块行为可追踪**
   阅读中查词、写作中使用目标词、笔友通信中的表达收藏等行为，应能通过关联字段回到对应模块，避免学习投入被孤立统计。

8. **建议有边界、可关闭**
   学习建议应说明触发原因，数量不宜过多，并允许用户忽略、稍后提醒或调整学习目标，避免把帮助提示变成干扰。

9. **隐私默认保护**
   学习记录页可以展示学习行为摘要，但不应直接暴露信件标题、信件正文片段、笔友昵称、用户私密备注等敏感信息。

10. **统计可长期维护**
    学习日志应有明细、汇总和清理策略，避免长期使用后日志无限增长，影响页面加载和统计性能。

#### 6.4.3 基础功能

| 功能点 | 说明 | 优先级 |
|---|---|---|
| 今日学习概览 | MVP 前端优先展示今日任务完成度、连续学习天数和模块完成数量，可在后续补充学习时长、待复习数量和最近一次学习时间 | 必须有 |
| 目标完成对比 | 展示每日/每周目标与实际完成情况，包括已完成、剩余和超额完成；MVP 可先记录目标口径，前端只展示今日完成度 | 应该有 |
| 单词学习统计 | MVP 前端展示已学/已复习数量；已掌握、待复习、困难词数量和判断标准可先作为后台统计项或模块详情页展示 | 应该有 |
| 学习质量统计 | 展示单词或阅读练习的正确率、掌握度变化、错误数量等基础质量指标；错误数量可按模块分类，并支持查看拼写错误、词义混淆等错误类型详情 | 应该有 |
| 阅读完成统计 | 展示完成阅读数量、阅读时长、题目正确率、新增生词数量和阅读中查词次数 | 应该有 |
| 写作练习统计 | 展示写作次数、累计字数、最近练习时间、目标词使用情况和可选的常见问题趋势 | 应该有 |
| 学习时长统计 | 展示今日和本周学习时长，并按单词、阅读、写作等活动类型拆分；重叠会话需要去重 | 可以有 |
| 时间趋势展示 | 展示本周学习趋势，并支持近 90 天日趋势和近 12 个月月度汇总的基础查询 | 可以有 |
| 连续学习天数 | MVP 前端展示连续学习天数；最长连续天数和中断后的重新开始提示可后续补充 | 必须有 |
| 跨模块关联统计 | 以关联卡片和模块明细形式展示阅读查词、生词本、写作目标词、收藏表达等行为；MVP 可先记录后台关联日志 | 可以有 |
| 行动建议 | 根据统计结果和用户目标给出轻量建议，如优先复习、补充阅读、继续写作或保持当前计划 | 可以有 |
| 建议偏好设置 | 允许用户选择当前学习重点，并影响建议排序、过滤和冷却频率 | 可以有 |
| 学习里程碑 | 在达到连续学习、累计学习或掌握数量节点时展示鼓励提示 | 可以有 |
| 学习总结与分享 | 生成周报、月报或可分享学习总结，但不公开用户隐私数据 | 可以有 |

#### 6.4.4 数据维度与统计口径

学习记录模块应优先保存学习行为日志，再由日志汇总出页面统计数据。基础日志建议包括：

| 数据维度 | 说明 |
|---|---|
| sessionId | 一次学习会话标识，用于串联同一时段内的多个学习行为 |
| segmentId | 会话内有效学习片段标识，用于处理暂停、恢复和碎片化学习 |
| activityType | 学习活动类型，如 word、reading、writing，后续可扩展 listening、speaking、course |
| activityId | 关联的单词、文章、写作任务、信件或课程标识 |
| parentActivityId | 上级学习行为标识，例如阅读中查词可以关联到本次阅读记录 |
| relatedWordIds | 与本次行为相关的单词列表，如阅读中收藏的生词或写作中使用的目标词 |
| relationType | 跨模块关系类型，如 lookup、collect、use_in_writing、review_from_reading |
| startedAt | 学习开始时间 |
| endedAt | 学习结束时间 |
| pausedAt | 本片段暂停时间，可为空 |
| resumedAt | 暂停后恢复时间，可为空 |
| durationSeconds | 本次有效学习时长，用于统计时间投入 |
| focusState | 页面或任务是否处于前台活跃状态，用于避免后台页面继续计时 |
| activeModule | 当前计时归属模块，避免多个页面同时打开时重复统计 |
| resultStatus | 本次学习结果，如完成、未完成、跳过、保存草稿 |
| score | 本次练习得分或掌握度变化，可为空 |
| accuracyRate | 正确率，适用于单词测验和阅读题 |
| masteryDelta | 本次学习对掌握度的影响，适用于单词复习等场景 |
| newItemsCount | 新学内容数量，如新词数量或新文章数量 |
| reviewItemsCount | 复习内容数量，如复习单词数 |
| wrongItemsCount | 错误数量或待强化数量 |
| userGoalMode | 用户当前学习重点，如 daily、exam、reading、writing，可用于解释行动建议 |
| privacyLevel | 行为摘要隐私级别，如 normal、sensitive、hidden，用于控制历史记录展示 |
| sourceModule | 数据来源模块，便于排查统计差异 |
| deviceId | 数据产生设备，用于后续多端同步排查 |
| syncVersion | 同步版本号，用于避免旧数据覆盖新数据 |
| updatedAt | 记录最后更新时间 |

页面统计口径应保持稳定，并在必要位置给出简短解释。MVP 阶段建议先采用以下默认口径，后续可根据实际测试统一调整：

| 指标或规则 | MVP 默认口径 | 说明 |
|---|---|---|
| 已学单词 | 至少完成一次新词学习、主动回忆练习或正式复习的单词 | 仅查看单词详情不直接计入已学，避免虚高 |
| 已掌握单词 | `masteryScore >= 80`、最近两次正式练习均正确，且当前未进入到期复习或错词强化队列 | 默认阈值应在配置中维护，页面可说明为“近期连续答对且掌握度较高” |
| 待复习数量 | `nextReviewAt <= 当前时间`，或因错误进入强化复习且尚未完成 | 阅读查词加入生词本后，不立即算待复习，需生成复习计划后再计入 |
| 困难词 | 最近 7 天内错误次数达到 2 次，或掌握度低于 50 且已经练习过 | 避免把完全没学过的新词直接标为困难词 |
| 阅读完成数量 | 完成正文阅读，并提交至少一组阅读题或手动确认完成 | 只打开文章不算完成 |
| 写作完成次数 | 发送信件、提交写作检查，或用户主动标记一次写作练习完成 | 普通草稿单独统计为“草稿数”，不与完成次数混淆 |
| 有效学习时长 | 学习会话中有阅读、答题、输入、切换卡片等有效操作；连续 5 分钟无操作自动暂停 | 单次会话最长有效时长可先限制为 120 分钟，避免挂机导致统计失真 |
| 会话恢复 | 自动暂停后 30 分钟内回到同一任务，作为同一 `sessionId` 下的新 `segmentId` 继续累计；超过 30 分钟或切换任务则新建会话 | 既保留碎片化学习片段，又避免把长时间离开合并为一次学习 |
| 暂停期间新行为 | 若暂停期间同一账户在其他任何端或模块产生新的有效学习行为，并写入新增行为日志，则原会话的暂停计时立即失效；用户返回原任务时，系统应开启一个全新的会话 | 避免把已经被其他学习行为打断的任务继续合并到原会话 |
| 多页面去重 | 同一用户同一设备同一时间只能有一个前台活跃 `activeModule` 计时；汇总时对重叠时间区间去重 | 同时打开阅读页和单词页时，不应把同一分钟统计两次 |
| 连续学习天数 | 当天有效学习时长不少于 3 分钟，或完成至少 1 个核心学习任务 | 只打开页面不计入连续学习 |
| 今日统计边界 | 按用户本地时区或账户设置时区的自然日统计 | 后续多端同步时应保持同一账户口径一致 |

以上数值不是对学习效果的绝对判断，而是为了让 MVP 有统一实现依据。实现时应把这些阈值集中配置，并在测试阶段验证是否容易造成误判。

#### 6.4.5 行动建议策略、偏好与冷却

行动建议应采用“规则触发 + 用户目标 + 历史趋势”的轻量策略，不宜在 MVP 阶段设计复杂推荐算法。建议逻辑可以先分为以下几类：

| 建议类型 | 触发条件示例 | 展示原则 |
|---|---|---|
| 复习优先 | 待复习数量超过今日复习目标，或困难词连续增加 | 给出复习入口，但如果用户当前选择考前突击模式，应降低重复提醒频率 |
| 质量提醒 | 正确率连续下降，或阅读题错误集中在同一题型 | 说明原因，例如“最近两次阅读正确率下降”，避免只给结论 |
| 时间管理 | 本周学习时长明显低于目标，或学习集中在单一模块 | 建议补足缺失任务，不用否定用户当前努力 |
| 目标偏移 | 用户设置了写作提升目标，但多日没有写作记录 | 提供写作入口或低压力练习建议 |
| 正向鼓励 | 达成连续学习、累计时长或掌握数量节点 | 以鼓励为主，不制造额外任务压力 |

建议偏好不应只是一个开关，而应影响建议的排序、过滤和冷却频率。MVP 阶段可采用以下规则：

| 用户学习重点 | 优先提高权重 | 降低权重或减少重复 | 不应屏蔽的建议 |
|---|---|---|---|
| 日常坚持 | 连续学习、今日任务、均衡学习、正向鼓励 | 过强的突击式新词建议 | 数据异常、保存失败、严重待复习堆积 |
| 考前突击 | 新词计划、重点词复习、错词强化、目标完成率 | 普通“补充阅读”或低优先级写作提醒 | 正确率明显下降、学习时长异常、复习严重超载 |
| 阅读提升 | 阅读时长、阅读正确率、阅读生词复习 | 与阅读无关的泛化建议 | 待复习过多、连续学习中断 |
| 写作提升 | 写作次数、目标词使用、表达收藏复用 | 与写作无关的泛化建议 | 数据安全、草稿保存失败、连续学习中断 |

为避免建议变成噪音，页面同一时间不宜展示超过 2 条主要建议。建议冷却时间建议采用以下默认口径：

| 用户操作或触发变化 | 默认处理 |
|---|---|
| 用户点击“忽略本次” | 同一建议 24 小时内不再展示 |
| 用户点击“稍后提醒” | 同一建议 4 小时内不再展示 |
| 用户点击“不再提醒此类” | 同类低优先级建议 7 天内不再展示，严重风险类建议除外 |
| 触发条件明显恶化 | 可提前解除冷却，例如待复习数量增加 50% 或正确率继续下降超过 10 个百分点 |
| 指标边界震荡 | 当一条建议因条件恶化而被召回后，应至少在页面保持可见 30 分钟，或直到用户主动操作；期间即使触发指标回落至恶化阈值以下，也不应立即隐藏或再次进入冷却 |
| 用户切换学习重点 | 重新计算建议排序，但不立即清空所有冷却状态 |
| 正向鼓励类建议 | 同一里程碑只展示一次，下一里程碑达成后再展示 |

建议文案应尽量包含“为什么提示”和“下一步入口”，并允许用户通过调整目标模式影响建议优先级。

#### 6.4.6 时间维度与历史数据生命周期

学习记录需要兼顾近期反馈和长期复盘。MVP 阶段应先保证今日与基础累计数据可靠，以下历史范围主要作为数据模型和后续迭代规划：

| 数据范围 | 展示用途 | MVP 处理建议 |
|---|---|---|
| 今日数据 | 首页概览、今日任务、即时反馈 | 实时更新 |
| 近 7 天 | 本周趋势、连续学习、短期习惯反馈 | 可先保留日明细，前端不强制展示趋势图 |
| 近 90 天 | 最近三个月复盘、阶段变化观察 | 预留日汇总结构，后续开放复盘视图 |
| 近 12 个月 | 月度学习时长、完成数量、阶段总结 | 预留月度汇总结构，后续开放月报 |
| 全部累计 | 总学习时长、累计掌握词数、累计阅读/写作数量 | 保留必要累计汇总 |

如果后续因本地存储空间限制需要清理明细日志，应先生成日、周、月汇总，再清理过旧明细，避免用户的长期学习成果消失。MVP 阶段可以先预留清理策略和字段，具体自动清理任务可在数据量增长后再实现：

| 清理对象 | 默认触发条件 | 清理前要求 |
|---|---|---|
| 原始行为明细日志 | 明细超过 12 个月，或单用户本地日志超过 10000 条 | 先生成日汇总、月汇总和累计汇总 |
| 页面停留与计时片段 | 超过 90 天的低价值片段明细 | 保留对应日期总时长和模块时长 |
| 隐私敏感历史摘要 | 用户删除信件、解除笔友关系或主动清除写作记录后 | 保留匿名统计，不保留标题、昵称、正文片段 |
| 失败重试和临时草稿事件 | 超过 30 天且已无排查价值 | 不影响正式学习统计和草稿内容 |

清理策略应优先软删除或标记归档，避免紧急清理时误删仍需同步的数据。学习总结与分享功能应优先使用汇总数据，不应暴露用户的具体写作内容、笔友信息或敏感学习记录。

#### 6.4.7 跨模块关联统计

单词、阅读和写作之间存在自然关联，学习记录不应把它们完全割裂。MVP 阶段至少应明确以下处理方式：

- 阅读中查词并加入生词本时，阅读模块记录一次查词或收藏行为，单词模块记录该词的来源为 reading，但该词只有完成正式学习或复习后才计入“已学单词”。
- 阅读中反复查同一个词时，可在行动建议中提示“该词多次出现，建议加入复习”，但不直接提高掌握度。
- 写作中使用目标词或收藏表达时，写作模块记录一次输出行为，单词模块可记录一次“语境使用”，但不应仅凭使用一次就判定已掌握。
- 如果写作检查或用户自评确认某个目标词使用正确，可以轻量提高该词的熟悉度或应用次数，但正式掌握仍应以复习或测验结果为主。
- 笔友通信中的优秀表达收藏可进入写作素材统计，也可作为后续写作练习入口，但不应公开展示给无关用户。
- 跨模块统计应区分“主行为”和“关联贡献”：例如一次阅读完成只能计入一次阅读完成，但其中产生的生词、查词和收藏可以作为关联记录展示，避免重复计数。

通过上述规则，系统既能承认用户在阅读和写作中的真实词汇接触，又能避免把“看过、查过、用过”直接等同于“掌握”。

#### 6.4.8 推荐展示结构

学习记录页面可按“先结论、后明细”的顺序组织，避免用户只看到一堆数字：

```text
学习记录页
  -> 今日概览：今日任务完成度、连续学习天数、模块完成数量
  -> 下一步建议：说明触发原因，提供复习/阅读/写作入口（可后续增强）
  -> 跨模块关联卡片：展示不超过 3 条高价值关联，如阅读生词、写作目标词、收藏表达（可后续增强）
  -> 本周趋势：每天学习时长、任务完成数、正确率或掌握度变化（可后续增强）
  -> 历史复盘：近 90 天日趋势、近 12 个月月度汇总（可后续增强）
  -> 分类统计：单词、阅读、写作分别展示数量、质量和最近记录
  -> 模块内关联明细：在阅读统计下展示生词来源，在写作统计下展示目标词使用和表达收藏
  -> 目标对比：每日/每周目标完成情况
  -> 脱敏历史记录：按时间倒序展示最近学习行为摘要
```

跨模块关联不应压过今日概览和行动建议。推荐采用两层展示：第一层在概览下方提供“关联卡片”，例如“本周阅读产生 8 个生词，其中 3 个已进入复习”；第二层嵌入各模块统计明细，展示来源、数量和下一步入口。历史记录中只展示脱敏摘要，例如“完成 1 次写作练习”“收藏 2 个表达”“完成一篇 B1 阅读”，不直接展示信件标题、信件正文、笔友昵称或敏感备注。

在视觉层级上，连续学习天数、今日完成度和待复习提醒应放在较明显位置；最近学习时间不应单独作为核心指标，而应与学习时长、学习时段和最近活动类型一起展示。

#### 6.4.9 隐私与脱敏要求

学习记录页属于个人学习数据页面，即使默认只对本人可见，也应避免在摘要中暴露不必要的敏感信息：

- 笔友通信相关历史只展示“完成一封笔友回信”“收到一封回信”等行为摘要，不展示笔友昵称、信件标题、信件正文片段或对方个人信息。
- 写作练习历史默认只展示任务类型、字数、完成状态和时间，不展示正文片段；用户主动进入写作详情页后再查看具体内容。
- 收藏表达可以展示数量和分类，若表达来自笔友通信，应默认隐藏来源人信息。
- 分享周报或月报时，只使用汇总数据和用户主动选择公开的内容。
- 如果用户删除信件、解除笔友关系或清除写作记录，学习统计可以保留匿名汇总，但历史记录不应继续显示可识别来源。

#### 6.4.10 数据更新与一致性要求

- 用户完成单词练习、阅读练习或写作练习后，应立即写入对应学习行为日志。
- 页面上的今日数据和最近记录应在操作完成后及时更新，避免用户刚完成学习却看不到变化。
- 周趋势、月趋势等汇总数据可以由日志计算，也可以使用缓存汇总，但应保证与原始学习记录可追溯。
- 指标阈值、暂停时间、连续学习天数、建议冷却时间和日志清理阈值等口径应由统一配置提供，前端展示和后端统计不应各自写一套规则。
- 如果存在同步延迟，页面应展示“最近更新时间”或加载状态，避免用户误以为数据丢失。
- 多端同步场景下，应优先合并行为日志，再重新计算汇总结果；汇总时需要对重叠时间片段去重，避免多页面或多端同时在线造成时长虚高。
- 删除或重置学习记录应作为高风险操作，需要二次确认，并说明对统计结果、历史记录和汇总数据的影响。

#### 6.4.11 异常与空状态

- 无学习记录时，页面不应只显示简单文字，而应提供明确行动入口，例如“开始背单词”“阅读一篇文章”“写一封练习信”。
- 只有部分模块有记录时，应展示已有数据，并提示用户可以补充其他类型练习。
- 统计数据加载失败时，应保留用户最近可见数据或展示重试入口，避免页面完全空白。
- 学习时长异常过长或过短时，应在统计时做保护处理，并避免生成误导性结论。
- 当系统检测到多个页面同时计时时，应只保留前台活跃页面计时，并在必要时合并或丢弃重叠片段。
- 当待复习数量明显偏多时，应优先提示用户完成复习；如果用户处于考前突击等特殊目标模式，应减少重复提醒并说明原因。
- 当用户连续学习中断时，应使用鼓励性文案，引导重新开始，不应使用惩罚性表达。
- 当跨模块关联数据缺失时，应展示可解释提示，例如“该阅读记录暂无关联生词”，而不是让用户误以为数据丢失。

#### 6.4.12 验收标准

- 系统可以展示今日学习概览，MVP 至少包括今日任务完成度、连续学习天数和模块完成数量。
- 系统可以展示单词、阅读、写作三类基础完成数量；正确率、学习时长、待复习数量等质量指标可先在模块内或后台统计中保留。
- 系统可以明确说明“阅读完成”“写作完成”“连续学习天数”等直接展示指标的统计口径；“已掌握”“待复习”“有效学习时长”等后台或后续指标应有统一配置草案。
- 连续学习天数等关键指标应有默认阈值，并由统一配置维护，避免前端和统计逻辑口径不一致。
- 学习记录数据应来自真实学习行为日志，与单词、阅读、写作模块的完成结果保持一致。
- 用户完成一次学习行为后，学习记录页面应能及时反映对应基础数量变化。
- 阅读查词加入生词本、写作使用目标词等跨模块行为应能形成后台关联记录，但 MVP 不要求全部以关联卡片或模块明细前端展示，也不应被重复计数或直接等同于掌握。
- 如前端展示历史记录，应展示脱敏学习行为摘要，不应直接暴露信件标题、正文片段、笔友昵称或其他敏感信息。
- 学习时长、多页面去重、跨设备同步、建议冷却、日志清理和近 90 天/近 12 个月复盘属于后续增强验收项；MVP 阶段应至少在数据模型和待确认事项中保留实现口径。
- 无学习记录时，页面应展示鼓励性空状态，并提供进入核心学习模块的操作入口。
- 数据加载失败、统计延迟、部分数据缺失或跨模块关联缺失时，系统应给出明确提示，不应展示误导性统计结论。

## 7. 非功能需求分析

### 7.1 用户体验需求

- 首页应清晰展示单词记忆、阅读练习、笔友写信三个入口。对于已经开始学习的用户，首页应优先展示基于学习进度的动态入口，例如“今日待复习 15 词”“继续阅读《极光之谜》”“1 封来自 Alex 的未读来信”。实际展示的数量、文章标题和来信状态应来自用户真实学习记录，不能固定写死。
- 用户完成一个学习任务后，应能明确看到反馈，包括完成结果、数据是否已保存、下一步建议和返回入口，避免用户不知道是否已经完成本次学习。
- 页面层级不宜过深，核心学习流程应控制在 3 层以内。例如，从首页进入单词练习、继续阅读文章或查看未读来信，都不应让用户反复进入多级菜单，避免产生空间迷失。
- 学习过程应减少干扰。在用户进行单词测验、阅读文章或撰写信件时，系统应抑制非紧急的全屏弹窗、营销通知和复杂动效。全局消息和弱提示应优先收纳到通知中心，不应实时打断学习过程。
- 文案应清晰、鼓励、直接，并具有行动导向。按钮、提示和反馈中应优先使用动词引导的表达，如“开始复习”“继续阅读”“查看回信”，避免使用模棱两可的措辞。对于错误和失败，应先说明当前状态，再提供可执行的解决方案或下一步入口。

### 7.2 性能需求

- 常规页面应在合理时间内完成加载，并给出可衡量的性能指标。核心页面（首页、单词学习页、阅读页）的首屏内容渲染时间（FCP）应控制在 1.5 秒以内，最大内容绘制时间（LCP）应控制在 2.5 秒以内。用户点击按钮、输入文字等交互的视觉反馈延迟应低于 100 毫秒，保证操作跟手。
- 页面切换应保持流畅。对于单词表、阅读文章等可能较长的内容，应采用虚拟滚动、懒加载、分页或代码分割等方式，避免一次性加载过多内容。页面切换时不应出现明显白屏或卡顿，用户感知到的过渡应尽量即时。
- 不应因为非核心动画、装饰性图片、非关键第三方脚本或其他非首屏必需资源影响学习页面加载。这类资源应延迟加载，不能阻塞首页、单词练习、阅读正文和写作编辑区等核心学习路径的渲染。
- 后续如接入外部接口，例如 AI 语言检查、内容审核、推荐服务等，应提供加载状态、超时控制和失败降级。外部接口调用应设置合理超时时间，建议不超过 5—8 秒。若 AI 润色或语言检查超时，应提示“检查暂不可用，您可以直接发送”一类的可操作反馈，而不是让页面长期停留在等待状态。

### 7.3 稳定性需求

- 单词状态、阅读结果、写作内容等关键数据应可靠保存。对于用户练习结果、阅读进度、写作草稿和信件内容，应采用实时自动保存与提交时事务性保存相结合的方式，尽量保证网络波动、页面崩溃或刷新后，已提交数据不丢失、不损坏，未提交草稿也能最大限度恢复。
- 用户提交操作失败时，应有明确、可操作的反馈，避免用户误以为已经保存成功。涉及数据改动的操作，如提交答案、保存阅读结果、发送信件等，失败时不能只显示“网络错误”，应尽量区分客户端网络问题、服务器繁忙或权限异常，并提供“检查网络”“稍后重试”“重新登录”等下一步建议。
- 系统应避免“静默失败”。如果保存、提交或发送没有真正成功，界面不能提示成功；如果后台需要异步处理，也应明确显示处理中、成功、失败或可重试状态。
- 关键学习流程应通过自动化测试或人工验收确认。单词练习、阅读答题和信件发送三个核心闭环应编写端到端（E2E）测试用例，每次发版前应通过这些测试，确保主流程不被破坏。

### 7.4 可扩展性需求

- 单词、阅读、写作、统计模块应保持相对独立。模块之间应通过定义明确的接口或事件通信，而不是直接引用其他模块的内部实现。例如，阅读模块将生词加入单词本时，应调用单词模块暴露的 API 或事件接口，避免模块间强耦合，便于后续单独升级或替换。
- 公共按钮、卡片、表单、提示等界面元素应组件化。
- 数据类型和接口结构应尽量统一，方便后续扩展听力、口语或学习报告功能。

### 7.5 可维护性需求

- 代码目录结构应清晰。
- 命名应符合项目规范。
- 公共逻辑应封装，避免复制粘贴。
- 需求、设计、测试和部署说明应形成文档，并与代码同步更新。至少需要产出《需求规格说明书》《系统设计概要》《接口文档（API/事件）》《前端组件库说明》和《关键流程 E2E 测试用例》。当功能、接口或流程发生变化时，相应文档也应同步修改，避免文档与实际系统不一致。

### 7.6 安全与隐私需求

- 用户写作内容和学习记录属于个人学习数据，应避免无意义公开展示。所有学习行为数据，如单词正确率、阅读速度、查词记录、写作草稿和通信内容，默认均应视为私有数据。
- 如果后续对学习数据进行去标识化、聚合分析，或用于推荐算法和学习建议生成，应在隐私政策中明确告知用户，并为用户提供关闭个性化推荐或减少数据使用的选项。
- 不应在代码仓库中提交真实密钥。
- 笔友通信涉及用户档案、信件内容和同伴互动记录，应默认私有，并提供举报、屏蔽、解除关系和必要的审核机制。
- 如果后续引入登录系统，应采用成熟、标准的身份认证方案，落实基础认证、权限控制、用户数据隔离和通信关系权限控制。必须避免用户通过修改 ID、接口参数等方式访问其他用户的练习记录、阅读笔记或私人信件。
- 如后续建设管理后台或审核后台，后台操作权限应分级控制并保留操作日志。管理员或审核人员默认不应直接访问用户私人通信内容，除非触发安全举报、内容审核或其他明确的安全处理流程。

## 8. MVP 范围定义

### 8.1 MVP 必须包含

首个可用版本建议包含：

1. 首页和基础导航。
2. 单词记忆模块：包含新词学习、至少一种主动回忆练习、即时反馈、自动复习排程、复习队列优先级、每日任务上限、基础掌握度记录和中断保存。
3. 阅读练习模块：包含分级文章、重点词辅助、至少三类阅读题、解析反馈、阅读记录和断点续读。
4. 笔友写信练习模块：MVP 阶段优先验证受控一对一写信闭环，至少包含安全档案、关系确认或受控测试关系、收件箱、信件编辑、草稿保存、内容安全审核、通信历史、举报、屏蔽或解除关系。
5. 基础学习记录：MVP 阶段应区分“前端展示项”和“后台统计项”。前端优先展示今日任务完成度、连续学习天数、模块完成数量三个核心信号，其中模块完成数量可包括已学/已复习单词数、阅读完成数量、写作或来信处理次数等。学习时长、目标完成对比、待复习数量、阅读正确率、默认统计口径、必要的跨模块关联记录、脱敏历史记录和日志清理策略等可先作为后台统计项记录，并为后续本周趋势、历史复盘和行动建议可视化预留接口，不要求 MVP 全部前端展示。
6. 加载、空状态和错误提示。
7. 基础响应式布局。

### 8.2 MVP 暂不包含

为控制范围，以下功能不纳入 MVP：

- 付费系统。
- 商城功能。
- 开放广场式大型社区。
- 实时笔友聊天。
- 群组、广场、动态流等多人公开互动。
- 复杂推荐算法。
- 复杂后台运营系统。
- 与核心学习目标关系不强的装饰性功能。

### 8.3 MVP 成功标准

MVP 阶段可以通过以下标准判断是否完成：

- 用户能从首页进入三个核心学习模块。
- 用户能完成一次包含主动回忆和即时反馈的单词学习。
- 系统能根据单词练习结果生成或更新下次复习计划，并避免错词完全挤占正常到期复习。
- 用户能完成一次包含重点词辅助、阅读题和解析反馈的阅读练习。
- 用户能完成一次受控笔友通信流程，包括创建档案、建立关系或处理测试邀请、撰写信件、安全检查和保存通信记录。
- 用户能查看基础学习记录，并能从今日任务完成度、连续学习天数和模块完成数量中判断当前学习进展；本周趋势、历史复盘、跨模块关联和行动建议可先保证后台数据口径与扩展接口。
- 内容安全审核能拦截明显违规内容，例如联系方式和明显敏感词；如果出现误判，用户已写内容不应丢失，应能修改后重新提交或进入必要的申诉、复核流程。
- 用户解除笔友关系后，对方不能继续向其发送信件；通信权限应随关系状态变化及时生效。
- 关键流程没有明显阻断性错误。

## 9. 项目边界与范围控制

### 9.1 核心边界

本项目围绕英语学习展开，核心是：

```text
单词记忆 + 阅读练习 + 笔友写信练习 + 学习反馈
```

所有新增需求都应判断是否服务于以上核心目标。

### 9.2 需求筛选标准

新增需求进入开发前，应回答以下问题：

1. 是否直接帮助用户学习英语？
2. 是否服务于单词、阅读、写作或学习反馈？
3. 是否会明显增加开发和维护成本？
4. 是否可以后续再做？
5. 是否会影响当前核心流程稳定性？

如果一个功能不能明显帮助用户学习，或会明显拖慢核心功能开发，应延后处理。

## 10. 初步技术约束

根据现有项目文档，平台倾向于采用 Web 前端技术实现，并强调组件化、模块化和自动化流程。第一阶段不做具体技术选型最终确认，但可以提出以下约束：

- 前端应采用组件化开发方式。
- 页面应支持常见浏览器访问。
- 样式和交互应保持一致。
- 数据结构应提前设计，避免后期混乱。
- 单词模块应预留学习行为日志、反应时间、复习排程字段、最大间隔字段、词库元数据、易混淆词字段、发音评分字段、会话恢复字段和算法替换空间。
- 阅读模块应预留文章元数据、段落结构、题型标签、原文依据、阅读位置和阅读行为记录字段。
- 笔友模块应预留用户档案、匹配关系、信件记录、举报审核、写作辅助记录和关系状态字段。
- 后续开发应配合类型检查、构建检查和自动化测试。
- 部署过程应尽量标准化，避免手工重复操作。

## 11. 风险分析

| 风险 | 表现 | 应对方式 |
|---|---|---|
| 范围膨胀 | 不断加入广场、群聊、动态流、商城等功能 | 坚持受控一对一通信的 MVP 边界和优先级管理 |
| 学习效果不明显 | 用户完成练习后没有反馈 | 增加学习记录、即时反馈和复习结果提示 |
| 单词记忆低效 | 仅查看单词并手动标记，缺少主动回忆和间隔重复 | 引入主动回忆练习、自动复习排程和掌握度数据 |
| 复习闭环不完整 | 错词记录与复习流程脱节 | 错词自动进入高优先级复习队列 |
| 复习队列失衡 | 错词过多挤占到期复习，导致正常复习延误 | 使用队列配额、每日上限和顺延机制 |
| 错词强化挫败 | 连续错误导致用户放弃，或长期停留在低难度题型 | 设置难度降级、恢复性升级、出现次数上限和稍后再练机制 |
| 诊断统计失真 | 词库缺少词性、难度、话题、易混淆词等元数据 | 在词库模型中预留元数据字段、易混淆词字段和数据质量状态 |
| 假性掌握 | 高掌握词复习间隔过长，实际遗忘后长期不再出现 | 设置最大复习间隔和低频抽样复查 |
| 自评与客观表现背离 | 用户频繁手动标记已掌握，但阅读或测验中表现下降 | 建立认知偏差指标，用温和提示引导词汇保鲜挑战，不强制覆盖用户选择 |
| 大词库管理困难 | 用户无法快速搜索、筛选和排序单词 | 提供基础搜索、状态筛选和排序能力 |
| 音频与录音风险 | 音频来源不明或跟读录音处理不清晰 | 记录音频授权，默认不长期保存录音，明确隐私提示 |
| 中断导致重复劳动 | 练习中途退出后进度丢失或重复提交 | 采用逐题实时提交，保存练习会话和已完成题目结果 |
| 功能重复开发 | 多页面重复写按钮、卡片、表单 | 建设公共组件库 |
| 数据保存不可靠 | 学习记录、写作草稿或通信内容丢失 | 明确数据保存策略，增加错误提示和草稿保护 |
| 页面体验复杂 | 用户找不到学习入口 | 首页突出三个核心模块 |
| 阅读体验浅层化 | 只有文章和对错反馈，无法形成能力提升 | 增加阅读辅助、多题型训练、解析和原文依据 |
| 阅读材料管理困难 | 用户无法按难度、主题或关键词找到合适文章 | 提供筛选、搜索、排序和分级元数据 |
| 阅读数据孤岛 | 生词、阅读记录和写作素材无法复用 | 打通单词模块、学习记录和写作模块 |
| 阅读推荐失准 | 新用户不知道从哪篇开始，老用户难度或主题不匹配 | 增加冷启动、自评、主题偏好和基于表现的动态难度调整 |
| 内容扩展瓶颈 | 高质量题目、长难句和解析完全依赖人工 | 预留 AI 辅助生成和人工审核流程 |
| 阅读隐私风险 | 查词、答题、高亮和阅读习惯被过度暴露 | 默认私有、支持删除、算法使用前匿名化或聚合化，删除记录不再进入个性化推荐 |
| 学习档案封闭 | 用户长期积累的数据只能留在系统内，难以导出或对接外部标准 | 预留 JSON/CSV 数据导出能力，并为能力指标预留 CEFR、Lexile 等外部标准映射字段 |
| 阅读专注受干扰 | 导航、通知或工具栏打断阅读心流，色彩提示对部分用户不可见 | 规划沉浸阅读模式，控制非核心 UI，并要求高亮提示同时使用颜色以外的语义标记 |
| 功能不可发现 | 策略模式、听读同步或长难句提示隐藏过深，用户不知道如何使用 | 使用低频、非模态、场景相关的适时教育提示，并记录关闭状态 |
| 内容质量发现滞后 | 有歧义的题目或质量较差的文章长期在线，仅依赖用户主动反馈 | 根据错误率、选项修改次数、未完成退出率和纠错频次生成主动复核任务 |
| 学习回归挫败 | 用户中断多日或重读低分文章时产生羞愧感 | 使用欢迎回归、回顾入口和低分重读鼓励等情感化提示 |
| 笔友安全风险 | 用户通过信件发送骚扰内容、联系方式或不当信息 | 建立敏感词、联系方式检测、举报、屏蔽、解除关系和审核流程 |
| 笔友匹配不合适 | 水平差距、兴趣差异或回复频率不一致导致关系中断 | 使用水平、兴趣、目标和频率的基础匹配规则，并允许用户自主筛选 |
| 匹配过严导致冷启动流失 | 用户长期找不到理想笔友，或发现页活跃度下降 | 预留受控主题公开信作为低压补充，并在匹配不足时提示调整条件 |
| 首信沉默 | 双向确认后双方都等待对方先写第一封信 | 关系建立后发送共同起点通知、破冰话题和空白信纸 |
| 求助标签压力 | 基础用户担心勾选语言帮助会暴露弱点 | 将语言帮助改为账户级学习风格偏好，并允许单封信临时调整 |
| 同伴反馈异化 | 用户把笔友关系当成互相批改作文，影响交流体验 | 将语言帮助设计为可选请求，限制建议数量，并强调友好沟通 |
| 反馈模板去人性化 | 用户机械点击系统建议，接收方感觉反馈缺少真诚 | 模板中保留少量手动输入空位，让用户必须加入自己的观察 |
| 笔友隐私风险 | 用户档案、信件内容或互动记录被过度公开 | 默认最小公开范围，提供隐私设置、关系权限控制和必要的访问记录 |
| 安全误判破坏交流 | 联系方式或敏感词检测误拦截正常表达，打断写作沉浸感 | 使用保护性提示文案、修改后快速重试和申诉复核机制 |
| 灰度话题处理失衡 | 对争议话题过度拦截会压制真实表达，放任不管又可能引发冲突 | 对未明显违规的激烈表达使用关系维护提醒，明显违规再进入审核 |
| 模板化低质量信件 | 写作模板被大量复制，降低真实交流质量 | 检测模板使用比例，发送前温和提醒补充个人故事，不直接替用户写作 |
| 离线与多端断裂 | 换设备或弱网时阅读进度、辅助状态、高亮新增、修改或删除丢失 | 规划本地保存、账号同步、操作日志、版本号和带时间戳的智能合并机制 |
| 内容运营低效 | 纠错和异常内容分散在日志中，运营人员难以及时发现重点问题 | 建立内容健康度仪表盘，聚合高纠错频次、高退出率和难度争议材料 |
| 内容库老化 | 时效性文章过多导致素材很快失去价值 | 以常青内容为主体，时效性内容作为补充和引流，并定期复核过期材料 |
| 召回打扰用户 | 沉默用户唤醒过于频繁或不符合兴趣 | 基于历史偏好低频推送，并提供关闭入口和频率限制 |
| 后期维护困难 | 模块混杂、代码耦合 | 按业务模块拆分目录和逻辑 |

## 12. 待确认事项

以下内容需要在后续阶段进一步确认：

- 是否需要用户登录，还是先支持本地匿名学习记录。
- 单词数据来源和分级标准。
- 单词复习算法采用简单规则、SM-2 变体、FSRS 还是其他间隔重复策略。
- 复习队列中到期词、错词、低掌握度词和长间隔兜底词的默认混合比例。
- 最大复习间隔 `maxIntervalDays` 的默认值。
- 首版主动回忆练习优先采用选择题、拼写题、听写还是卡片翻转。
- 题型难度降级和恢复性升级的阈值，例如连续错误几次降级、连续正确几次升级。
- 每日新词上限、每日复习上限和复习强度模式的默认值。
- 单词发音和例句朗读的数据来源、授权信息、技术实现方式及失败降级策略。
- 跟读录音是否保存、保存位置、保存周期和用户授权方式。
- 自定义词库导入的格式规范、去重规则和质量校验标准。
- 自定义词条字段完整度与可用题型之间的映射规则，例如哪些字段缺失时需要降级到卡片回忆或基础选择题。
- 干扰项轮换和易混淆词对降级的阈值，例如干扰项近期出现次数、最近使用时间和连续正确区分次数如何影响后续抽取。
- 练习中断后是默认续接上次会话，还是由用户选择续接或重新开始。
- 作答结果逐题提交失败时的重试策略和本地暂存策略。
- 阅读材料来源、难度划分、主题标签和版权边界。
- 阅读题型首版覆盖范围和每篇文章题目数量。
- 阅读解析中的原文依据是人工标注还是由内容结构自动关联。
- 阅读中生词加入单词本后的默认学习队列和复习策略。
- 新用户阅读冷启动采用自评、诊断题还是默认推荐。
- 阅读动态难度调整的阈值，例如正确率、阅读速度和查词次数如何影响推荐，以及 `estimatedLevel` 的更新规则；实现时建议使用指数移动平均等平滑机制，避免单次异常表现导致能力值剧烈波动，并明确客观数据权重高于主观自评。
- 用户自主标记与系统客观表现之间的认知偏差指标如何计算，例如手动已掌握词在阅读中的查词率、答题正确率和阅读停顿如何加权。
- `estimatedReadingTime` 的个性化展示阈值，例如积累多少篇阅读记录后可使用个人平均有效阅读速度校准预计时间。
- AI 辅助生成阅读材料、题目和解析时的审核流程与发布边界。
- 阅读数据删除、匿名化和多端同步策略，包括删除后是否保留匿名聚合贡献。

- 学习档案导出范围、JSON/CSV 字段结构，以及 `estimatedLevel` 与 CEFR、Lexile 等外部标准映射的版本管理。
- 沉浸阅读模式的入口、默认开关、可呼出工具范围，以及字号、行距、背景色等个性化阅读设置。
- 阅读适时教育提示的触发条件、频率上限和关闭规则，例如长难句停留多久后提示、用户忽略后多久不再出现。
- 泛读转精读后的统计归属和重复计数规则。
- 有效阅读时长的暂停阈值、Page Visibility API 支持范围和无操作提示方式。
- 高亮标记和辅助展开状态的本地/服务端同步范围，以及 `add/update/remove` 操作日志、版本号、`lastModified` 合并规则。
- 题目纠错入口的问题类型、后台复核优先级和处理状态。

- 内容质量主动预警阈值，例如异常错误率、选项修改次数、未完成退出率和纠错频次如何触发人工复核任务。

- 内容健康度仪表盘的指标范围，例如高纠错题目、异常退出文章、难度争议材料和待复核队列排序规则。
- 常青阅读材料与时效性主题阅读的内容比例、更新频率和过期复核机制。
- 能力叙事的生成口径、文案风格和是否允许用户关闭个性化总结。
- 沉默用户唤醒的触发周期、频率上限、兴趣匹配规则和关闭入口。
- 连续答错、频繁查词、中断回归和低分重读等情绪敏感场景的提示文案边界。
- 推荐算法中主题兴趣权重、难度匹配权重和探索随机因子的比例，以及主题口味测试信号如何平滑更新偏好。
- 阅读后自我评估进入 `estimatedLevel` 计算的低权重边界，以及如何作为信心指数或认知偏差维度生成即时反馈文案。
- 听读同步所需时间戳来源、句子级 `audioAlignment` 格式、精度要求、拖动同步和不支持时的降级策略。
- 策略模式成效反馈的指标口径和展示方式，例如扫读耗时、目标定位数量、历史趋势图或进度环。
- 高亮笔记生成复习卡片或写作素材时的接口契约、`cardGenerated`、`generatedCardId`、`sourceArticleId`、目标队列，以及 MVP 置灰入口的交互文案。
- 主题口味测试如何与诊断、自评或首次阅读完成页合并，避免新用户流程过长。
- 补签卡获取条件、有效期、持有上限和自动消耗规则；获取条件应明确“连续 3 天完成阅读”与“完成 1 次精读”为任一满足即可。
- 破冰话题、写作支架和友好通信模板的数量、分类与更新方式。
- 模板使用比例的检测阈值和提示文案，例如模板化内容超过多少比例时提示补充个人想法。
- 学习记录保存方式。
- 笔友模块首版是否必须引入真实账号系统，还是先以受控测试账号验证通信流程。
- 用户档案中年龄组、地区、英语水平和兴趣标签的展示范围。
- 笔友匹配推荐的最低规则，例如共同兴趣数量、水平差距范围和通信频率权重。
- 当理想匹配不足时，是否启用主题公开信作为低压补充，以及公开信的审核、可见范围和邀请入口。
- 双向确认后的首信破冰机制，包括通知文案、共同兴趣展示、空白信纸和首信模板是否默认出现。
- 语言帮助偏好采用账户级学习风格、单封信临时设置还是二者结合。
- 同伴反馈模板的数量、类型和限制，例如每次最多指出几个问题、是否必须包含正向表达。
- 同伴反馈模板中人工输入空位的设计，例如哪些模板必须填写原句、感受或替代表达，如何避免完全自动化反馈。
- 笔友邀请、拒绝、解除关系、屏蔽和休眠归档的具体交互规则。
- 信件内容安全审核采用本地规则、服务端规则还是人工复核，联系方式检测和敏感词策略如何落地。
- 安全拦截提示的语气规范、误判申诉入口、复核时限和修改后快速重试机制。
- 灰度讨论的识别和处理边界，例如争议话题、激烈语气、观点分歧与人身攻击之间如何区分，以及关系维护提醒何时出现。
- 举报处理流程、管理员可见范围、处理时限和违规处罚层级。
- 未成年人保护策略，包括是否开放未成年人使用、是否需要监护人同意以及匹配范围限制。
- 发送前语言检查是否接入 AI，检查内容是否上传、如何告知用户以及如何关闭。
- 同伴互助反馈的边界，例如每封信最多建议几处语言问题，如何避免过度纠错。
- 是否需要接入 AI 辅助解释、阅读生成或写作建议。
- 首版是否需要管理后台。
- 部署目标环境和访问方式。

这些问题不影响第一阶段需求分析结论，但会影响第二阶段产品设计和系统设计。

## 15. 阶段结论

EnglishLover 在线英语学习平台 Web 端的第一阶段任务已经明确：项目应定位为一个完全免费的英语自主学习平台，围绕单词记忆、阅读练习和笔友写信练习三个核心模块展开，辅以基础学习记录和反馈机制。

首个版本应优先保证核心学习流程完整、页面入口清晰、数据能够保存、反馈及时明确。对于开放广场式社区、实时聊天、群组互动、付费系统和复杂推荐等功能，应暂不纳入 MVP。笔友模块只保留服务英语写作训练所必需的受控一对一通信、安全审核、写作辅助和关系管理能力，以保证项目范围可控、开发过程稳定，并为后续模块化扩展和自动化流程建设打下基础。
