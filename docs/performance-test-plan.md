# EnglishLover 性能测试计划

> 文档用途：将第四阶段架构中的性能目标转化为可执行的压测场景、数据规模、并发目标和时间表。  
> 事实边界：当前仓库未核验到真实应用源码、数据库或部署配置，因此本文为目标测试计划。实际执行前需根据真实技术栈、接口、环境和数据规模修订。

---

## 1. 测试目标

性能测试目标不是证明系统“高并发”，而是验证 MVP 核心学习路径在目标环境下是否稳定、响应是否可接受、瓶颈是否可定位。

本计划重点验证：

1. 首页今日任务聚合；
2. 单词复习队列查询；
3. 单词逐题提交；
4. 阅读文章详情和答题提交；
5. 写信草稿自动保存；
6. 安全检查与发送；
7. 今日成果查询；
8. 管理后台举报/内容列表的低频访问；
9. AI 能力开启和关闭两种模式下的降级表现。

## 2. 测试环境基线

### 2.1 MVP 测试环境

| 组件 | 参考配置 | 说明 |
|---|---|---|
| API 服务 | 1 实例，1 vCPU，1-2GB RAM | Node/Express、NestJS 或 Go 均需记录实际版本 |
| PostgreSQL | 1-2 vCPU，2-4GB RAM | 开启慢查询日志 |
| Redis | 256MB-1GB | 用于缓存、限流、队列 |
| 前端 | 静态资源本地或测试 CDN | 记录构建大小和缓存策略 |
| 网络 | 同一区域内网优先 | 记录压测机与服务位置 |

### 2.2 小规模生产模拟环境

| 组件 | 参考配置 | 说明 |
|---|---|---|
| API 服务 | 2 实例，各 1-2 vCPU，2GB RAM | 通过 Nginx/网关负载均衡 |
| PostgreSQL | 2 vCPU，4-8GB RAM | 验证连接池和索引 |
| Redis | 1GB | 验证缓存命中率和限流 |
| 监控 | Prometheus/Grafana 或等价工具 | 收集 P95、错误率、资源占用 |

## 4. 核心性能指标

| 场景 | MVP 目标 | 小规模生产目标 | 备注 |
|---|---|---|---|
| 首页今日任务 `GET /today` | P95 < 300ms | P95 < 200ms | 包含聚合查询和缓存 |
| 复习队列 `GET /reviews/due` | P95 < 250ms | P95 < 150ms | 依赖 `user_word_states` 索引 |
| 学习提交 `POST /reviews/submit` | P95 < 300ms | P95 < 200ms | 必须保证事务和幂等 |
| 阅读详情 `GET /readings/{id}` | P95 < 300ms | P95 < 200ms | 可缓存文章内容 |
| 阅读答题提交 | P95 < 300ms | P95 < 200ms | 写事件和更新记录 |
| 草稿保存 `POST /penpal/drafts` | P95 < 300ms | P95 < 200ms | 高频、小 payload、需防抖 |
| 安全检查 | P95 < 800ms | P95 < 500ms | 规则检查目标；AI 检查另算 |
| 发送信件 | P95 < 500ms | P95 < 300ms | 含关系权限和安全结果校验 |
| 今日成果 `GET /analytics/today` | P95 < 300ms | P95 < 200ms | 优先读聚合表 |
| AI 语言检查 | 5-8 秒内超时 | 5-8 秒内超时 | 不承诺低延迟，必须降级 |

错误率目标：核心 API 5xx < 1%，写接口重复提交不得重复计数。

## 5. 压测场景设计

### 5.1 场景 S1：首页打开

```text
登录态用户 -> GET /api/v1/today -> GET /api/v1/analytics/today
```

并发目标：

- MVP：20、50 并发；
- 小规模生产模拟：100、300 并发。

关注指标：P95、数据库查询数、缓存命中率、错误率。

### 5.2 场景 S2：单词复习闭环

```text
GET /api/v1/reviews/due
  -> 连续 POST /api/v1/reviews/submit 10 次
  -> GET /api/v1/analytics/today
```

并发目标：

- MVP：20 并发用户，每人 10 题；
- 小规模生产模拟：100 并发用户，每人 10-30 题。

关注指标：事务耗时、幂等冲突、`user_word_states` 锁等待、统计队列积压。

### 5.3 场景 S3：阅读练习闭环

```text
GET /api/v1/readings
  -> GET /api/v1/readings/{id}
  -> POST /api/v1/readings/{id}/answers
  -> POST /api/v1/readings/{id}/words/{wordId}/queue（可选）
```

并发目标：MVP 20-50，小规模生产 100。

关注指标：文章缓存、题目提交延迟、生词重复加入处理。

### 5.4 场景 S4：写信草稿自动保存

```text
GET /api/v1/penpal/relations
  -> POST /api/v1/penpal/drafts 多次
```

并发目标：

- MVP：20 并发，每用户每 10 秒保存一次，持续 5 分钟；
- 小规模生产：100 并发，每用户每 10 秒保存一次，持续 10 分钟。

关注指标：写入放大、版本冲突、本地恢复和服务端状态一致性。

### 5.5 场景 S5：安全检查与发送

```text
POST /api/v1/penpal/letters/check
  -> POST /api/v1/penpal/letters/send
```

测试分组：

- 80% 正常内容；
- 15% 命中联系方式；
- 5% 命中敏感词或骚扰规则。

关注指标：规则检查耗时、拦截准确返回、高亮范围、发送幂等、关系状态校验。

### 5.6 场景 S6：举报与后台处置低频压测

```text
POST /api/v1/reports
GET /api/v1/admin/reports
POST /api/v1/admin/reports/{id}/actions
```

并发目标：普通用户举报 10-20 并发；后台 1-5 并发。

关注指标：证据快照写入、Admin 权限校验、审计日志写入。

### 5.7 场景 S7：AI 服务降级

测试两种模式：

1. AI 开关关闭：接口不应调用外部 AI；
2. AI 开关开启但外部超时：5-8 秒内返回可操作降级提示。

关注指标：超时控制、用户草稿不丢失、核心流程不阻塞。

## 7. 执行时间表

| 阶段 | 时间 | 任务 | 交付物 |
|---|---|---|---|
| T0 准备 | 第 1-2 天 | 确认接口、部署环境、压测工具 | 测试环境说明、接口清单 |
| T1 数据准备 | 第 3-4 天 | 生成测试用户、单词、阅读、信件、举报数据 | 数据生成脚本和规模报告 |
| T2 冒烟压测 | 第 5 天 | 以 5-10 并发跑通所有场景 | 冒烟报告、接口错误清单 |
| T3 MVP 压测 | 第 6-7 天 | 20-50 并发核心场景 | MVP 性能报告 |
| T4 小规模模拟 | 第 8-10 天 | 100-300 并发混合场景 | 容量评估和瓶颈分析 |
| T5 优化复测 | 第 11-14 天 | 修复慢查询、缓存、限流后复测 | 对比报告和上线建议 |

如项目周期较短，至少完成 T0-T3，不能在未压测情况下承诺 P95 指标。

## 8. 通过与不通过标准

### 8.1 通过标准

- 核心写接口 5xx < 1%；
- 学习提交、草稿保存、信件发送无重复计数；
- 首页、复习队列、学习提交达到 MVP 目标或有明确瓶颈解释；
- AI 超时不阻塞草稿保存和基础学习；
- 数据库无明显全表扫描热点；
- Redis 故障时核心流程有可接受降级；
- 压测报告包含环境、数据量、并发、结果和瓶颈。

### 8.2 不通过标准

- 写接口重复提交导致重复统计；
- 草稿保存高频失败或版本冲突不可恢复；
- 安全检查失败导致草稿丢失；
- 首页或复习队列在目标并发下大量超时；
- 数据库 CPU 或连接池长期打满且无优化方案；
- AI 超时导致页面无限等待；
- 压测数据规模与真实目标不匹配却直接宣称达标。

## 10. 计划结论

EnglishLover 的性能目标必须与运行环境、并发用户、数据量级和成本预算绑定。MVP 阶段优先验证核心学习闭环的稳定性和可恢复性，而不是追求夸大的高并发指标。只有在完成数据准备、压测执行和瓶颈分析后，才能在论文或评审材料中声明具体性能结果。
