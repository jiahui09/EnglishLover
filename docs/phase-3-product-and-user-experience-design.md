### 1.2 本文采用的设计假设

| 编号 | 假设 | 设计影响 |
|---|---|---|
| A1 | 首版为 Web 端 MVP，以学习闭环验证为主 | 页面结构优先简单直接，不追求复杂运营和大规模社区 |
| A2 | 用户多数是希望提高英语能力的学生或自学者 | 文案应鼓励、直接、低压力，避免竞赛化和焦虑化 |

| A6 | 首版不依赖复杂 AI 或推荐算法 | 选材、排程和反馈采用简单规则和明确说明 |

## 2. 阶段目标与交付边界

### 2.1 阶段目标

第三阶段的目标是把第二阶段的需求规格转化为可用于原型设计、界面评审和前端开发的产品与用户体验方案，重点回答：

1. 用户从首页进入三个核心学习模块时，路径是否足够短。
2. 每个核心闭环中，用户能否明确知道“当前在做什么、完成了什么、下一步做什么”。
3. 页面、组件、文案和反馈是否支持学习任务，而不是制造干扰。

5. 哪些体验属于 MVP 必须做，哪些只是后续增强。

## 3. 产品体验定位

### 3.1 产品体验定位

EnglishLover 的 Web 端体验定位为：**清晰、克制、任务导向的英语自学工作台**。

它不是偏娱乐化的闯关游戏，也不是复杂的社交社区，而是帮助用户完成三个核心学习动作：

1. 复习该复习的单词。
2. 完成一篇有反馈的阅读。
3. 写出一封安全、可保存、可发送的英文信。

### 3.2 体验关键词

| 关键词 | 解释 | 设计落点 |
|---|---|---|
| 清晰 | 用户进入页面后知道今天应完成什么 | 首页今日任务清单、统一主按钮、清楚标题 |
| 低干扰 | 学习中不被无关弹窗、复杂动效和营销信息打断 | 专注阅读、测验页弱化非核心入口、通知中心收纳弱提示 |
| 有反馈 | 用户完成行为后能看到结果和下一步 | 即时答题反馈、保存状态、任务完成度 |
| 可恢复 | 中断、刷新、网络失败后尽量不丢失进度 | 草稿自动保存、练习会话保存、断点续读 |
| 安全 | 写信交流不暴露隐私，不放任违规内容 | 安全档案、关系权限、发送前审核、举报和解除关系 |
| 可扩展 | 未来可加入 AI、推荐、趋势分析，但首版不暴露不可用入口 | 数据字段和接口可预留，界面不展示“敬请期待”或灰色占位入口 |

### 3.3 体验非目标

本阶段不追求以下体验方向：

- 不做开放广场、群聊、动态流和实时聊天体验。
- 不做以排行榜、打卡炫耀、强竞争为核心的激励体验。
- 不做复杂数据大屏、能力雷达和重型学习分析面板。
- 不做自动代写整封信或替用户生成虚假互动的体验。
- 不做为了视觉丰富而增加无学习价值的动画、装饰和弹窗。

## 4. 用户角色、核心场景与体验目标

### 4.1 本期主用户假设

> 说明：现有材料尚未给出最终用户画像。为避免 UX 设计继续停留在抽象层面，本文先采用一个可执行的 MVP 主用户假设；如果后续确认目标用户不是该人群，阅读内容难度、写作主题、任务节奏和反馈形式必须同步修订。

| 用户层级 | 本期定义 | 设计影响 |
|---|---|---|
| 主用户 | 国内大学生或同等水平英语自学者，正在准备大学英语四六级、课程考试或基础英语能力提升 | 内容难度以基础到中等为主，任务时间控制在 10—25 分钟 |
| 次用户 | 希望提高英文邮件、日常交流和短文表达能力的初级到中级学习者 | 写信主题以低压力日常交流和学习分享为主，不设计商务复杂场景 |
| 非本期重点用户 | 儿童启蒙用户、退休旅行用户、高阶商务写作用户、专业翻译训练用户 | 不按这些人群定制首版内容、语气和任务结构 |

### 4.2 具体用户画像

| 画像 | 背景 | 使用场景 | 主要诉求 | 体验风险 |
|---|---|---|---|---|
| P1：大二学生“小林” | 词汇量不稳定，准备四级或课程考试 | 晚上在宿舍或图书馆用 15 分钟完成当天英语任务 | 知道今天该复习什么，完成后有明确反馈 | 首页信息太杂会放弃；阅读太难会挫败 |
| P2：自学提升者“Alex” | 有一定基础，但缺少输出场景 | 通勤或午休时读短文，晚上写一封简单英文信 | 想把阅读词汇迁移到写作表达 | 写信失败或草稿丢失会严重破坏信任 |
| P3：内容维护/测试人员 | 检查文章、题目和安全规则 | 在测试环境验证题目解析、违规拦截和举报流程 | 快速发现内容问题和安全断点 | 缺少纠错和状态记录会影响质量闭环 |

### 4.3 核心场景故事

#### 场景 A：晚间 15 分钟完成今日英语任务

小林晚上 21:30 打开平台，心态是“不要再花时间找资料，尽快完成今天该做的事”。首页直接显示固定结构的今日任务清单：待复习单词、继续阅读、写信/查看回信。任务清单不是完全命令式课表，而是“系统建议 + 用户可选择”的结果：系统根据复习到期、阅读进度和未读来信给出建议，用户可以继续、跳过或重新开始当天任务。她先完成 10 个待复习单词，过程中看到“5/10”等进度，完成后看到本轮正确数、困难词和下次复习提示；随后回到首页，今日完成度发生变化。

体验要求：

- 首页必须像任务清单，而不是内容信息流。
- 单词任务完成标准必须清楚，不让用户猜“还要不要继续”。
- 完成后必须提供下一步入口，而不是停在结果页。

#### 场景 A-2：中断数日后的友好回流

小林周末外出，3 天没有登录。周一晚上再次打开平台时，首页不展示一串积压任务，也不显示“你落后了 3 天”这类压力文案，而是展示回流提示：“欢迎回来！你已 3 天未学习，可以从上次中断处继续，也可以根据今天重新生成任务。”页面提供两个同级但有推荐顺序的操作：【继续上次】和【开始新任务】。如果选择开始新任务，系统不把过去 3 天的任务机械累加到当天，而是根据当前日期、复习上限和学习负荷重新生成可完成的任务量。

体验要求：

- 错过任务后不制造积压羞耻感，不用红色警告或责备语气。
- 任务清单应支持“继续上次”和“开始新任务”两个回流路径。
- 重新生成任务时必须遵守每日上限，不能把历史未完成任务一次性压给用户。
- 首页清晰不等于强命令，用户应保有选择权和恢复掌控感。

#### 场景 B：阅读一篇短文并带走可用反馈

用户午休时打开阅读模块，选择一篇预计 5—8 分钟的中等难度文章。读完后完成主旨题、细节题和词义猜测题，结果页不仅显示正误，还给出简短解析和原文依据。用户可以把重点词加入单词本，也可以直接返回首页。

体验要求：

- 反馈不能只停留在“对/错”，必须解释依据。
- 阅读完成后要有明确终点：记录已保存、题目已提交、下一步可选。
- 加入生词本是后续动作，不强制打断完成体验。

#### 场景 C：写完信后被安全规则拦截

用户花 12 分钟给笔友写了一封英文信，其中包含邮箱地址。点击“检查并发送”后，系统提示检测到联系方式，阻止发送，但正文仍完整保留。界面必须同时做到两件事：一是在正文中高亮疑似联系方式；二是在页面顶部或侧边给出友好的解释条，说明“我们暂不支持在信中交换邮箱、电话等私人联系方式，以保护您的隐私安全。请修改高亮内容后重试。”用户删除邮箱后再次检查，安全通过并发送成功，通信历史中出现该信件。

体验要求：

- 安全审核失败不得清空或改写用户草稿。
- 用户必须同时看到“哪里有问题”和“为什么不能发送”，不能只给笼统错误。
- 发送成功以“信件已进入发送/历史记录”为用户任务终点；对方是否已读是后续状态，不作为用户完成写信任务的必要条件。
- 安全规则需要前置说明，用户应能从写信页进入“安全与隐私指南”，查看哪些内容可能被拦截。

### 4.4 核心用户任务

| 任务编号 | 用户任务 | 完成体验 |
|---|---|---|
| JT-01 | 今天完成一轮单词复习 | 过程中看到进度条、题目序号和实时正确情况，完成后看到正确数、困难词、下次复习提示和今日任务变化 |
| JT-02 | 完成一篇适合自己的阅读 | 读完文章并提交题目后，看到正确率、解析依据、生词处理入口和记录保存状态 |
| JT-03 | 写并发送一封英文信 | 确认收信对象，写作中持续看到自动保存状态，安全检查可解释，发送成功进入通信历史 |
| JT-04 | 查看今天学习成果 | 看到今日任务完成度、今日学习时长和各模块今日完成情况，且数据来自真实行为；连续学习天数只作为首页轻量激励，不抢占今日成果页 |

## 5. 信息架构设计

### 5.1 总体信息架构

第三阶段采用“首页今日任务清单 + 三个学习模块 + 今日成果记录”的结构。页面层级控制在 3 层以内，避免用户在多级菜单中迷失。

```text
首页 / 今日学习
├── 单词记忆
│   ├── 今日复习会话
│   ├── 单词库 / 待复习 / 新词 / 困难词
│   └── 单词详情
├── 阅读练习
│   ├── 阅读列表 / 推荐文章
│   ├── 阅读详情
│   └── 阅读答题与结果
├── 笔友写信
│   ├── 安全档案 / 关系状态
│   ├── 收件箱 / 通信历史
│   └── 写信编辑（含检查中、检查失败、检查成功状态）
└── 今日成果
    ├── 今日任务完成度
    ├── 今日学习时长（可选）
    └── 各模块今日完成情况
```

### 5.2 一级导航

| 导航项 | 目标 | 本期优先级 | 说明 |
|---|---|---|---|
| 今日学习 | 展示固定结构的今日任务清单和总体进度 | P0 | 默认首页，承担任务分发 |
| 单词 | 进入今日复习会话、单词库和单词详情 | P0 | 首页任务卡直达今日复习会话，自主浏览再进入单词库 |
| 阅读 | 进入阅读列表和继续阅读 | P0 | 支持继续上次阅读 |
| 写信 | 进入收件箱、写信和关系状态 | P0 | 不展示开放社区入口 |
| 今日成果 | 查看今日任务完成度、今日学习时长和各模块今日完成情况 | P0 | 专注即时成就反馈；连续学习天数放在首页轻量展示 |

### 5.3 页面层级控制

| 核心流程 | 推荐路径 | 层级控制 |
|---|---|---|
| 首页进入单词复习 | 首页 -> 今日复习会话 | 2 层 |
| 首页进入阅读 | 首页 -> 继续阅读 / 推荐文章详情 -> 答题结果 | 2—3 层 |
| 首页进入写信 | 首页 -> 写信编辑（页内检查状态） -> 通信历史/首页 | 2—3 层；发送前检查不是独立页面 |
| 查看今日成果 | 首页 -> 今日成果 | 2 层 |
| 从阅读加入生词 | 阅读详情 -> 加入单词本提示 | 当前页内完成，不跳深层页面 |

### 5.4 页面命名原则

页面标题应直接说明任务，不使用抽象或营销化命名：

| 推荐命名 | 不推荐命名 |
|---|---|
| 今日学习 | 学习中心、成长空间 |
| 开始复习 | 词汇能量站 |
| 继续阅读 | 阅读探索宇宙 |
| 写一封信 | 灵感创作岛 |
| 查看今日成果 | 数据洞察中心 |

## 6. 核心学习路径设计

### 6.1 今日学习总路径

```text
进入首页
  -> 查看固定结构的今日任务清单
  -> 选择一个当前最明确的任务：开始复习 / 继续阅读 / 查看回信或写信
  -> 完成对应学习任务
  -> 返回首页或今日成果
  -> 查看今日完成度变化
```

设计重点：

- 首页采用“固定任务槽位 + 数据驱动状态”，不是内容流、推荐流或动态推送页。
- 三个槽位顺序稳定：单词任务、阅读任务、写信任务；槽位内容根据真实学习状态变化。
- 已开始学习的用户看到任务状态，例如“今日待复习 15 词”“继续阅读《极光之谜》”“1 封来自 Alex 的未读来信”。这里的“动态”仅指状态数据变化，不指随机推荐或信息流。
- 新用户没有历史数据时，展示冷启动任务，例如“先复习 10 个基础词”“阅读一篇短文”“创建安全笔友档案”。
- 每完成一个任务，首页完成度应变化，形成正反馈。
- 不在首页展示灰色的“敬请期待”功能入口、AI 占位入口或不可用的后续模块。

#### 6.1.1 今日任务清单生成逻辑

| 任务槽位 | 生成依据 | 用户可控项 | 错过后的处理 |
|---|---|---|---|
| 单词任务 | 到期复习词、困难词比例、每日新词/复习上限 | 开始复习、调整今日任务量、重新生成今日任务 | 不机械累加历史任务，按当前日期和每日上限重新计算 |
| 阅读任务 | 上次未完成文章、默认难度、用户已完成记录 | 继续上次、换一篇、稍后阅读 | 保留断点；长时间未读时可重新选择文章 |
| 写信任务 | 未读来信、未完成草稿、已建立关系状态 | 查看回信、继续草稿、写新信 | 草稿保留；关系失效时提示不能继续发信 |

设计原则：

- 今日任务清单是“系统建议 + 用户选择”，不是强制课表。
- 默认推荐一个最轻量的下一步，但保留用户切换任务的权利。
- 未完成任务不应无限堆积；连续未学习时优先提供回流选择，而不是制造压力。
- 任务量应受每日上限保护，避免一次中断导致后续任务不可完成。

#### 6.1.2 回流与追赶策略

| 回流场景 | 首页提示 | 可选动作 | 设计底线 |
|---|---|---|---|
| 1 天未完成 | 温和提示“昨天的任务未完成，可继续或开始今天任务” | 继续上次 / 开始今日任务 | 不用责备语气 |
| 2—3 天未学习 | 提示“欢迎回来”，说明可恢复或重启 | 继续上次 / 开始新任务 | 不显示巨大积压清单 |
| 3 天以上未学习 | 建议重新生成轻量任务 | 开始新任务 / 查看上次进度 | 不强制补完历史任务 |
| 有未完成草稿 | 优先保护写作成果 | 继续写信 / 稍后处理 | 不自动删除草稿 |

### 6.2 单词复习路径

```text
首页待复习入口
  -> 今日复习会话
  -> 查看题目或回忆提示
  -> 作答
  -> 查看即时反馈
  -> 进入下一题
  -> 完成会话
  -> 查看本次结果和下一次建议
```

关键体验要求：

| 设计点 | 要求 |
|---|---|
| 进入前 | 首页任务卡直达今日复习会话；显示本次任务数量和预计用时，避免用户不确定负担 |
| 练习中 | 页面只保留题目、答题区、进度条、题目序号、轻量实时正确情况和退出保存，不放复杂统计 |
| 答题后 | 明确显示正确/错误、正确答案、简短解释和下一题按钮 |
| 中断时 | 已提交答案保存；未完成会话可恢复或重新开始 |
| 完成后 | 展示完成数、正确数、困难词数量和下一步入口 |

### 6.3 阅读练习路径

```text
首页继续阅读或阅读列表
  -> 选择文章
  -> 阅读正文与重点词
  -> 完成题目
  -> 查看答案和原文依据
  -> 保存阅读记录
```

关键体验要求：

| 设计点 | 要求 |
|---|---|
| 选材 | 展示难度、主题、预计阅读时间和重点词数量 |
| 阅读正文 | 正文区域保持专注，不出现非紧急弹窗和复杂动效 |
| 重点词 | MVP 默认采用内联轻提示，点击或长按查看释义，不遮挡正文 |
| 答题 | 题目按主旨、细节、词义猜测分类，提交后显示解析 |
| 解析 | 必须标出原文依据或说明依据段落，避免只显示“正确/错误” |
| 断点 | 用户离开后可继续阅读或重新开始 |

### 6.4 笔友写信路径

```text
进入笔友模块
  -> 创建或确认安全档案
  -> 查看关系状态和收件箱
  -> 阅读来信或选择写信对象
  -> 撰写英文信
  -> 自动保存草稿
  -> 在写信编辑页内完成发送前安全检查
  -> 检查通过后发送成功并写入通信历史
```

关键体验要求：

| 设计点 | 要求 |
|---|---|
| 安全档案 | 明确哪些信息对笔友可见，不要求真实姓名、电话、邮箱、地址 |
| 关系状态 | 展示“已建立关系/已屏蔽/已解除/测试关系”等状态 |
| 编辑器 | 显示收信对象、字数、草稿保存状态和写作支架入口 |
| 安全检查 | 在写信编辑页内检查联系方式、敏感词和明显违规内容，不跳离编辑上下文 |
| 审核失败 | 不清空正文，提示原因和修改建议 |
| 解除关系 | 解除后对方不能继续发信，页面状态要明确 |

### 6.5 今日成果路径

```text
完成学习任务
  -> 首页今日任务状态更新
  -> 进入今日成果页
  -> 查看今日任务完成度
  -> 查看今日学习时长（可选）
  -> 查看各模块今日完成情况
  -> 选择下一步学习入口或结束学习
```

关键体验要求：

- 今日成果页服务 JT-04“查看今天学习成果”，优先展示当天完成内容和即时成就感。
- 连续学习天数属于长期习惯激励，首版放在首页顶部问候区轻量展示，不作为今日成果页核心信息。
- 今日成果页不展示私人信件正文、笔友敏感信息或可识别通信内容。
- 今日成果页的主要价值是“告诉用户今天已经完成什么、还能做什么”，而不是做长期数据看板。

### 6.6 核心任务“完成”的 UX 定义

| 任务 | 本期完成口径 | 完成后必须反馈 | 后续可选动作 | 不计为完成的情况 |
|---|---|---|---|---|
| 单词复习 | 用户完成本轮任务中全部题目；本轮默认建议 10 词，或完成当天到期词中受每日上限保护的任务量 | 本轮完成数、正确数、困难词数、下次复习提示、今日任务变化 | 继续复习、去阅读、返回首页 | 只浏览单词详情、未提交答案、主动放弃本轮 |
| 阅读练习 | 用户阅读文章并提交本篇要求的基础题；MVP 至少包含主旨、细节、词义猜测三类题中的配置题目 | 正确率、每题解析、原文依据、阅读记录已保存 | 加入生词本、再读一篇、返回首页 | 只打开文章、未提交题目、只收藏文章 |
| 写信练习 | 信件通过安全检查并成功写入发送记录或通信历史 | 发送状态、收信对象、发送时间、草稿清理或归档说明 | 返回通信历史、写下一封、查看关系状态 | 只保存草稿、安全审核未通过、关系已解除或被屏蔽 |
| 今日学习 | 至少一个核心模块任务完成，或按后续配置完成今日计划中的指定模块 | 今日完成度变化、今日学习时长（可选）、各模块今日完成情况；连续学习天数仅在首页轻量更新 | 继续其他模块或结束学习 | 只打开页面、只浏览内容、虚构或手动写死记录 |

> 说明：对方是否已读或回信不作为用户“写信任务完成”的必要条件。已读/回信属于后续通信状态，避免用户完成感被他人行为绑架。

### 6.7 中断与恢复策略

“可恢复”不能只有一句自动保存。不同任务的状态、用户意图和恢复方式应区分处理：

| 场景 | 系统意外刷新/断网 | 用户主动退出 | 再次进入时 | 设计底线 |
|---|---|---|---|---|
| 单词测验 | 已提交题目立即保存；当前未提交题可恢复题面和本地选择，但需用户再次确认提交 | 弹出轻量确认：退出并保存进度 / 放弃本轮 | 显示“继续上次练习”和“重新开始本轮”两个明确选项 | 不重复提交已完成题；放弃本轮不计完成 |
| 阅读正文 | 保存文章位置、已查看重点词和已提交题目 | 返回时保留位置；如用户选择重新阅读，可重置位置但不删除已完成记录 | 默认继续上次位置，提供“从头阅读”次级操作 | 不因只打开文章就记完成 |
| 阅读答题 | 已提交答案保存；未提交选择可本地恢复但不计入结果 | 离开前提示“题目尚未提交” | 返回题目页时标明哪些题已提交、哪些未提交 | 结果页只基于已提交答案生成 |
| 写信编辑 | 草稿自动保存；保存失败显示可重试，不清空正文 | 返回前提示草稿已保存或尚未保存 | 默认恢复到最后保存或本地暂存内容 | 安全失败、网络失败、语言检查失败都不能丢稿 |
| 发送流程 | 检查中断时保持草稿和检查状态，不显示发送成功 | 用户可取消检查并回到编辑 | 若未写入发送记录，回到草稿；若已发送，进入历史 | 不出现“提示成功但历史无记录”的静默失败 |

### 6.8 过程中的进度感知

“清晰”必须贯穿任务进行中，而不是只出现在起点和终点。首版至少需要以下过程反馈：

| 任务 | 过程反馈 | 展示要求 | 不推荐做法 |
|---|---|---|---|
| 单词复习 | 当前题号、总题数、进度条、轻量实时正确情况 | 如“15/30”，进度条固定在顶部或题目上方；实时正确情况可弱展示 | 只显示当前题，不告诉用户还剩多少 |
| 阅读 | 阅读位置、预计剩余时间或段落进度、已答题数量 | 阅读进度不遮挡正文，答题页显示“已完成 2/5 题” | 用强弹窗提醒阅读进度 |
| 写信 | 草稿保存状态、最后保存时间、字数、关系状态 | 状态指示器轻量、不可点击，常态显示“已自动保存”或“上次保存 21:35” | 保存中无提示，或保存失败仍显示成功 |
| 发送检查 | 检查状态、可取消/返回编辑、错误原因 | 检查中禁用重复提交但允许返回编辑 | 无限加载或重复点击无反馈 |

### 6.9 自动保存状态规范

| 状态 | 显示文案 | 交互要求 |
|---|---|---|
| 已自动保存 | “✅ 已自动保存”或“已保存 21:35” | 常态轻量展示，不抢主按钮焦点 |
| 正在保存 | “💾 正在保存…” | 短暂显示，避免用户重复离开 |
| 保存失败 | “草稿暂未保存，请重试” | 提供重试或保留本地暂存，不清空正文 |
| 本地暂存 | “已临时保存在本设备” | 说明可能未同步，不误导为最终保存 |

## 7. 页面级产品与 UX 设计

### 7.1 首页 / 今日学习

#### 页面目标

首页不是宣传页，也不是推荐信息流，而是用户每天进入平台后的今日任务清单。它需要帮助用户在 5 秒内判断：今天必须先做什么、做完后还剩什么。

#### 信息结构

```text
页面标题：今日学习
├── 顶部问候与轻量激励：日期、问候语、连续学习天数（弱展示）
├── 今日概览入口：今日任务完成度 + “查看今日成果”
├── 今日任务卡片
│   ├── 单词：今日待复习 / 开始基础词
│   ├── 阅读：继续阅读 / 推荐一篇短文
│   └── 写信：未读来信 / 写一封信 / 创建安全档案
├── 最近进度或提醒
└── 弱提示/通知入口
```

#### 核心区块

| 区块 | 内容 | 交互 | 优先级 |
|---|---|---|---|
| 今日概览入口 | 只展示一个核心指标：今日任务完成度；可配一句“查看今日成果” | 点击进入今日成果 | P0 |
| 轻量激励 | 日期、问候语、连续学习天数 | 弱展示，不抢今日成果焦点 | P1 |
| 单词任务卡 | 今日待复习词数、新词入口或继续会话 | “开始复习” | P0 |
| 阅读任务卡 | 继续阅读文章、默认短文、预计时间 | “继续阅读”或“选择文章” | P0 |
| 写信任务卡 | 未读来信、草稿、写信入口、安全档案提示 | “查看回信”“继续写信” | P0 |
| 最近进度或提醒 | 最近一条需要用户知道的弱提醒，例如草稿保存失败、关系状态变化或回流提示；无内容时不展示 | 点击进入对应模块或通知中心 | P1 |
| 通知中心 | 保存失败、系统提醒、弱提示 | 点击查看，不实时打断 | P1 |

> 首页“今日概览入口”不平铺今日学习时长和各模块完成列表，避免与今日成果页重复。首页负责提醒用户“今天进度到哪里”，今日成果页负责展示“今天具体完成了什么”。

#### 首页状态设计

| 状态 | 展示方式 | 示例文案 |
|---|---|---|
| 新用户 | 三个模块的低门槛启动卡 | “先从 10 个基础词开始” |
| 有待复习 | 单词卡优先展示待复习数量 | “今日待复习 15 词” |
| 有未读来信 | 写信卡展示未读来信，不暴露敏感正文 | “1 封来自 Alex 的未读来信” |
| 有草稿 | 写信卡展示继续写作 | “你有一封未完成草稿” |
| 今日已完成 | 展示完成反馈和轻量下一步 | “今天的核心任务已完成，可以继续阅读一篇短文” |
| 数据加载失败 | 明确说明并保留模块入口 | “进度暂时加载失败，你仍可以进入学习模块” |
| 多日未学习 | 提供回流选择，不显示积压羞耻 | “欢迎回来！继续上次，还是开始新任务？” |

#### 通知中心 P1 兜底规则

通知中心在本期被定义为 P1 / 可选项。若开发资源紧张，UX-13 可以延后，但首页“最近进度或提醒”必须承担最低限度的弱提示出口，避免关键提醒完全不可见。

| 情况 | MVP 最低要求 |
|---|---|
| 实现通知中心 | 首页最近提醒可链接到通知中心，通知中心收纳非紧急系统消息和历史弱提示 |
| 暂不实现通知中心 | 首页最近提醒只显示最近一条关键弱提示，并能跳转到对应模块或当前问题位置 |
| 必须露出的弱提示 | 草稿保存失败、关系状态变化、网络离线/恢复、回流提醒、系统级错误恢复入口 |
| 不应露出的内容 | 营销消息、成就刷屏、无关推荐、私人信件正文 |

> 结论：UX-13 是否进入 MVP 可由排期决定，但“最近进度或提醒”不能删除；它是通知中心延后时的兜底信息出口。

### 7.2 单词模块页面

#### 7.2.1 今日复习会话页

| 设计项 | 要求 |
|---|---|
| 页面目标 | 承接首页“开始复习”，直接进入系统生成的今日复习任务 |
| 主要内容 | 本轮目标数量、预计用时、开始按钮、继续上次/重新开始选项 |
| 主操作 | 开始复习 / 继续上次练习 |
| 不做内容 | 不先展示待复习、新词、困难词等列表让用户二次选择 |

建议结构：

```text
今日复习会话
├── 本轮任务：今日待复习 10 词 / 预计 5 分钟
├── 任务来源说明：到期词 + 少量困难词
├── 可选：调整今日任务量
└── 主按钮：开始复习 / 继续上次
```

#### 7.2.2 单词库页

| 设计项 | 要求 |
|---|---|
| 页面目标 | 支持用户自主浏览、搜索和管理单词 |
| 主要内容 | 状态筛选、关键词搜索、单词卡片、空状态提示 |
| 主操作 | 查看详情、加入今日任务、学习新词 |
| 与今日复习关系 | 单词库是自主学习入口，不是首页“开始复习”的默认落点 |

建议结构：

```text
单词库
├── 筛选：全部 / 待复习 / 新词 / 困难词
├── 搜索框
├── 单词列表卡片
└── 空状态或加载失败提示
```

#### 7.2.3 单词练习页

| 设计项 | 要求 |
|---|---|
| 页面目标 | 支持主动回忆和即时反馈 |
| 主要内容 | 题目、选项/输入框、当前进度、反馈区 |
| 主操作 | 提交答案、下一题、退出并保存 |
| 干扰控制 | 不展示排行榜、复杂统计、营销弹窗 |

建议结构：

```text
单词练习
├── 顶部：本轮进度 15/30、轻量正确情况、退出并保存
├── 题目区：英文/中文提示/例句空缺
├── 作答区：选项或输入框
├── 反馈区：正确/错误、答案、简短解释
└── 下一步：下一题 / 查看结果
```

#### 7.2.4 单词结果页

| 结果项 | 展示要求 |
|---|---|
| 完成数量 | 显示本轮完成词数 |
| 正确情况 | 显示正确数或正确率，但不制造挫败 |
| 困难词 | 提示有多少词需要后续复习 |
| 下一步 | “继续复习”“去阅读”“返回首页” |

#### 7.2.5 单词详情页与未来扩展边界

| 设计项 | 要求 |
|---|---|
| 页面目标 | 展示单词释义、例句、复习状态、掌握情况和来源说明 |
| 首版边界 | 不展示“关联阅读”等暂无真实数据支撑的入口，不放灰色占位或“敬请期待”按钮 |
| 未来扩展点 | 组件和数据结构可预留可选字段，例如该词出现过的阅读文章 ID、标题和上下文片段 |
| 展示条件 | 只有在阅读材料来源、词汇抽取规则和跨模块索引可靠后，才可显示“在哪篇阅读里见过” |

> 说明：该扩展点用于支持未来“阅读-词汇”学习闭环，但在 MVP 中只作为组件/接口层预留，不改变当前单词学习主路径。

### 7.3 阅读模块页面

#### 7.3.1 阅读列表页

| 设计项 | 要求 |

| 页面目标 | 帮助用户找到合适难度和主题的文章 |

文章卡片建议字段：

| 字段 | 说明 |

| 难度 | 如基础/中等/较难，具体分级待确认 |
| 主题 | 科普、校园、生活等 |
| 预计阅读时间 | 降低开始成本 |
| 重点词数量 | 提示词汇负担 |
| 进度状态 | 未开始、进行中、已完成 |

#### 7.3.2 阅读详情页

| 设计项 | 要求 |
|---|---|
| 页面目标 | 提供专注阅读和重点词辅助 |
| 主要内容 | 标题、难度、正文、重点词、阅读进度 |

| 干扰控制 | 阅读过程中不出现非紧急全屏弹窗 |

阅读页布局建议：

```text
阅读详情
├── 文章信息：标题、难度、预计时间
├── 正文区域：段落清晰、行距适中
├── 重点词辅助：MVP 默认采用内联轻提示
├── 阅读进度：轻量显示，不遮挡正文
└── 主按钮：开始答题 / 继续答题
```

重点词辅助默认交互：

| 方案 | MVP 口径 | 说明 |
|---|---|---|
| 内联轻提示 | 默认采用 | 对重点词使用浅色下划线或轻标记，点击或长按弹出小型释义卡片；不默认展开大段解释 |
| 侧栏 | 延后或桌面端增强 | 适合宽屏，但移动端占用空间，首版不作为默认形态 |
| 折叠面板 | 延后或 A/B 测试 | 可降低干扰，但需要用户额外点击，首版不作为默认形态 |

> 内联轻提示必须可关闭或不触发，不应强制打断阅读；若重点词数据缺失，应正常显示正文，不出现空白侧栏或占位面板。

| 设计项 | 要求 |
|---|---|
| 题目组织 | 主旨题、细节题、词义猜测题分组或按顺序展示 |
| 提交反馈 | 提交后展示正确答案、解析和原文依据 |
| 错误处理 | 保存失败时不丢失作答状态 |
| 结果页 | 展示正确率、完成状态、重点词处理入口 |

### 7.4 笔友写信模块页面

#### 7.4.1 安全档案页

| 设计项 | 要求 |

| 字段建议 | 昵称、英语水平、兴趣标签、学习目标、可见范围说明 |

| 验收重点 | 用户能清楚知道哪些信息会被笔友看到 |

#### 7.4.2 收件箱与通信历史页

| 设计项 | 要求 |
|---|---|
| 页面目标 | 展示来信、未读状态和关系状态 |
| 主要内容 | 笔友昵称、最后通信时间、未读状态、关系状态 |
| 主操作 | 查看来信、写回信、屏蔽/解除关系 |
| 隐私要求 | 不在列表中展示过多信件正文 |

#### 7.4.3 写信编辑页

| 设计项 | 要求 |
|---|---|
| 页面目标 | 支持用户完成英文写作并安全发送 |
| 主要内容 | 收信对象、主题/提示、正文编辑器、字数、草稿状态 |
| 主操作 | 保存草稿、检查并发送、返回收件箱 |
| 写作支架 | 可提供开头句、表达建议、沟通检查清单，但不自动代写整封信 |

建议结构：

```text
写一封信

├── 写作支架：可折叠
└── 主按钮：检查并发送
```

#### 7.4.4 写信编辑页内安全检查状态

本期安全检查按“规则/服务自动检查 + 可降级本地规则”设计，并作为写信编辑页内的状态/模态流程呈现，不作为独立页面跳转。这样用户在检查失败后仍处于编辑上下文，可以直接修改高亮内容并重新检查。若后续引入人工审核，必须单独设计“待审核”状态和预计时延，不能让用户无限等待。

| 状态 | 体验要求 | 示例文案 |
|---|---|---|
| 草稿已保存，待检查 | 发送前先确认草稿已保存 | “草稿已保存，可以进行发送前检查” |
| 检查中 | 显示短暂状态，禁用重复点击，但不阻塞草稿保存 | “正在检查内容安全，请稍候” |
| 检查通过 | 明确进入发送动作 | “检查通过，正在发送” |
| 语言建议 | 以低强度信息条或编辑器底部建议展示语法、拼写或表达建议；不阻止发送，不使用红色阻断样式 | “发现 2 处可优化表达，可修改后发送，也可以直接发送” |
| 发现联系方式 | 阻止发送，保留正文；正文内高亮疑似文本；顶部/侧边解释原因和修改方式 | “我们暂不支持在信中交换邮箱、电话等私人联系方式，以保护您的隐私安全。请修改高亮内容后重试。” |
| 发现敏感内容 | 阻止发送，保留正文；高亮可定位内容或说明问题类型，不自动改写内容 | “部分内容不适合发送，请根据提示修改后重试” |
| 检查服务超时 | 使用本地基础规则降级；若本地规则也不可用，则保留草稿并暂不发送 | “检查暂不可用，草稿已保存，请稍后重试” |
| 发送失败 | 不显示成功，保留草稿或标记为未发送 | “信件暂未发送成功，草稿已保留，请重试” |
| 发送成功 | 写入通信历史，显示收信对象和发送时间 | “信件已发送，已保存到通信历史” |

> 说明：语言检查和安全检查必须在 UI 上区分。“语言检查不可用”不应阻断发送；“安全检查未通过”必须阻断发送。系统不得自动篡改用户信件正文，只能提示用户修改。

#### 7.4.5 页内安全拦截反馈规则

| 反馈元素 | 是否必须 | 说明 |
|---|---|---|
| 正文高亮 | 必须 | 对可定位的疑似联系方式、敏感词或违规片段进行高亮，帮助用户快速修改 |
| 解释条 | 必须 | 在页面顶部或侧边说明为什么被拦截，以及修改后可重试 |
| 草稿保留 | 必须 | 拦截后正文、收信对象和未发送状态必须保留 |
| 规则说明入口 | 必须 | 提供“安全与隐私指南”入口，说明哪些内容可能被拦截 |
| 重新检查按钮 | 必须 | 用户修改后可再次检查并发送 |
| 自动改写 | 禁止 | 系统不得自动替用户删除、改写或发送内容 |

> 规则识别能力也应透明表达：MVP 主要识别文本中的邮箱、电话、社交账号、明显敏感词等。图片、变形拼写、截图中的文字等复杂识别不作为首版承诺；若识别不到，不应在文档中宣称已经覆盖。

#### 7.4.6 关系解除与屏蔽体验

| 场景 | 设计要求 |
|---|---|
| 解除关系前 | 说明解除后双方不能继续发送私人信件 |
| 解除关系后 | 通信历史可按隐私规则保留，但写信入口禁用 |
| 被屏蔽状态 | 不展示攻击性文案，只说明“当前无法向该用户发送信件” |
| 举报入口 | 每封信或通信记录提供低成本举报入口 |

### 7.5 今日成果页面

#### 页面目标

今日成果页用于承接用户完成任务后的即时成就反馈，不做复杂数据看板。首版只需要回答三个问题：

1. 今天完成了多少。
2. 今天在哪些模块取得了成果。
3. 现在可以继续做什么，或安心结束学习。

#### 页面结构

```text
今日成果
├── 今日任务完成度：环形图/进度条
├── 今日学习时长：可选，来自真实学习行为
├── 各模块今日完成情况
│   ├── ✅ 单词复习已完成 10/10 词
│   ├── ✅ 阅读已完成《AI 的未来》
│   └── ✅ 已发送 1 封英文信
└── 下一步建议入口
```

> 完成口径必须与 6.6 节一致：已保存草稿不计入“今日成果”的完成项。草稿状态应在首页写信任务卡或写信模块入口提示，例如“你有一封未完成草稿”，但不作为成果页中的勾选成就展示。

#### 展示边界

| 可展示 | 放在其他位置或延后 |
|---|---|
| 今日任务完成度 | 复杂能力雷达 |
| 今日学习时长（可选） | 周报/月报趋势大屏 |
| 各模块今日完成情况 | 私人信件正文或笔友敏感信息 |
| 简单下一步入口 | 复杂跨模块推荐算法 |
| 首页弱展示连续学习天数 | 今日成果页大面积展示连续学习 0 天等可能挫败的新手信息 |
| 已发送信件数量 | 已保存但未发送的草稿作为成果勾选项 |

> 说明：连续学习天数是长期习惯激励，不是“今日成果”的核心内容。首版更适合在首页问候区弱展示，例如“晚上好，小林。已连续学习 3 天，坚持很棒。”新用户或中断用户不应在今日成果页看到突兀的“连续学习 0 天”。

#### 未来迭代观察：今日成果的轻量仪式感

当用户完成全部今日任务后，今日成果页未来可以从“查询页”轻量增强为“庆祝页”，例如标题短暂显示“今日成就达成”，或生成一张仅保存到本地的学习卡片。该方向不进入 MVP，且不得默认公开分享、上传到社区或制造社交压力。只有在今日成果数据口径稳定、隐私边界明确、用户确实需要保存学习回顾时，才建议进入后续版本评审。

### 7.6 页面体验原则

所有核心页面应遵循以下体验原则，作为原型评审和后续前端实现的统一检查项：

| 原则 | 设计要求 | 适用页面 |
|---|---|---|
| 操作路径尽量短 | 从首页进入核心学习任务不超过 3 层，常用操作不隐藏在多级菜单中 | 首页、单词、阅读、写信 |
| 主按钮位置统一 | 同类流程的主操作位置保持稳定，避免用户重复寻找 | 单词练习、阅读答题、写信编辑 |
| 表单字段尽量少 | 首版只收集完成任务和保障安全所需字段 | 安全档案、写信编辑 |
| 不频繁弹窗打断学习 | 非紧急提示收纳到通知中心或页面内弱提示 | 单词测验、阅读正文、写信编辑 |
| 颜色和字体保持一致 | 不用颜色堆砌区分模块，优先通过标题、图标和文案表达含义 | 全站 |
| 学习进度清晰可见 | 让用户知道当前进度、已完成内容和下一步 | 首页、练习页、结果页、记录页 |
| 移动端内容不被遮挡 | 底部操作、键盘和导航不能遮挡正文、选项或输入框 | 阅读页、写信页、练习页 |
| 状态提示完整 | 空数据、加载中、错误、保存失败和成功状态均有明确说明 | 全站 |

## 8. 组件与设计系统规划

### 8.1 组件设计原则

1. **少而稳定**：优先定义能覆盖多个模块的基础组件。
2. **状态完整**：每个组件应考虑加载、空、错误、禁用、成功等状态。
3. **文案清楚**：组件不只依赖颜色表达状态，必须有文字或图标辅助。
4. **可复用**：单词、阅读、写信模块的卡片、按钮、提示和表单不重复设计。
5. **不过度抽象**：首版不建设庞大组件库，只保留必要规范。

### 8.2 MVP 组件清单

| 组件 | 使用场景 | 必要状态 | 优先级 |
|---|---|---|---|
| AppShell 应用框架 | 全站导航、页面容器 | 默认、移动端折叠 | P0 |
| PrimaryNav 主导航 | 首页、单词、阅读、写信、记录 | 当前选中、禁用、移动端 | P0 |
| TaskActionCard 今日任务卡 | 首页固定任务槽位 | 容器层：默认、加载、空、错误；内容层：按单词、阅读、写信区分任务语义 | P0 |
| RecoveryPrompt 回流提示 | 首页、中断后进入任务 | 继续上次、开始新任务、关闭 | P0 |
| ProgressIndicator 过程进度 | 单词练习、阅读、写信检查 | 进行中、已完成、保存中、失败 | P0 |
| ProgressSummary 进度概览 | 首页、今日成果 | 加载、无数据、更新中 | P0 |
| LearningCard 学习卡片 | 单词卡、文章卡、笔友卡 | 默认、进行中、已完成 | P0 |

#### 8.2.1 TaskActionCard 今日任务卡插槽规则

TaskActionCard 不应被设计成完全统一内容的卡片，而应采用“统一容器 + 模块内容插槽”的组合方式。容器负责视觉骨架、状态标签位置、主按钮位置和加载/错误/空状态；内容插槽负责呈现不同学习任务的专属信息。

| 任务卡类型 | 内容插槽应展示 | 主操作 | 完成语义 |
|---|---|---|---|
| 单词任务卡 | 今日待复习词数、总词数、本轮进度，例如“待复习 15 词” | 开始复习 / 继续复习 | 已完成本轮目标词数，例如“已完成 15/15 词” |
| 阅读任务卡 | 文章标题、预计阅读时间、阅读进度或答题状态 | 继续阅读 / 开始阅读 | 已提交本篇阅读题目并保存结果 |
| 写信任务卡 | 未读来信数量、发信对象昵称、草稿状态或关系状态 | 查看回信 / 继续写信 / 写一封信 | 信件已通过检查并进入发送记录 |

设计约束：

- 容器层只定义“默认、加载、空、错误”等通用状态，不把所有模块强行映射为同一种“完成”文案。
- 模块内容区可差异化，但卡片标题、主按钮位置、状态标签和错误提示样式保持一致。
- 写信任务卡若显示“1 封未读来信”，应提供“查看回信”入口；若关系已解除，应显示禁用原因而不是空白卡片。

#### 8.2.2 SafetyNotice 安全提示触发与消失规则

SafetyNotice 是笔友写信安全体验的关键组件，必须明确触发时机、阻断条件和消失条件。

| 提示类型 | 触发时机 | 展示方式 | 是否阻断 | 消失条件 |
|---|---|---|---|---|
| 普通提示 | 打开写信页、安全档案未完善、存在一般风险提醒时 | 页面顶部或编辑区上方 banner | 否，不阻止编辑和发送前检查 | 用户可手动关闭；风险状态变化后可重新出现 |
| 阻断提示 | 用户点击“检查并发送”后，检测到联系方式、明显敏感词或其他必须修改的内容 | 页面顶部或侧边红色提示条，并联动正文高亮 | 是，阻止发送 | 用户修改违规内容并点击“重新检查”；检查通过后自动消失 |
| 不可定位阻断 | 检测到问题但无法定位到具体文本片段时 | 显示问题类型和修改建议，不做虚假高亮 | 是，阻止发送 | 用户修改后重新检查通过 |

阻断提示不可被直接关闭，避免用户误以为问题已解除。系统不得自动删除或改写用户正文；用户修改后，需通过“重新检查”从阻断状态恢复到正常编辑或发送成功状态。

#### 8.2.3 LetterEditor 与 SaveStatus 组合关系

LetterEditor 与 SaveStatus 是组合关系，而不是两个互不相关的独立组件。写信编辑页中，SaveStatus 应作为 LetterEditor 的固定子组件或紧邻状态栏展示。

| 组合关系 | 设计要求 |
|---|---|
| 空间位置 | SaveStatus 放在编辑器底部状态栏或编辑器标题区右侧，必须与正文输入区形成明确关联 |
| 状态同步 | 用户输入、自动保存、保存失败、重新保存均应即时更新 SaveStatus |
| 移动端 | 输入法弹出时，保存状态不能被底部按钮或键盘完全遮挡 |
| 失败处理 | 保存失败时保留本地草稿，并提供“重试保存”或检查网络提示 |

### 8.3 组件状态规范

| 状态 | 设计要求 | 不推荐做法 |
|---|---|---|
| 加载中 | 使用骨架屏或简洁文字，保留页面结构 | 整页空白或长时间无反馈 |
| 空状态 | 说明为什么为空，并提供下一步入口 | 只显示“暂无数据” |
| 错误 | 说明状态、可能原因和解决方式 | 只显示“系统异常” |
| 成功 | 告知完成结果和下一步 | 只闪现绿色提示后消失 |
| 禁用 | 说明为什么不可用 | 灰掉按钮但不解释 |
| 处理中 | 防止重复提交，显示进度 | 让用户反复点击无反馈 |

### 8.3.1 空状态与首次体验组件规则

空状态不是简单的“暂无数据”，而是新用户或当前筛选条件下的起步引导。每个空状态至少包含三部分：原因说明、下一步动作和可恢复路径。

| 场景 | 空状态视觉内容 | 主动作 |
|---|---|---|
| 单词库为空 | 说明当前还没有加入单词，可从基础词开始或从阅读中加入 | 开始基础词 / 返回首页 |
| 阅读列表为空 | 说明当前筛选无结果，建议放宽难度、主题或使用默认文章 | 清空筛选 / 查看默认短文 |
| 收件箱为空 | 说明暂无来信，不暗示用户被忽略 | 写一封信 / 返回今日学习 |
| 今日成果为空 | 说明今天还没有完成记录，并给出低门槛入口 | 开始复习 / 选择一篇短文 |
| 搜索无结果 | 保留搜索词，提示换关键词或清空筛选 | 清空搜索 |

空状态不应使用夸张插画或大面积装饰抢占学习入口；移动端必须保证主动作在首屏可见。

### 8.4 可复用页面模板

| 模板 | 适用页面 | 结构 | 包含组件 |
|---|---|---|---|
| 今日任务模板 | 首页 | 顶部问候 + 今日概览入口 + 任务卡 + 最近进度或提醒 + 回流提示 | AppShell、PrimaryNav、ProgressSummary（仅核心完成度）、TaskActionCard、RecoveryPrompt、NotificationCenter（可选） |
| 列表选择模板 | 单词库、阅读列表、收件箱 | 标题 + 筛选/搜索 + 卡片列表 + 空/错状态 | AppShell、PrimaryNav、LearningCard、EmptyState、ErrorState、FeedbackPanel |
| 专注学习模板 | 单词练习、阅读正文、写信编辑 | 任务标题 + 主内容 + 进度/保存状态 + 主操作 | AppShell、PrimaryNav、ProgressIndicator、PracticeQuestion 或 LetterEditor、SaveStatus（写信/可保存场景）、FeedbackPanel、SafetyNotice（写信） |
| 结果反馈模板 | 单词结果、阅读结果、今日成果 | 完成结果 + 关键指标 + 下一步入口 | AppShell、PrimaryNav、CompletionSummary、ProgressSummary、FeedbackPanel、TaskActionCard（下一步入口） |
| 页内安全检查模板 | 写信编辑页内检查状态 | 正文高亮 + 解释条 + 重新检查 + 返回编辑 | LetterEditor、SaveStatus、SafetyCheckStatus、SafetyNotice、SafetyHighlight、SafetyGuideLink、FeedbackPanel |

### 8.5 组件层未来扩展点

以下内容只作为后续版本的组件扩展预留，不属于 MVP 可见功能。首版不得把这些扩展点做成灰色入口或“敬请期待”占位。

| 扩展点 | 所属组件/页面 | 预留方式 | 首版展示规则 |
|---|---|---|---|

| 今日学习卡片 | 今日成果页 | 可选接入今日完成摘要、本地图片生成能力和隐私说明 | MVP 不实现，不提供公开分享入口 |

## 9. 交互反馈设计

### 9.1 全局反馈原则

| 原则 | 说明 |
|---|---|
| 即时反馈 | 用户点击、输入、提交后应立即看到视觉反馈 |
| 明确状态 | 告诉用户正在加载、已保存、保存失败或已完成 |
| 可操作 | 错误提示要给出下一步，不只说明失败 |
| 不打断学习 | 非紧急信息收纳到通知中心或页面内弱提示 |
| 不误导 | 保存失败不能提示成功；安全审核未通过不能显示发送成功 |

### 9.2 保存反馈

| 场景 | 状态 | 文案建议 |
|---|---|---|
| 写信编辑 | 保存中 | “正在保存草稿…” |
| 写信编辑 | 已保存 | “草稿已保存” |
| 写信编辑 | 保存失败 | “草稿暂未保存，请检查网络或点击重试” |
| 单词练习 | 已提交 | “答案已记录” |
| 阅读进度 | 已保存 | “阅读进度已保存” |

### 9.3 错误反馈

| 错误类型 | 页面表现 | 示例文案 |
|---|---|---|
| 字段输入错误 | 字段下方内联提示，焦点保留在可修改区域 | “昵称不能包含联系方式，请修改后继续” |
| 局部数据加载失败 | 局部错误卡，不影响其他模块入口 | “学习进度加载失败，点击重试” |
| 网络异常 | 页面内错误条或局部错误卡，保留用户输入 | “网络连接不稳定，已保留当前内容，请稍后重试” |
| 服务繁忙 | 页面内横幅或局部错误卡，提供重试和返回入口 | “服务暂时繁忙，你可以稍后重试” |
| 保存失败 | 编辑器附近持久状态栏或错误条，不用短暂 Toast 代替 | “草稿暂未保存，请检查网络或点击重试” |
| 内容安全拦截 | 阻断提示条 + 正文高亮 + 修改入口，必须保留正文 | “信中可能包含联系方式，请修改后再发送” |
| 外部能力超时 | 低强度提示并降级，不阻断核心流程 | “语言检查暂不可用，草稿已保存” |
| 系统级错误 | 独立错误页，保留返回首页/重试入口 | “页面暂时无法打开，你可以返回今日学习” |

错误展示层级规则：

| 层级 | 使用场景 | 不适合场景 |
|---|---|---|
| 字段内联提示 | 表单字段、局部输入校验 | 全局服务失败 |
| 局部错误卡 | 某个列表、卡片或模块加载失败 | 写信发送失败、安全拦截 |
| 顶部/页面内横幅 | 保存失败、网络不稳定、关系状态变化 | 普通成功提示 |
| 阻断提示条 | 安全检查未通过、未保存退出、不可逆操作确认 | 语言建议、普通提醒 |
| 系统错误页 | 404、403、500、离线或核心页面无法加载 | 单个组件失败 |

> 禁止用浏览器原生 `alert()` 承载安全检查失败、草稿保存失败或发送失败等关键状态。关键失败提示必须可见、可操作，并在问题解决前保持可追踪。

### 9.4 成功反馈

| 场景 | 成功反馈 | 下一步入口 |
|---|---|---|
| 完成单词练习 | “本轮已完成 10 个单词” | 继续复习 / 去阅读 / 返回首页 |
| 完成阅读 | “阅读完成，已保存记录” | 查看解析 / 加入生词 / 返回首页 |
| 信件发送成功 | “信件已发送” | 返回通信历史 / 写下一封 |
| 今日任务完成 | “今天的核心任务已完成” | 继续自由学习 / 查看记录 |

### 9.5 弹窗使用规则

| 可用弹窗 | 不应使用弹窗 |
|---|---|
| 解除关系确认 | 普通学习鼓励语 |
| 放弃未保存内容确认 | 营销、无关推荐 |
| 安全风险强提示 | 非紧急通知 |
| 删除或不可逆操作确认 | 阅读过程中的装饰动画 |

## 10. 内容与文案规范

### 10.1 文案语气

文案应保持清楚、鼓励、直接，避免夸张和责备。

| 推荐 | 不推荐 |
|---|---|
| “开始复习” | “开启词汇能量挑战” |
| “继续阅读” | “探索阅读世界” |
| “查看回信” | “发现你的惊喜” |
| “草稿已保存” | “系统已为你守护灵感” |
| “请修改联系方式后再发送” | “内容违规” |

### 10.2 按钮文案规则

| 场景 | 推荐按钮 |
|---|---|
| 首页单词入口 | 开始复习 |
| 首页阅读入口 | 继续阅读 / 选择文章 |
| 首页写信入口 | 查看回信 / 继续写信 / 创建档案 |
| 单词答题 | 提交答案 / 下一题 |
| 阅读答题 | 提交答案 / 查看解析 |
| 写信 | 保存草稿 / 检查并发送 |
| 错误恢复 | 重试 / 返回首页 / 检查网络 |

### 10.3 错误文案规则

错误文案采用三段式：

```text
发生了什么 -> 可能原因 -> 下一步操作
```

示例：

| 场景 | 文案 |
|---|---|
| 草稿保存失败 | “草稿暂未保存，可能是网络不稳定。请检查网络后重试。” |
| 阅读记录失败 | “阅读记录暂未保存。你可以先留在当前页面，稍后再次提交。” |
| 信件审核拦截 | “信中可能包含联系方式。请删除电话、邮箱或社交账号后再发送。” |
| 练习结果提交失败 | “本题结果暂未提交。请重试，避免重复刷新页面。” |

### 10.4 隐私与安全文案

| 场景 | 文案要求 |
|---|---|
| 创建笔友档案 | 明确“不需要真实姓名、电话、邮箱或详细地址” |
| 写信前 | 提醒不要发送联系方式和敏感个人信息 |
| 安全审核 | 说明是为了保护用户通信安全 |
| 解除关系 | 说明解除后的通信限制 |
| 今日成果 | 说明今日成果默认私有 |

## 11. 视觉语言与版式原则

### 11.1 整体视觉方向

视觉风格建议为：**温和、中性、清爽、以内容可读性为主**。

- 背景使用浅色或柔和中性色，避免高饱和大面积色块。
- 主按钮使用统一强调色，不同模块可用轻量辅助色区分，但不能只靠颜色表达含义。
- 文本层级通过字号、字重、间距体现，避免过多阴影和装饰。
- 学习内容页优先保证可读性，不追求复杂视觉表现。

### 11.2 色彩建议

| 色彩角色 | 使用场景 | 要求 |
|---|---|---|
| 主色 | 主按钮、当前导航、关键行动入口 | 全站一致，避免每页主色不同 |
| 成功色 | 答对、保存成功、发送成功 | 配合文字或图标，不只用绿色 |
| 警告色 | 安全提示、未保存、即将退出 | 控制使用频率，避免造成焦虑 |
| 错误色 | 保存失败、审核不通过、答错 | 文案必须说明原因和下一步 |
| 信息色 | 重点词、提示、弱通知 | 不抢主操作焦点 |
| 中性色 | 背景、边框、正文、次级文字 | 保证对比度和阅读舒适 |

### 11.3 字体与阅读性

| 项目 | 要求 |
|---|---|
| 正文 | 行高适中，长文本左对齐 |
| 英文阅读正文 | 段落间距清晰，避免过窄行宽 |
| 学习反馈 | 重点信息可加粗，但不过度使用大字号 |
| 表单提示 | 靠近输入区域，不放在页面远处 |
| 错误文案 | 使用普通语言说明，不使用技术错误码作为主文案 |

### 11.4 间距与布局

| 布局项 | 要求 |
|---|---|
| 页面宽度 | 学习内容区不宜过宽，阅读页保持舒适行长 |
| 卡片间距 | 同类卡片间距一致 |
| 主按钮位置 | 同类流程中位置保持稳定 |
| 底部操作 | 移动端可固定主操作，但不能遮挡正文或输入框 |
| 信息密度 | 首页适中，练习和阅读页更克制 |

### 11.5 动效原则

| 可用动效 | 避免动效 |
|---|---|
| 轻量加载、保存状态过渡 | 大面积背景动画 |
| 答题反馈的短暂状态变化 | 长时间庆祝动画 |
| 折叠面板展开收起 | 阅读中不断闪烁的提示 |
| 页面切换的轻微过渡 | 阻塞学习操作的复杂转场 |

动效必须服从性能目标和低干扰原则：

- 首页、单词练习页和阅读页不得为视觉效果引入大体积动画库或阻塞首屏资源。
- 加载态优先使用静态骨架屏或简洁文字；如使用 shimmer，应短时、低对比、可关闭，不作为信息表达的唯一方式。
- 状态过渡建议控制在 150ms 左右；超过 300ms 的动效不得用于核心学习路径。
- 保存、提交、检查等状态必须先改变文案和结构，再考虑动效；不能让动画延迟用户操作。
- 动效实现优先使用 CSS opacity/transform 等低成本属性，避免大面积重排、复杂 SVG 动画或持续循环动画。

前端实现约定：

| 对象 | 时间口径 | 示例 |
|---|---|---|
| 操作反馈 | ≤ 100ms | 按钮按下态、输入响应、选项选中态、保存按钮进入处理中 |
| 常规状态过渡 | 约 150ms | 卡片轻微展开、局部提示出现、轻量折叠展开 |
| 核心路径动效上限 | ≤ 300ms | 页面主内容切换、结果卡片出现；超过该时长不得用于单词练习、阅读正文、写信编辑等核心路径 |

> 这三个数字服务于不同对象：100ms 约束用户操作后的即时视觉反馈；150ms 约束普通 UI 过渡的舒适速度；300ms 是核心学习路径中动效时长上限。前端开发规范应沿用该口径。

减少动态效果实现路径：

| 实现层 | MVP 要求 |
|---|---|
| 系统偏好 | 使用 `prefers-reduced-motion: reduce` 作为默认识别方式 |
| 应用内设置 | MVP 不强制新增独立开关；若后续建设设置页，可提供覆盖系统偏好的选项 |
| 具体减少方式 | 关闭非必要过渡、骨架 shimmer、庆祝动画、复杂页面转场；保留必要状态文本和图标 |
| 验收口径 | 开启减少动态后，核心学习流程仍能通过文案、图标和布局理解状态 |

### 11.6 核心业务状态的视觉映射

视觉设计必须与业务状态对齐，不能只设计“正常有数据”的页面。以下状态为 MVP 视觉稿和前端实现的最低映射要求。

| 业务状态 | 推荐视觉形态 | 必须包含 | 禁止做法 |
|---|---|---|---|
| 加载中 | 保留页面结构的静态骨架屏或短文案 | 正在加载的对象、可预期区域 | 整页白屏、复杂循环动画 |
| 空状态 | 空状态卡片 + 下一步主按钮 | 为空原因、下一步入口 | 只写“暂无数据” |
| 保存中 | 编辑器附近的轻量状态栏 | “正在保存…”和不可丢稿暗示 | 用全屏遮罩阻断输入 |
| 保存失败 | 编辑器附近持久错误条或状态栏 | 失败说明、重试入口、本地暂存说明 | 短暂 Toast 后消失、仍显示保存成功 |
| 未保存退出 | 确认对话框 | 留在页面、重试保存、仍然离开 | 直接离开或浏览器原生弹窗替代 |
| 完成 | 结果页或完成卡片 | 完成数、下一步入口 | 只闪现绿色提示 |
| 恢复/回流 | 首页或流程入口的回流提示卡 | 继续上次、开始新任务 | 历史任务堆积式提醒 |
| 安全检查失败 | 写信编辑页内阻断提示条 + 正文高亮 | 拦截原因、问题位置/类型、重新检查、规则说明入口 | 跳出编辑页、清空正文、只用 alert |
| 语言建议 | 编辑器底部低强度信息条 | 建议数量、可修改也可直接发送 | 使用红色阻断条或阻止发送 |
| 系统错误 | 系统错误页 | 错误类型、重试、返回今日学习 | 浏览器默认错误页或空白页 |

安全检查失败的视觉要求：

- 可定位内容使用浅色背景、高亮下划线或边框，并配合图标/文本说明，不能只依赖红色。
- 顶部或侧边解释条应与正文高亮联动，点击说明可定位到对应文本。
- “重新检查”按钮在用户修改后可用；检查通过后阻断提示自动消失。
- 不可定位问题应说明问题类型，例如“可能包含私人联系方式”，不得伪造高亮范围。

未保存退出的视觉要求：

- 使用确认对话框，标题直接说明“草稿尚未保存”或“本轮进度尚未保存”。
- 主按钮优先为“留在页面”或“重试保存”，危险按钮为“仍然离开”。
- 若已有本地暂存，应说明“当前内容已暂存在本地，但尚未同步到服务器”。

### 11.7 错误页与系统级异常视觉规范

系统级错误页属于用户体验的一部分，不应交给浏览器默认页面。

| 错误页 | 使用场景 | 页面内容 | 主操作 |
|---|---|---|---|
| 404 页面不存在 | 路由不存在、内容已删除 | “页面不存在或已被移动”，不显示技术堆栈 | 返回今日学习 / 返回上一页 |
| 403 无权限 | 未登录、关系已解除、访问他人数据 | “当前无法访问该内容”，说明可能原因 | 返回首页 / 查看关系状态 |
| 500 服务异常 | 服务端异常、核心接口不可用 | “服务暂时不可用”，说明可稍后重试 | 重试 / 返回今日学习 |
| 离线/网络中断 | 浏览器离线或请求持续失败 | “网络连接不可用，已尽量保留当前内容” | 重试 / 查看已保存内容 |

系统错误页不得暴露数据库、接口栈、服务器路径、异常堆栈等技术细节；也不得显示没有出口的白屏。

## 12. 响应式与无障碍设计

### 12.1 响应式原则

| 设备 | 布局策略 |
|---|---|
| 桌面端 | 首页可采用多列行动卡；阅读页 MVP 默认仍采用正文内联轻提示，辅助侧栏仅作为后续增强 |
| 平板端 | 保持两列或单列自适应，主操作清晰 |
| 移动端浏览器 | 单列布局，底部操作不遮挡正文和编辑器 |

### 12.2 移动端重点

- 一级导航应可折叠或底部展示，但不能占用过多屏幕。
- 阅读正文行宽、字号和行距应适合小屏阅读。
- 写信编辑器应避免被键盘遮挡主要操作。
- 表单字段不宜过多，安全档案首版尽量简化。
- 触控按钮需要有足够点击区域。

### 12.3 无障碍要求

| 项目 | 要求 |
|---|---|
| 键盘操作 | 主要按钮、表单、答题选项可通过键盘访问 |
| 焦点状态 | 当前焦点清晰可见 |
| 对比度 | 正文、按钮、错误提示应有足够对比度 |
| 语义结构 | 标题、列表、按钮、表单标签语义清楚 |
| 颜色使用 | 正确/错误/警告不能只靠颜色表达 |
| 动效 | 支持减少动效偏好 |
| 表单错误 | 错误提示靠近字段，并说明如何修正 |

## 13. 隐私、安全与信任体验

### 13.1 数据默认私有

所有今日成果和通信内容默认私有。界面应避免让用户误以为学习结果、信件正文或笔友互动会公开展示。

| 页面 | 隐私设计 |
|---|---|
| 首页 | 只展示自己的学习进度和任务 |
| 今日成果 | 不展示信件正文、笔友敏感信息或公开排名 |
| 笔友档案 | 明确可见字段，不要求真实身份信息 |
| 收件箱 | 列表只展示必要摘要，不泄露过多正文 |

### 13.2 笔友安全体验

| 安全点 | UI 要求 |
|---|---|
| 关系权限 | 写信入口根据关系状态显示可用/不可用 |
| 联系方式拦截 | 拦截时说明问题并保留草稿 |
| 举报 | 入口清楚但不过度吓人 |
| 屏蔽/解除关系 | 操作可找到，结果明确 |
| 审核失败 | 不责备用户，提供修改建议 |

### 13.3 安全审核交互状态机

```text
编辑中
  -> 草稿已保存
  -> 点击“检查并发送”
  -> 安全检查中
      -> 通过：写入发送记录 -> 通信历史可见
      -> 拦截：保留草稿 -> 显示原因 -> 用户修改 -> 重新检查
      -> 超时/服务失败：保留草稿 -> 本地规则降级或提示稍后重试
```

| 状态 | 用户应看到什么 | 系统不得做什么 |
|---|---|---|
| 编辑中 | 收信对象、关系状态、字数、保存状态 | 不隐藏关系异常 |
| 草稿已保存 | 最后保存时间或“草稿已保存” | 不在未保存时诱导用户离开 |
| 安全检查中 | 明确检查正在进行，可取消返回编辑 | 不让页面无限转圈 |
| 拦截 | 正文高亮疑似内容、顶部/侧边解释条、修改建议、原文仍在编辑器 | 不清空、不自动改写、不只说“违规”、不让用户自己全文搜索 |
| 超时/失败 | 草稿已保留、可重试或稍后发送 | 不提示发送成功 |
| 发送成功 | 收信对象、发送时间、通信历史入口 | 不在历史未写入时显示成功 |
| 待人工审核（后续可选） | 预计处理时间、可撤回或继续编辑副本 | 不作为 MVP 默认流程，不无限等待 |

### 13.4 信任建立方式

- 在写信前提示“不要发送联系方式和敏感个人信息”。
- 在写信页提供“安全与隐私指南”入口，前置说明哪些内容可能被拦截。
- 在档案页说明“昵称和兴趣会被笔友看到，真实身份信息不会被要求”。
- 在今日成果页说明“今日成果仅用于个人进度反馈”。
- 在 AI 或外部服务接入前，必须说明数据使用和降级方案。

## 14. 性能与稳定性体验要求

### 14.1 体验侧性能要求

性能体验必须用可测试指标表达，避免只使用“快”“即时”等主观词。

| 要求 | 量化口径 | 设计含义 |
|---|---|---|
| 核心页面首屏快 | 首页、单词练习、阅读详情 FCP ≤ 1.5 秒，LCP ≤ 2.5 秒 | 首屏优先渲染标题、主任务卡、正文或练习主体；不依赖装饰图片、复杂动效或 AI 结果 |
| 交互反馈及时 | 点击、输入、选择题作答、保存按钮等视觉反馈 ≤ 100ms；异步操作超过 300ms 必须显示处理中状态 | 用户操作后必须立即看到按钮状态、输入响应、保存中或检查中提示 |
| 页面切换流畅 | 核心页面切换不出现长时间白屏；长列表首屏只渲染必要项目，后续分页、懒加载或虚拟滚动 | 单词库、阅读列表和收件箱不能一次性渲染过多内容 |
| 非核心资源延迟 | 非首屏图片、装饰插画、复杂动效、AI 辅助、第三方脚本不得阻塞 FCP/LCP | 学习主路径先可用，增强资源后加载 |
| 加载状态可感知 | 请求超过 300ms 显示局部 loading；超过 1 秒给出更明确的等待说明或骨架屏 | 避免用户误以为页面卡死 |

> 说明：以上指标与第二阶段的非功能需求保持一致。若后续真实设备或网络环境无法达到指标，应在测试报告中说明环境、原因和优化计划，不能在设计验收中把“快”作为模糊判断。

### 14.2 稳定性体验要求

| 场景 | 设计要求 |
|---|---|
| 单词练习刷新 | 已提交结果不应丢失；未完成会话可恢复或说明 |
| 阅读中断 | 保留阅读位置和已答题状态 |
| 写信刷新 | 默认恢复到最后保存或本地暂存内容；安全失败、网络失败、语言检查失败都不能导致草稿丢失 |
| 网络状态变化 | 从在线变为离线时，在页面内显示非阻断提示，并切换为本地暂存/稍后同步；从离线恢复在线时提示可重试同步或自动重试结果 |
| 外部接口超时 | 按接口类型分级降级，不让非核心依赖阻塞核心学习流程 |
| 保存失败 | 不提示成功，不清空用户输入 |

网络状态变化提示：

| 状态变化 | 用户感知要求 | 可继续操作 |
|---|---|---|
| 在线 -> 离线 | 页面内横幅或编辑器附近状态条提示“网络已断开，当前内容将暂存在本地” | 可继续写信、阅读或查看已加载内容；提交类操作进入待同步或提示稍后重试 |
| 离线中 | 保存状态显示“本地暂存”或“等待网络恢复”，不显示“已同步” | 可继续编辑草稿；不允许误提示发送成功 |
| 离线 -> 在线 | 提示“网络已恢复，可重新同步/重试发送”；若自动重试，必须显示结果 | 可手动重试保存、提交或发送 |

外部接口超时的场景化降级：

| 依赖类型 | 示例 | 超时建议 | 降级策略 |
|---|---|---|---|
| 安全检查 | 写信发送前安全审核 | 5—8 秒 | 先使用本地基础规则降级；若本地规则不可用，保留草稿并暂不发送，不得绕过安全底线 |
| 语言检查 | 语法、拼写、表达建议 | 5—8 秒 | 提示“语言检查暂不可用”，允许用户直接发送或稍后再检查，不阻断发送 |
| 内容推荐 | 阅读推荐、任务推荐 | 3—5 秒 | 展示默认短文、最近阅读或手动选择入口，不阻塞阅读列表 |
| 学习记录统计 | 今日成果汇总、完成度刷新 | 3—5 秒 | 展示最近一次已知结果或局部错误卡，核心学习入口仍可用 |
| 非核心增强 | 装饰图、成就卡、AI 润色候选 | 3—5 秒 | 延迟加载或隐藏该增强，不影响学习主流程 |

> 稳定性底线：凡涉及用户已输入内容、已提交答案、阅读位置和信件草稿的场景，均应优先保留数据和状态；“尽量恢复”不能替代“默认恢复到最后保存或本地暂存内容”的设计口径。

## 15. 原型页面清单与评审标准

### 15.1 MVP 原型页面清单

| 页面编号 | 页面名称 | 主要目标 | 是否 MVP 必须 |
|---|---|---|---|
| UX-01 | 今日学习首页 | 固定任务清单和核心进度 | 是 |
| UX-02 | 今日复习会话页 | 从首页直达本轮复习任务 | 是 |
| UX-02B | 单词库页 | 自主查找新词、待复习词、困难词 | 是 |
| UX-03 | 单词练习页 | 主动回忆、即时反馈 | 是 |
| UX-04 | 单词结果页 | 展示本轮结果和下一步 | 是 |
| UX-05 | 阅读列表页 | 选择文章或继续阅读 | 是 |
| UX-06 | 阅读详情页 | 专注阅读和内联轻提示式重点词辅助 | 是 |
| UX-07 | 阅读答题/结果页 | 提交题目、查看解析依据 | 是 |
| UX-08 | 笔友安全档案页 | 创建安全交流身份 | 是 |
| UX-09 | 收件箱/通信历史页 | 查看来信和关系状态 | 是 |

| UX-12 | 今日成果页 | 今日任务完成度、今日学习时长、各模块今日完成情况 | 是 |
| UX-13 | 通知中心/弱提示页 | 收纳非紧急消息 | 可选 P1 |
| UX-14 | 系统错误页 | 404、403、500、离线/网络中断的返回与重试引导 | 是 |

### 15.2 页面评审标准

每个核心页面评审时至少检查：

1. 页面目标是否明确。
2. 主按钮是否清楚，且同类流程位置一致。
3. 从首页到核心任务是否不超过 3 层。
4. 是否包含加载、空状态、错误状态和成功状态。
5. 是否避免学习过程中非紧急弹窗打断。
6. 是否包含必要的隐私和安全提示。
7. 是否有移动端适配方案。
8. 是否没有引入非本期范围功能。
9. 任务进行中是否能看到进度、保存或检查状态。
10. 多日未学习时是否提供友好回流，而不是堆积任务。
11. 安全拦截是否同时提供正文高亮、解释条和规则说明入口。
12. 错误、空状态、保存失败、系统错误页是否按视觉层级展示，且提供下一步。
13. 动效是否不阻塞核心页面 FCP/LCP 和点击输入反馈，且支持 `prefers-reduced-motion`。
14. 网络在线/离线变化是否有用户可感知提示，且写信草稿默认恢复到最后保存或本地暂存内容。
15. 外部接口超时是否按安全检查、语言检查、内容推荐、学习记录统计等场景分别降级。

## 16. UX 验收标准

### 16.1 总体验收

| 编号 | 验收项 | 验收口径 |
|---|---|---|
| UX-A01 | 首页任务清晰 | 用户进入首页后能看到单词、阅读、写信三个核心行动入口，并能直达今日复习会话 |
| UX-A02 | 页面层级受控 | 从首页进入单词练习、阅读文章、写信编辑均不超过 3 层 |
| UX-A03 | 反馈完整 | 学习、保存、发送、失败均有明确状态和下一步 |
| UX-A04 | 干扰受控 | 单词测验、阅读正文、写信编辑中不出现非紧急全屏弹窗 |
| UX-A05 | 文案行动导向 | 主按钮使用动词表达，如“开始复习”“继续阅读”“检查并发送” |
| UX-A06 | 隐私安全可感知 | 笔友档案和写信页说明可见信息与联系方式限制 |
| UX-A07 | 草稿保护可见 | 写信页显示保存状态，审核失败不清空正文 |
| UX-A08 | 移动端可用 | 核心内容不被导航、底部按钮或键盘遮挡 |
| UX-A09 | 过程进度可见 | 单词、阅读、写信过程中能看到进度、保存或检查状态 |
| UX-A10 | 中断回流友好 | 多日未学习时提供继续上次/开始新任务，不展示压迫式积压任务 |
| UX-A11 | 视觉状态完整 | 加载、空、错误、保存中、保存失败、完成、恢复、安全拦截和系统错误页均有对应视觉形态 |
| UX-A12 | 动效性能受控 | 不使用阻塞首屏的大体积动画或长转场；减少动态效果偏好下状态仍可理解 |

### 16.2 模块体验验收

| 模块 | 验收重点 |
|---|---|
| 首页 | 任务卡数据来自真实记录；无数据时有冷启动任务；多日未学习时有友好回流选择；连续学习天数只弱展示 |
| 单词 | 首页开始复习直达今日复习会话；每题有即时反馈，并显示进度条和题目序号 |
| 阅读 | 正文可读，解析有原文依据，重点词辅助默认采用内联轻提示且不遮挡正文 |
| 写信 | 安全检查在写信编辑页内完成；有草稿状态、关系权限反馈；安全拦截同时高亮正文并解释原因；语言建议与安全拦截视觉区分，且语言建议不阻断发送 |
| 今日成果 | 专注今日任务完成度、今日学习时长和各模块今日完成情况，不把连续学习 0 天或未发送草稿作为核心成果，不暴露私人通信内容 |

### 16.3 不通过标准

出现以下任一情况，本阶段 UX 设计不得视为通过：

- 首页只有静态模块入口，没有今日任务清单、真实状态提示、中断回流选择，或“开始复习”不能直达今日复习会话。
- 单词、阅读或写信任一核心流程超过 3 层且无必要理由。
- 写信审核失败会跳离编辑上下文、清空正文、没有草稿恢复提示，或只给笼统错误但不高亮/解释具体问题。
- 解除关系后界面仍允许继续发信。
- 阅读或测验中出现非紧急全屏弹窗。
- 首页今日概览平铺今日学习时长和各模块完成列表，导致与今日成果页内容重复。
- 阅读重点词辅助默认采用占用空间明显的侧栏或强制折叠面板，移动端遮挡正文。
- 语言建议使用与安全拦截相同的红色阻断样式，或语言检查不可用时阻断发送。
- 今日成果展示虚构数据、私人信件敏感内容，把连续学习 0 天作为新用户完成后的核心反馈，或把未发送草稿作为已完成成果。
- 页面只有正常状态，没有加载、空、错误、失败、过程进度和恢复状态设计。
- 关键错误使用浏览器原生 `alert()`、短暂 Toast 或后端技术错误码替代可操作的页面内反馈。
- 保存失败、安全检查失败、未保存退出没有持久可见提示，或没有重试/修改/返回入口。
- 首次使用或无数据时只有空白页或“暂无数据”，没有起步引导。
- 404、403、500、离线等系统异常显示浏览器默认错误页、白屏或无返回入口页面。
- 核心流程引入大面积循环动画、复杂转场或无法响应 `prefers-reduced-motion` 的动效。
- 设计稿中加入开放社区、实时聊天、付费、电商等非本期功能。

## 18. UX 范围控制

### 18.1 本期必须完成的 UX 范围

- 首页固定今日任务清单、三个核心模块导航、今日复习会话直达、任务生成逻辑和中断回流选择。
- 今日复习会话、单词库、练习、过程进度、反馈、结果页面体验。
- 阅读列表、详情、答题、解析结果页面体验。
- 笔友安全档案、收件箱、写信编辑、页内安全检查状态、草稿保存状态、正文高亮拦截、解释条、规则说明入口和关系状态体验。
- 今日成果页的今日任务完成度、今日学习时长（可选）和各模块今日完成情况展示体验。
- 加载、空、错误、保存失败、审核失败、过程进度、回流恢复等关键状态设计。
- 基础响应式和无障碍设计要求。

### 18.2 可以延后的 UX 范围

| 体验项 | 延后原因 |
|---|---|
| 高亮笔记和写作素材管理 | 不影响阅读 MVP 闭环 |
| 阅读策略模式和能力雷达 | 需要更多内容和数据支持 |
| AI 润色和复杂语言反馈界面 | 外部能力、隐私告知和降级方案未确认 |
| 学习周报、趋势图、历史复盘 | 首版只需关键反馈信号 |
| 主题公开信和同伴反馈 | 可能引入社区化和审核成本 |
| 复杂成就系统和排行榜 | 容易偏离学习任务导向 |
| 今日成果仪式感与本地学习卡片 | 属于情感化正反馈增强，需先验证今日成果页使用价值和隐私保存边界 |
| 单词详情关联阅读入口 | 需要跨模块词汇-阅读索引、内容授权和数据质量支持，首版只保留组件/数据扩展点 |

### 18.3 明确不设计

- 付费、会员、订单、商城页面。
- 开放广场、群聊、动态流、实时聊天页面。
- 大规模笔友搜索和复杂匹配推荐页面。
- 管理后台完整工作台和运营数据大屏。
- 自动代写整封信的交互。
- 用虚构学习数据营造活跃感的页面。
- 公开分享型学习卡片、社交炫耀式成就页或强竞争展示。

### 18.4 防止 UX 需求蔓延的判断标准

新增 UX 方案进入本期前，必须同时满足：

1. 直接服务单词、阅读、写信或今日成果核心闭环。
2. 不增加核心流程页面层级。
3. 不引入复杂后台、实时通信、社区运营或重型推荐。
4. 有明确用户场景和验收标准。
5. 能在加载失败、空状态和异常状态下提供可控体验。
6. 不弱化隐私、安全和草稿保护底线。

## 19. 风险与缓解措施

| 风险 | 可能影响 | 缓解措施 |
|---|---|---|
| 首页信息过多 | 用户不知道先做什么 | 首页只突出三个固定任务槽位和核心进度，不做信息流 |
| 任务积压造成挫败 | 用户中断后不愿回流 | 提供继续上次/开始新任务，不机械累加历史任务 |
| 过程进度不可见 | 用户不知道还剩多久或是否已保存 | 单词显示 15/30，写信显示自动保存状态，阅读显示进度 |
| 单词模块退化为浏览词典 | 记忆闭环不成立 | 练习页必须包含主动回忆和即时反馈 |
| 阅读页辅助信息过多 | 打断阅读专注 | MVP 重点词采用内联轻提示；侧栏、折叠面板和策略提示延后 |
| 写信模块社交化过度 | 安全和审核成本增加 | 只做受控关系，不做开放社区和实时聊天 |
| 安全审核误判 | 用户写作成果受损 | 审核失败必须保留草稿、正文高亮问题内容并提供解释条 |
| 今日成果复杂化 | 首版开发成本过高且稀释即时成就感 | 只展示今日任务完成度、今日学习时长和各模块今日完成情况，趋势分析延后 |
| 文案过度营销化 | 降低学习产品可信度 | 使用直接、鼓励、行动导向文案 |
| 移动端操作被遮挡 | 小屏不可用 | 设计评审必须检查输入框、底部按钮和导航遮挡 |

## 21. 阶段结论

第三阶段的产品与用户体验设计结论是：**EnglishLover 首版应围绕“今日学习”建立清晰、低干扰、可恢复的学习工作台体验。首页以“系统建议 + 用户选择”的固定今日任务清单把用户引导到单词、阅读和写信三个核心任务，其中“开始复习”直达今日复习会话，并在中断后提供继续上次或开始新任务的友好回流；各模块页面只保留完成学习闭环所需的信息、操作、过程进度和反馈；今日成果页专注今日任务完成度、今日学习时长和各模块今日完成情况；笔友模块必须把安全档案、关系权限、草稿保护、页内发送前审核、正文高亮拦截和规则透明作为体验底线。**

本期 UX 设计必须坚持范围控制：不把首版做成开放社区、复杂数据平台、AI 自动写作工具或商业化学习商城。后续扩展应建立在 MVP 流程稳定、真实用户反馈明确、安全机制可控和数据口径可靠的基础上。
