## 1. 信息缺口与假设前提

### 1.1 快速识别的信息缺口

当前材料已经能支撑第二阶段需求规格与范围控制，但以下信息仍会影响后续详细设计、排期和验收：

1. **账号策略未最终确认**：首版是本地匿名学习记录、测试账号，还是正式登录账号。
2. **学习内容来源未最终确认**：单词库、阅读材料、音频和例句的来源、授权和分级标准尚未确定。
3. **输入型题型与音频能力后续范围未最终确认**：首版单词主动回忆题型已锁定为选择题；拼写、听写、朗读等输入型或音频增强能力作为后续范围确认。
4. **正式笔友关系机制未最终确认**：首版仅采用管理员预配置的测试笔友关系；用户搜索、邀请、同意和开放匹配机制延后。
5. **诊断/自评流程后续形态未最终确认**：首版阅读冷启动不提供自评或诊断入口，后续是否加入需单独评估。
6. **是否接入 AI 能力未最终确认**：语言检查、阅读生成、内容审核是否使用 AI，以及数据上传和隐私告知方式尚未确定。
7. **部署目标和测试工具未最终确认**：会影响性能指标、E2E 测试方案和上线验收。

### 1.2 本文采用的合理假设

在未获得进一步补充材料前，本文按以下假设推进：

| 编号 | 假设 | 对需求的影响 |
|---|---|---|
| A1 | 首版以 Web 端 MVP 为目标，不建设移动 App | 响应式适配以浏览器访问为主 |
| A2 | 平台保持免费，不包含付费课程、会员、电商和商业转化流程 | 本期不设计支付、订单、会员权益 |
| A3 | 首版优先验证核心学习闭环，而不是一次性建设完整社区和复杂推荐 | 推荐、公开社区、后台运营均做范围控制 |
| A4 | 笔友模块采用受控一对一写信闭环，首版关系由管理员在后台或配置表中预先绑定测试账号 | 必须做关系权限和安全审核，但不实现用户主动搜索、邀请和同意机制 |
| A5 | 学习记录前端只展示关键反馈信号，复杂趋势先留数据口径和接口 | 避免首版数据看板过重 |
| A6 | 若 AI 能力暂不可用，系统必须有人工/规则或降级流程 | AI 不作为核心学习流程的唯一依赖 |
| A7 | 首版单词主动回忆题型为 4 选 1 释义选择题，阅读冷启动直接推荐 3 篇中低难度短文 | 降低开发和测试不确定性，输入型题型与诊断流程延后 |

## 3. 业务目标与问题定义

### 3.1 业务目标

| 目标编号 | 业务目标 | 说明 | 成功表现 |
|---|---|---|---|
| BO-01 | 帮助用户稳定积累词汇 | 通过主动回忆、复习排程和即时反馈形成词汇记忆闭环 | 用户能完成一次单词学习，并生成后续复习计划 |
| BO-02 | 帮助用户在语境中理解英语 | 通过分级文章、重点词辅助、阅读题和解析反馈提升理解训练质量 | 用户能完成一篇阅读并获得解析与记录 |
| BO-03 | 帮助用户进行英文输出训练 | 通过受控笔友写信流程提供真实或近真实的表达场景 | 用户能完成一次受控写信、发送和保存流程 |
| BO-04 | 帮助用户获得清晰学习反馈 | 通过基础学习记录展示今日进度、连续学习和模块完成数量 | 用户能判断自己今天做了什么、下一步做什么 |
| BO-05 | 保证首版范围可控、可交付、可扩展 | 避免复杂社区、复杂推荐和重型数据看板拖慢 MVP | 研发能围绕核心闭环排期和验收 |

### 3.2 问题定义

当前英语自主学习容易出现以下问题：

1. **词汇学习停留在浏览层面**：用户看过单词但没有主动回忆和复习排程，容易遗忘。
2. **阅读训练缺少闭环**：文章、题目、解析和阅读记录未形成连续训练路径。
3. **写作练习缺少真实场景**：用户不知道写给谁、写什么，也缺少安全的互动反馈。
4. **学习反馈过弱**：用户完成任务后难以判断学习进展和下一步行动。
5. **需求容易膨胀**：社区、AI、推荐、后台、数据看板等功能都可能扩大范围，影响首版稳定上线。

### 3.3 本期解决思路

本期采用“核心闭环优先 + 数据口径预留 + 后续增强分层”的策略：

```text
首页入口
  -> 单词主动回忆与复习排程
  -> 阅读练习与解析反馈
  -> 受控笔友写信闭环
  -> 基础学习记录反馈
  -> 后续按数据与风险逐步增强
```

## 5. 核心使用场景

### 5.1 场景 S1：用户完成一次单词复习

```text
进入首页 -> 查看今日待复习 -> 进入单词练习 -> 完成主动回忆题 -> 查看反馈 -> 系统更新掌握度与下次复习时间 -> 查看今日完成度
```

| 业务价值 | 让用户从“看单词”转向“回忆 + 复习”，提升记忆闭环完整性 |
|---|---|
| 本期要求 | 支持 4 选 1 释义选择题、即时反馈、基础排程、复习队列优先级和中断保存 |
| 验收重点 | 练习结果可保存，下次复习计划可生成，到期词优先进入复习队列，错词和低掌握词在上限内回流 |

### 5.2 场景 S2：用户完成一篇阅读练习

```text
进入阅读模块 -> 选择分级文章 -> 阅读正文和重点词 -> 完成三类基础题 -> 查看解析和原文依据 -> 保存阅读记录
```

| 业务价值 | 让阅读从静态文章浏览变成有任务、有反馈、有记录的理解训练 |
|---|---|
| 本期要求 | 分级文章、重点词辅助、主旨/细节/词义猜测题、解析反馈、断点续读 |
| 验收重点 | 用户能完成阅读闭环，题目和解析有原文依据，记录可保存 |

### 5.3 场景 S3：用户完成一次受控笔友写信

```text
进入笔友模块 -> 创建安全档案 -> 选择管理员预配置测试笔友 -> 阅读来信或进入写信 -> 自动保存草稿 -> 安全检查 -> 发送成功 -> 写入通信历史
```

| 业务价值 | 为写作练习提供具体对象和真实交流感，同时控制安全风险 |
|---|---|
| 本期要求 | 安全档案、管理员预配置测试关系、收件箱、编辑器、草稿、审核、举报/解除关系 |
| 验收重点 | 违规内容可拦截，误判不丢稿，解除关系后不能继续发信 |

### 5.4 场景 S4：用户查看学习进展

```text
完成学习任务 -> 进入首页或学习记录页 -> 查看今日任务完成度、连续学习天数和模块完成数量 -> 选择下一步入口
```

| 业务价值 | 给用户关键反馈，降低坚持学习的不确定感 |
|---|---|
| 本期要求 | 前端优先展示三个核心信号，复杂趋势先后台记录或预留接口 |
| 验收重点 | 数据来自真实学习行为，更新及时，不暴露隐私内容 |

## 6. 版本分层与优先级规则

### 6.1 版本分层

| 版本层级 | 定义 | 交付原则 |
|---|---|---|
| V1.0 MVP 必须 | 没有就无法形成首版核心学习闭环或安全底线 | 必须排期、必须验收 |
| V1.1 体验增强 | 不影响核心闭环成立，但能明显提升学习体验或降低挫败 | 有资源则排期，可独立延期 |
| V2.0 扩展能力 | 涉及复杂算法、社区运营、后台管理、多端同步或 AI 深度能力 | 需单独立项或进入后续迭代 |
| 明确不做 | 与免费学习平台目标不一致或明显扩大范围 | 本期及近期均不纳入 |

### 6.2 优先级定义

| 优先级 | 含义 | 判断标准 |
|---|---|---|
| P0 必须有 | 缺失会导致核心流程不可用、安全不可控或数据不可保存 | 阻断上线 |
| P1 应该有 | 对学习效果、体验或稳定性有明显价值，但可通过降级方案上线 | 可视资源排期 |
| P2 可以有 | 有价值但不影响首版闭环，可作为增强项 | 延后不影响 MVP |
| P3 暂不做 | 复杂度高、偏离核心目标或缺少前置条件 | 本期排除 |

### 6.3 优先级依据

需求优先级按以下顺序判断：

1. 是否支撑单词、阅读、写作、学习反馈四个核心闭环。
2. 是否影响数据保存、内容安全、隐私保护等底线。
3. 是否能显著降低用户学习阻力。
4. 是否存在低成本降级方案。
5. 是否会引入复杂运营、复杂算法、跨端同步或后台管理成本。

## 7. 功能需求规格

### 7.0 通用学习会话管理规则

为避免单词练习和阅读练习在中断恢复、重复提交和进度保存上出现不同实现，本期统一采用“学习会话”规则。一次学习会话指用户从某个学习入口开始，到完成、暂停或重新开始为止的一组连续学习任务。

| 状态 | 触发条件 | 系统行为 | 验收口径 |
|---|---|---|---|
| 创建 | 用户从入口点击“开始学习”“开始复习”或“开始阅读” | 创建会话记录，写入任务范围、开始时间和初始进度 | 会话有唯一 `sessionId`，任务范围可追踪 |
| 进行中 | 用户完成题目、阅读步骤或提交答案 | 实时保存会话进度和已提交结果 | 刷新页面后，已完成步骤仍可识别 |
| 完成 | 用户完成会话内所有任务并点击“完成” | 关闭会话，生成学习记录和统计事件 | 完成后不再作为未完成会话提示 |
| 中断 | 用户主动退出、关闭页面、刷新失败或网络中断 | 会话状态标记为 `interrupted`，保留已提交结果和当前位置 | 不静默丢失已答题目、阅读位置或草稿 |
| 恢复 | 用户再次进入同一学习单元 | 系统提示“检测到未完成的学习，是否继续？”；选择“继续”则从断点恢复，选择“重新开始”则清除当前会话进度并创建新会话 | 已提交答题结果永久保留，不可覆盖，也不可重复计入统计 |

通用规则：

- GSL-R01：单词练习和阅读练习均应使用学习会话记录支撑中断恢复。
- GSL-R02：已提交答题结果属于正式学习日志，不因重新开始会话而被覆盖或重复计算。
- GSL-R03：重新开始只清除当前未完成会话的进度，不删除已产生的练习日志、阅读记录或学习统计事件。
- GSL-R04：如果恢复所需内容已下架或数据损坏，应提示用户原因，并提供重新开始入口。

| 数据对象 | 最小字段 | 说明 |
|---|---|---|
| LearningSession | sessionId、userId/localUserId、module、entryType、targetIds、status、startedAt、lastSavedAt、completedAt、resumePayload | 通用学习会话；`module` 可取 `word`、`reading`，`status` 可取 `active`、`completed`、`interrupted`、`discarded` |

### 7.2 单词记忆模块

#### 7.2.1 本期目标

单词模块本期必须完成“学习 -> 主动回忆 -> 即时反馈 -> 保存结果 -> 生成复习计划”的最小闭环，避免退回到只浏览单词和手动标记状态。

#### 7.2.2 功能需求

| 需求编号 | 需求名称 | 说明 | 业务价值 | 优先级 | 依赖 | 验收口径 |
|---|---|---|---|---|---|---|
| VOC-01 | 单词列表与分类入口 | 展示新词、待复习、困难词等入口，支持关键词搜索和学习状态筛选 | 帮助用户找到要学的词 | P0 | 单词数据 | 用户可按状态进入对应列表，搜索有结果/空状态提示 |

| REC-R06 | 笔友写作完成次数以 `penpal_activity_completed` 事件为准，由学习记录模块准实时消费该事件并更新 `DailySummary.writingCompletedCount`；信件草稿保存不计入写作完成数量。定时任务如后续存在，仅用于对账和修复异常汇总，不作为首版前端展示的主路径。 |

#### 7.5.4 数据要求

| 数据对象 | 最小字段 | 说明 |
|---|---|---|
| LearningActivityLog | logId、activityType、activityId、startedAt、endedAt、resultStatus、sourceModule、createdAt | 统一行为日志 |
| DailySummary | date、wordCompletedCount、readingCompletedCount、writingCompletedCount、taskCompletionRate、streakIncluded | 每日汇总；`writingCompletedCount` 由 `penpal_activity_completed` 事件准实时更新 |
| StreakState | userId/localUserId、currentStreak、lastActiveDate、timezoneRule | 连续学习天数 |
| ModuleCompletionSummary | module、completedCount、lastCompletedAt | 模块完成数量 |

## 9. 系统边界与上下游依赖

### 9.0 跨模块接口契约（草稿）

为支持前端、后端和测试并行推进，本期先固定关键跨模块交互的最小接口契约。正式接口字段可在接口设计文档中继续细化，但不得改变以下业务语义；如确需改变，应按 14.5 的需求变更控制流程同步修订本文、接口文档和测试用例。

#### 9.0.1 阅读模块 -> 单词模块：加入待学习队列

| 项目 | 约定 |
|---|---|
| 接口名称 | `addToWordLearningQueue` |
| 提供方 | 单词模块服务 |
| 调用方 | 阅读模块 |
| 触发条件 | 用户在阅读文章重点词区域点击“加入学习” |
| 输入 | `{ userId, wordId, source: "reading", articleId }` |
| 输出 | `{ status: "success" \| "duplicate" \| "error" }` |
| 业务语义 | 仅负责在用户-单词关联表中创建或确认一条 `learningStatus: "queued"` 的记录，不执行实际学习操作，不直接计入已掌握 |
| 验收口径 | 首次加入返回 `success`；重复加入返回 `duplicate` 或等价提示；失败时用户能看到可操作错误提示 |

#### 9.0.2 笔友模块 -> 学习记录模块：写作活动完成事件

| 项目 | 约定 |
|---|---|
| 事件名称 | `penpal_activity_completed` |
| 触发条件 | 信件成功发送，且 `Letter` 表或等价信件记录插入成功 |
| 生产方 | 笔友模块 |
| 消费方 | 学习记录模块 |
| 载荷 | `{ userId, activityType: "letter_sent", timestamp, relationId, letterId }` |
| 业务语义 | 学习记录模块监听该事件，用于更新 `DailySummary.writingCompletedCount` 和模块完成数量；草稿保存、安全检查失败、发送失败均不触发该事件 |
| 验收口径 | 成功发送 1 封信后，当日写作完成数量增加 1；重复刷新发送成功页不重复计数 |

## 10. 验收标准

### 10.1 模块级验收

| 模块 | MVP 验收标准 |
|---|---|
| 首页导航 | 用户能从首页进入单词、阅读、笔友写信三个核心模块；加载/空/错状态有提示 |
| 单词模块 | 用户能完成 4 选 1 释义选择题主动回忆练习；系统给出即时反馈并生成或更新复习计划；中断后已完成结果不丢失 |
| 阅读模块 | 用户能选择文章、查看重点词、完成三类基础阅读题、查看解析和原文依据、保存阅读记录 |
| 笔友模块 | 用户能创建安全档案、看到管理员预配置测试笔友、写信、保存草稿、通过安全检查发送、查看通信历史 |
| 学习记录 | 用户能查看今日任务完成度、连续学习天数和模块完成数量，数据来自真实学习行为 |
| 安全底线 | 命中 NFR-SEC-03 V1.0 规则的联系方式、社交账号关键词和敏感词可拦截；误判不丢稿；解除关系后不能继续发信 |

### 10.2 核心 E2E 验收场景

| 场景编号 | 场景 | 通过标准 |
|---|---|---|
| E2E-01 | 完成一次单词主动回忆练习 | 反馈展示、状态保存、下次复习时间更新 |
| E2E-02 | 单词练习中断后恢复 | 已完成题目不丢失、不重复计数，未完成任务可继续或重开 |
| E2E-03 | 完成一篇阅读并答题 | 阅读记录保存，解析和原文依据可查看 |
| E2E-04 | 阅读中加入生词 | 生词进入单词模块待学习或关联记录，不直接算已掌握 |
| E2E-05 | 完成一次笔友写信 | 草稿保存、安全检查、发送成功、通信历史可见 |
| E2E-06 | 笔友安全拦截 | 命中 NFR-SEC-03 V1.0 规则的联系方式、社交账号关键词或敏感词被拦截，原文保留可修改 |
| E2E-07 | 解除关系后发信 | 对方无法继续发送信件 |
| E2E-08 | 查看学习记录 | 今日完成度、连续天数、模块数量随真实行为更新 |

### 10.3 不通过标准

出现以下情况之一，不建议进入上线验收：

- 核心学习流程只能浏览，不能产生可保存的学习结果。
- 单词练习无法生成或更新复习计划。
- 阅读题只有对错，没有解析或原文依据。
- 信件发送失败或审核误判会导致草稿丢失。
- 解除笔友关系后仍可继续发信。
- 学习记录使用固定假数据，不能反映真实学习行为。
- 关键错误无提示，用户误以为保存或发送成功。

## 12. 约束条件

### 12.1 产品约束

- 平台定位为免费在线英语学习平台，不设计付费课程、会员、电商和商业转化。
- 本期以 Web 端为主，不建设移动 App。
- 本期优先学习任务，不把社交、排行榜、娱乐化装饰作为核心卖点。
- 笔友模块只做受控一对一写信闭环，不做开放广场或实时聊天。

### 12.2 技术约束

- 模块之间应通过明确接口或事件通信，避免直接依赖内部实现。
- 数据结构应预留扩展字段，但前端不必展示所有后台统计项。
- 外部服务不可作为核心流程唯一依赖；AI、音频、审核外部接口必须有降级。
- 核心流程需支持基础浏览器兼容和响应式布局。

### 12.3 内容与合规约束

- 阅读材料、音频、例句和图片需记录来源和授权信息。
- 用户学习行为、写作草稿和通信内容默认私有。
- 如涉及未成年人使用，需单独确认监护人同意、匹配范围和通信限制。
- 如使用第三方 AI 或审核服务处理信件内容，应明确告知数据用途并提供关闭或跳过选择。

## 13. 风险与缓解措施

## 14. 范围控制

### 14.1 本期范围

本期必须完成以下内容：

1. **首页与基础导航**：三个核心入口、基础响应式、加载/空/错状态。
2. **单词记忆 MVP**：单词列表、详情、4 选 1 释义选择题主动回忆、即时反馈、基础复习排程、队列优先级、每日上限、中断保存。
3. **阅读练习 MVP**：阅读列表、冷启动选材、文章详情、重点词辅助、三类基础题型、解析与原文依据、阅读记录、断点续读。
4. **受控笔友写信 MVP**：安全档案、管理员预配置测试关系、收件箱、编辑器、草稿保存、安全审核、通信历史、举报、屏蔽、解除关系。
5. **基础学习记录**：今日任务完成度、连续学习天数、模块完成数量。
6. **非功能底线**：关键数据保存、错误反馈、安全隐私、核心 E2E 测试。

### 14.2 非本期范围

以下内容不纳入本期：

- 付费课程、商城、会员、订单和支付。
- 开放广场式社区、群组、动态流、实时聊天。
- 大规模笔友公开搜索和复杂匹配推荐。
- 复杂个性化推荐、能力雷达、周报月报、重型数据看板。
- 完整管理后台、内容健康度仪表盘、复杂人工审核工作台。
- 多端实时同步、离线完整学习包、移动 App。
- 发音评测、跟读打分、听读同步、复杂策略模式。
- AI 自动生成完整文章、自动代写信件或替用户确认真实经历。

### 14.3 可延后范围

以下内容有价值，但可延后：

| 功能 | 延后原因 | 后续进入条件 |
|---|---|---|
| 高亮笔记与写作素材 | 不影响阅读基本闭环 | 阅读完成率稳定，素材复用需求明确 |
| 阅读策略模式 | 交互成本较高 | 基础阅读题和解析稳定后再做 |
| 主题公开信 | 涉及内容审核和社区运营 | 一对一匹配冷启动困难，并有审核能力 |
| 同伴互助反馈 | 容易变成互相批改作文 | 笔友通信稳定后再验证低压力反馈 |
| AI 语言检查 | 隐私和稳定性依赖强 | 明确服务、告知和降级方案 |
| 本周趋势和行动建议 | 需要足够行为数据 | 基础记录真实可靠后再开放 |
| 管理后台 | 开发和权限成本高 | 内容量、举报量或运营需求达到阈值 |

### 14.4 明确不做

| 内容 | 不做原因 |
|---|---|
| 付费课程和电商 | 偏离免费学习平台定位 |
| 大型开放社区 | 安全、审核、运营成本过高 |
| 实时聊天 | 增加社交风险和实时系统复杂度 |
| 排行榜作为核心卖点 | 容易造成焦虑，偏离学习任务导向 |
| 自动代写整封信 | 不利于写作训练，也有学术和真实性风险 |
| 虚构学习数据或虚假用户互动 | 违反真实性和验收要求 |

### 14.5 需求变更控制原则

新增需求进入本期前，必须满足以下条件：

1. 直接服务单词、阅读、写作或学习反馈核心闭环。
2. 不破坏 MVP 排期和已确定安全底线。
3. 有明确业务价值、使用场景和验收口径。
4. 有清楚的数据来源和依赖关系。
5. 能说明为什么不能延后。
6. 不引入复杂社区、支付、重型后台或难以降级的外部依赖。
7. 不改变第 9.0 节跨模块接口契约中的核心业务语义；如确需调整，应同步更新本文档、接口设计文档、相关模块实现和 E2E 验收场景。

变更流程建议：

```text
提出变更 -> 说明业务价值 -> 标注影响模块和依赖 -> 判断优先级 -> 评估排期影响 -> 决定纳入本期/延后/拒绝
```

### 14.6 防止需求蔓延的判断标准

如果一个需求符合以下任一情况，默认不进入本期：

- 不能直接帮助用户完成英语学习任务。
- 主要提升“看起来丰富”，但不影响核心闭环成立。
- 需要新增复杂后台、复杂审核、复杂推荐或实时通信能力。
- 需要尚未确定的外部服务作为唯一依赖。
- 会让页面层级明显变深或打断学习过程。
- 缺少可验证的验收标准。
- 只是后续运营想象，尚无用户行为数据支撑。
- 凡是建议在本期实现的功能，必须能对应到本文档中的一个 P0 级需求编号及其验收口径；找不到对应 P0 需求时，默认拒绝或延后。

## 16. 范围控制结论

### 16.1 必须做

本期必须完成核心学习闭环和安全底线：

- 首页三入口和基础导航。
- 单词 4 选 1 释义选择题主动回忆、即时反馈、基础排程、队列优先级和中断保存。
- 阅读选材、重点词、三类基础题、解析依据、阅读记录和断点续读。
- 受控笔友写信闭环，包括安全档案、管理员预配置测试关系、收件箱、编辑器、草稿、安全审核、通信历史、举报、屏蔽和解除关系。
- 基础学习记录三项前端信号：今日任务完成度、连续学习天数、模块完成数量。
- 关键数据保存、错误反馈、隐私保护和核心 E2E 测试。

### 16.2 可以延后

以下内容可以延后，不应阻断 MVP：

- 高亮笔记、写作素材生成、听读同步、阅读策略模式。
- AI 润色、AI 阅读生成、复杂内容审核后台。
- 本周趋势、历史复盘、行动建议、能力雷达、学习周报。
- 主题公开信、同伴互助反馈、点赞感谢统计。
- 多端同步、离线学习、移动 App。

### 16.3 明确不做

本期明确不做：

- 付费、电商、会员体系。
- 开放广场式大型社区、群聊、动态流、实时聊天。
- 复杂推荐算法和重型运营后台。
- 自动代写整封信或虚构学习/通信数据。

### 16.4 后续扩展条件

后续扩展必须满足以下条件之一：

1. MVP 核心流程已经稳定，E2E 测试通过。
2. 有真实用户行为或测试反馈证明该功能能提升学习效果或安全性。
3. 相关内容来源、隐私告知、审核机制和数据口径已经明确。
4. 功能可以独立上线和回滚，不影响现有核心闭环。
5. 有明确验收标准和维护责任人。

综上，第二阶段的范围控制结论是：**本期只做能支撑英语学习闭环和安全上线底线的功能；所有偏社区化、算法化、运营后台化、AI 深度化和复杂数据可视化的内容均延后或排除。**
