## 2. 为什么要做环境隔离

### 2.1 本项目为什么需要环境隔离

既有项目说明曾将 EnglishLover Web 端描述为位于 `front/` 目录下的 React + Vite + TypeScript 应用，并配有 Express 服务端。当前仓库快照未核验到这些源码文件，因此下面内容应作为后续恢复或创建源码后的部署学习口径，而不是当前已可运行事实：

- 开发命令：`npm run dev`（需以真实 `package.json` 为准）
- 构建命令：`npm run build`（需以真实 `package.json` 为准）
- 生产启动命令：`npm run start`（需以真实 `package.json` 为准）
- 服务端入口：`front/server.ts`（待恢复或创建后确认）
- 生产产物：`dist/server.cjs`（待构建配置确认）
- 关键环境变量：`GEMINI_API_KEY`（如后续仍接入 Gemini 能力）

如果不用环境隔离，后期部署经常会遇到这些问题：

- 你本机 Node 版本和服务器不同
- `npm install` 出来的依赖和另一台机器不一致
- 项目迁移到新服务器要重新配置环境
- 多个项目同时部署时互相污染
- 本地能跑，服务器不能跑，却不知道差异出在哪

所以“环境隔离”的本质，不是为了追求新技术，而是为了回答一个工程问题：

> **怎样让同一份项目在不同机器上，以尽量一致的方式构建、运行和更新。**

### 2.2 环境隔离解决什么，不解决什么

Docker 主要解决：

- 运行环境一致性
- 依赖打包与迁移
- 部署流程可重复
- 多项目隔离
- 服务启动方式标准化

Docker 不直接解决：

- 业务逻辑错误
- 云服务器安全策略配置错误
- 域名解析错误
- HTTPS 证书申请失败
- 应用本身性能瓶颈

也就是说，Docker 是**部署基础设施层**的一部分，不是万能修复工具。

### 2.3 为什么不只用 PM2

PM2 主要是 Node 进程管理工具，适合做：

- 进程守护
- 崩溃重启
- 日志管理
- 后台运行

但它本身不解决“环境可复制”问题。它更像是在**已有环境上管理进程**，而 Docker 更像是在**先把环境封装出来，再运行进程**。

### 2.4 为什么现在不先学 Kubernetes

Kubernetes 很重要，但它解决的是“很多容器、很多机器、很多服务”的调度问题。

如果你现在连这些问题还没吃透：

- 镜像是什么
- 容器是什么
- 端口映射怎么工作
- 环境变量怎么注入
- 反向代理为什么存在

那么直接学 Kubernetes 很容易只记住一堆术语，却没有真正理解。

所以这份手册选择：

**Docker → Compose → 服务器部署 → 反向代理 → 再为 Kubernetes 打基础**

## 3. 先理解底层：Docker 到底是什么

### 3.1 不要把 Docker 当成“高级虚拟机”

Docker 容器和虚拟机都能实现隔离，但实现思路不同。

- **虚拟机**：模拟一整台计算机，带完整操作系统
- **容器**：共享宿主机内核，只隔离进程、文件系统、网络等运行视图

所以你要记住一句非常重要的话：

> **容器本质上更接近“被隔离起来运行的一组进程”，不是一台独立电脑。**

这也是为什么容器通常更轻、更快、更适合应用部署。

### 3.2 你必须分清的 8 个核心对象

#### 1）Image（镜像）

镜像是一个**静态的运行模板**，里面包含：

- 基础环境
- 系统依赖
- Node 运行时
- 项目依赖
- 项目代码
- 默认启动命令

镜像不是进程，它只是“运行说明书 + 文件快照”。

#### 2）Container（容器）

容器是镜像运行后的实例。

- 镜像像“模板”
- 容器像“运行中的对象”

同一个镜像可以启动多个容器。

#### 3）Dockerfile

Dockerfile 是“如何制作镜像”的配方文件。

它描述：

- 从哪个基础镜像开始
- 拷贝什么文件
- 执行什么安装命令
- 暴露什么端口
- 容器启动时跑什么命令

#### 4）Layer（层）

Docker 构建镜像时不是一次性打包，而是按步骤形成层。

这样做的设计意图是：

- 便于缓存
- 便于复用
- 减少重复构建成本

这也是为什么 Dockerfile 的顺序很重要。

#### 5）Build Context（构建上下文）

当你执行 `docker build` 时，Docker 会拿到一个“构建上下文”，通常是当前目录。

这个目录里的内容，Dockerfile 才有机会 `COPY` 进去。

所以：

- 构建上下文太大，会导致构建变慢
- 不该进镜像的东西，要用 `.dockerignore` 排除

#### 6）Volume（卷）

卷用于保存容器外部持久化数据。

本项目当前没有数据库，但如果以后加入：

- PostgreSQL
- Redis 持久化
- 上传文件目录

就会明显需要卷。

#### 7）Network（网络）

容器之间不是天然互通到公网，它们通过 Docker 网络通信。

网络层解决：

- 容器与容器的通信
- 宿主机与容器的端口映射
- 服务之间如何互相发现

#### 8）Entrypoint / CMD / 主进程

容器启动时会跑一个主进程。主进程退出，容器通常就停止。

这就是为什么你必须理解：

- 容器不是“后台系统环境”
- 容器活着，通常只是因为里面的主进程还在运行

## 4. Linux 视角：为什么说容器本质上是被隔离的进程

在 Linux 世界里，一个 Web 服务本质就是一个进程监听某个端口。

例如当前项目在开发时：

- `npm run dev` 实际会执行 `tsx server.ts`
- `server.ts` 启动 Express 应用
- Express 监听端口（当前代码写死为 `3000`）

从操作系统视角看，本质上就是：

- 一个 Node 进程
- 占用一个端口
- 读取环境变量
- 访问文件系统
- 输出日志到终端

Docker 只是把这个进程放到了一个隔离出来的运行环境里，并给它单独的：

- 文件系统视图
- 网络视图
- 进程视图
- 环境变量空间

理解这一点后，你会更容易明白为什么排障时经常要看：

- 进程是否启动
- 端口是否监听
- 环境变量是否存在
- 日志里有没有报错

## 5. 先认识本项目后续源码的运行方式

这一步非常关键。不要在没看懂项目的情况下照搬 Docker 模板。

### 5.1 待复核运行命令

既有说明中的 `front/package.json` 曾包含以下命令；当前仓库未核验到该文件，恢复源码后必须以真实 `package.json` 为准：

- `npm run dev`：`tsx server.ts`
- `npm run build`：`vite build && esbuild server.ts ... --outfile=dist/server.cjs`
- `npm run start`：`node dist/server.cjs`
- `npm run lint`：`tsc --noEmit`

### 5.2 开发模式和生产模式的区别

#### 开发模式

开发时，`server.ts` 会使用 Vite 开发能力，强调热更新、开发调试。

#### 生产模式

生产时，应该先执行：

```bash
npm run build
```

然后运行：

```bash
npm run start
```

这说明容器化不能简单照抄开发命令，而是要区分：

- 开发容器怎么跑
- 生产容器怎么跑

本手册的主线以**生产部署容器**为重点。

### 5.3 当前环境变量

根据 `front/.env.example`，至少有：

- `GEMINI_API_KEY`
- `APP_URL`

其中最关键的是 `GEMINI_API_KEY`。它不应该被写死在镜像中，而应该在运行容器时注入。

这背后的设计理念叫：

> **配置与代码分离**

也就是：

- 镜像放代码和运行环境
- 环境变量放配置
- 同一个镜像可以跑在开发、测试、生产环境

## 6. 单容器部署：先把项目装进 Docker

这一章是整份手册最重要的“第一闭环”。

目标不是追求一步到位，而是让你先真正理解：

- 镜像怎么做出来
- 容器怎么跑起来
- 端口怎么打通
- 环境变量怎么传进去
- 日志怎么查看

### 6.1 为什么先做单容器

因为如果一开始就上 Compose、Nginx、HTTPS、云服务器，很容易把问题混在一起。

单容器阶段只解决：

- 这个 Node 应用能不能在容器里独立跑起来

一旦这个闭环通了，再叠加 Compose 和服务器部署会稳很多。

### 6.2 生产容器的推荐思路

建议使用**多阶段构建**。

核心思想：

- 第一阶段：安装依赖、构建项目
- 第二阶段：只保留运行所需内容

这样做的原因：

1. 降低最终镜像体积
2. 减少无关构建工具残留
3. 分离“构建环境”和“运行环境”
4. 提高部署镜像的整洁度与安全性

### 6.3 Dockerfile 应该表达什么

后续你真正编写 Dockerfile 时，它应该表达这些意思：

1. 选定明确的 Node 基础镜像版本
2. 设置工作目录
3. 先复制 `package.json` / `package-lock.json`
4. 执行 `npm ci`
5. 复制项目源代码
6. 执行 `npm run build`
7. 在运行阶段只保留生产运行需要的内容
8. 容器启动时执行 `npm run start` 或等价命令

### 6.4 为什么要先复制 `package*.json`

这是 Docker 层缓存思想最典型的实践。

如果你先复制全部代码再安装依赖，只要代码有一点变化，依赖安装层就会失去缓存。

而如果先复制：

- `package.json`
- `package-lock.json`

那么只要依赖没变，`npm ci` 这一步就可以复用缓存。

这体现的是一个非常值得举一反三的工程思想：

> **把变化少的步骤放前面，把变化多的步骤放后面。**

### 6.5 为什么部署时优先 `npm ci`

`npm ci` 的特点：

- 严格按锁文件安装
- 更适合 CI / 部署
- 更容易复现同样依赖树

`npm install` 更适合开发时调整依赖。

### 6.6 为什么必须写 `.dockerignore`

`.dockerignore` 的作用和 `.gitignore` 类似，但针对的是 Docker 构建上下文。

至少应避免把这些东西送进构建上下文：

- `node_modules`
- `dist`
- `.git`
- 本地缓存
- 本地私密环境文件

否则可能带来：

- 构建变慢
- 镜像变大
- 把本地脏文件带进镜像
- 把不该进镜像的配置或密钥带进去

### 6.7 单容器最小实操路径

等你后续正式编写配置文件时，建议按这个顺序验证：

#### 步骤 1：先在宿主机确认项目本身可构建

在 `front/` 下先执行：

```bash
npm run lint
npm run build
```

如果原生环境都构建失败，就不要急着怪 Docker。

#### 步骤 2：构建镜像

示意命令：

```bash
docker build -t englishlover-app ./front
```

这里 `./front` 是构建上下文，意味着 Dockerfile 及其所需项目文件应以 `front/` 为中心组织。

#### 步骤 3：运行容器

示意命令：

```bash
docker run --name englishlover-app -p 3000:3000 --env-file .env englishlover-app
```

这条命令表达了三个关键概念：

- `--name`：容器名字
- `-p 3000:3000`：宿主机端口映射到容器端口
- `--env-file`：把环境变量注入运行中的容器

#### 步骤 4：验证服务是否可访问

验证方式：

- 浏览器访问对应端口
- 或用 `curl` 请求健康接口（当前项目有 `/api/health`）

#### 步骤 5：查看日志

```bash
docker logs englishlover-app
```

学会看日志，比学会背命令更重要。

#### 步骤 6：进入容器内部排查

```bash
docker exec -it englishlover-app sh
```

进入容器后，你可以检查：

- 环境变量是否存在
- 文件是否按预期复制
- 进程是否正确启动

## 7. 端口、监听地址与访问失败的本质

这是初学 Docker 时最容易糊涂的一章。

### 7.1 端口映射到底做了什么

例如：

```bash
docker run -p 3000:3000 ...
```

含义是：

- 宿主机的 `3000` 端口
- 转发到容器内部的 `3000` 端口

注意：

- 左边是**宿主机端口**
- 右边是**容器内部服务监听端口**

### 7.2 为什么代码里监听地址也重要

有些应用只监听 `127.0.0.1`，这样即使做了端口映射，外部仍可能访问失败。

更稳妥的生产实践通常是监听：

```text
0.0.0.0
```

因为这意味着接受来自容器网络接口的连接。

既有说明曾提到 `front/server.ts` 使用 `3000` 端口；如果后续正式容器化，最好同时规划：

- 端口支持环境变量覆盖
- 监听地址适合容器运行

这属于后续代码适配工作，不代表当前仓库已经实现。

### 7.3 为什么“容器启动了但打不开页面”

常见根因通常只有几类：

- 应用实际没启动成功
- 监听地址不对
- 容器内部端口与映射端口不一致
- 服务器防火墙没放行
- 反向代理没转发对

学排障时，要把问题拆成层级：

1. 进程有没有起来
2. 容器内端口有没有监听
3. 宿主机映射有没有生效
4. 外部请求有没有到宿主机

## 8. Compose：为什么它不是“多写几条 docker run”

### 8.1 Compose 要解决什么问题

当系统不止一个容器时，手工 `docker run` 会越来越混乱。

例如以后你可能会有：

- 应用容器
- Nginx / Caddy 容器
- 数据库容器
- Redis 容器

这时候你需要的不只是“启动几个容器”，而是要描述：

- 谁和谁一起启动
- 谁依赖谁
- 谁暴露端口到外部
- 谁只在内部网络访问
- 配置怎么统一管理

这就是 Compose 的价值。

### 8.2 Compose 的本质

`docker-compose.yml` 本质上是在写一份**应用拓扑说明书**。

它不是单纯为了省命令，而是在描述：

- 系统有哪些服务
- 服务之间的关系
- 服务的运行参数
- 网络与卷如何分配

这是一种非常重要的工程思想：

> **用声明式配置描述系统，而不是靠人工临场操作记忆。**

### 8.3 本项目的 Compose 学习路径

建议分两版理解。

#### 版本 A：只有应用服务

先只把当前项目应用写入 Compose。

重点学习：

- `services`
- `build`
- `ports`
- `environment` / `env_file`
- `restart`

#### 版本 B：加入反向代理

再引入 Nginx 或 Caddy。

重点学习：

- 多服务网络
- 只暴露代理层到公网
- 应用服务只在内部网络被代理访问

这样你的系统边界会更清晰。

### 8.4 Compose 中服务名为什么能互相访问

Compose 会创建默认网络，同一网络内的服务通常可以通过**服务名**互相访问。

例如代理服务可以把上游指向：

- `app:3000`
- 或你在 Compose 里定义的应用服务名

这背后本质是 Docker 网络中的服务发现能力。

## 9. 为什么上线时通常还要反向代理（Nginx / Caddy）

### 9.1 为什么不直接把 Node 应用裸露到公网

直接暴露应用当然能跑，但工程上通常不是最优选择。

反向代理层的职责包括：

- 域名接入
- HTTPS 终止
- 静态资源缓存
- 请求转发
- 压缩
- 安全控制
- 统一入口

### 9.2 职责分离思想

这是理解部署架构的关键：

- **应用容器**：负责业务逻辑
- **代理容器**：负责对外接入

这样做的好处：

- 应用不用直接处理所有外部访问细节
- 切换代理策略更灵活
- 以后替换应用服务更容易
- 更接近真实生产环境

### 9.3 Nginx 和 Caddy 怎么选

学习阶段：

- 想更贴近传统生产环境：选 **Nginx**
- 想更快完成 HTTPS：选 **Caddy**

如果你的目标是“学到底层”，建议优先学 Nginx，因为它更能帮助你理解：

- 反向代理
- location / upstream
- 端口转发
- 静态资源与动态请求分工

## 10. 本项目推荐的部署演进路线

### 阶段 1：本地原生运行

目标：确认项目本身可用。

你需要会：

- 安装依赖
- 运行 `npm run dev`
- 运行 `npm run build`
- 运行 `npm run start`

### 阶段 2：本地单容器部署

目标：理解镜像、容器、端口映射、环境变量注入。

你需要会：

- 写 Dockerfile
- 写 `.dockerignore`
- 构建镜像
- 运行容器
- 看日志
- 进入容器调试

### 阶段 3：本地 Compose 部署

目标：用 `docker compose` 管理服务。

你需要会：

- 定义应用服务
- 用 `env_file` 或 `environment` 注入配置
- 统一启停
- 读 Compose 日志

### 阶段 4：加入代理层

目标：理解真实线上访问链路。

你需要会：

目标：完成真实部署闭环。

## 11. Linux 服务器上线实战思路

这一章重点讲“上线链路应该怎么理解”。

### 11.1 服务器需要准备什么

最低要求通常包括：

- 一台 Linux 云服务器
- 已安装 Docker
- 已具备 `docker compose` 功能
- 已配置开放所需端口
- 已有项目代码或镜像获取方式

### 11.2 两种常见部署路径

#### 路径 A：服务器上直接构建

流程：

1. 把项目拉到服务器
2. 在服务器执行 `docker compose build`
3. 再执行 `docker compose up -d`

优点：

- 简单
- 适合个人项目 / 小项目

缺点：

- 服务器上也要承担构建工作
- 构建速度受服务器性能影响

#### 路径 B：本地或 CI 构建镜像，再推送到镜像仓库

流程：

1. 本地或 CI 构建镜像
2. 推送到镜像仓库
3. 服务器拉取镜像
4. 用 Compose 启动

优点：

- 更接近标准化交付
- 部署与构建解耦

缺点：

- 学习成本更高
- 需要镜像仓库

对当前项目，建议你先从**路径 A** 学起。

### 11.3 目录规划建议

服务器上建议有明确的部署目录，例如：

```text
/opt/englishlover/
```

目录中集中管理：

- Compose 文件
- 代理配置
- 环境变量文件
- 版本更新说明

这样后期维护更清晰。

### 11.4 更新流程应该怎么理解

一个规范的更新流程通常不是“登录服务器手改”，而是：

1. 修改代码
2. 提交代码
3. 构建镜像
4. 替换运行中的容器
5. 验证日志与访问状态

这体现的是不可变基础设施思想：

> **尽量通过替换实例发布新版本，而不是直接在运行中的环境里手工修补。**

## 12. 为什么说 Dockerfile 和 Compose 都是“声明式配置”

声明式配置的意思不是“按步骤描述我怎么敲命令”，而是“描述我想要什么状态”。

### 12.1 Dockerfile 的声明性

Dockerfile 不是在记录你手工配置服务器的全过程，而是在定义：

- 我要什么基础环境
- 我要什么依赖
- 我要什么构建步骤
- 我要什么默认启动方式

### 12.2 Compose 的声明性

Compose 不是在说“先启动 A，再启动 B，再连起来”，而是在说：

- 系统中有这些服务
- 这些服务这样联网
- 这些端口暴露出去
- 这些配置注入进去

声明式的好处是：

- 可重复
- 可读
- 可审查
- 可迁移
- 更容易自动化

这也是为什么后续你学 Kubernetes 时会觉得很多思想并不陌生。

## 13. 最重要的设计理念：真正值得学会的 6 件事

### 13.1 配置与代码分离

不要把密钥、环境差异写死进代码或镜像。

应用代码应该尽量稳定；环境差异通过环境变量注入。

### 13.2 构建环境与运行环境分离

多阶段构建不是炫技，而是把：

- 编译所需环境
- 生产运行环境

分开处理。

### 13.3 不可变交付物

镜像一旦构建完成，就应该尽量作为稳定的交付物使用。

### 13.4 单一职责

应用服务管业务，代理服务管接入，数据库服务管数据。

### 13.5 层缓存与增量构建

Docker 层缓存不是单纯为了快，而是在鼓励你用稳定结构组织构建过程。

### 13.6 基础设施也要“像代码一样被管理”

部署配置不该只存在脑子里或聊天记录里，而应该进入文档和配置文件。

## 15. 常见报错与排障方法

下面这部分非常重要。真正会部署的人，不是不会出错，而是知道按层排查。

### 15.1 构建失败

#### 常见现象

- `npm ci` 失败
- `npm run build` 失败
- 找不到文件
- TypeScript 报错

#### 排查顺序

1. 先在宿主机执行 `npm run lint`、`npm run build`
2. 再确认 Dockerfile 的复制顺序是否正确
3. 检查构建上下文是否选对
4. 检查 `.dockerignore` 是否误排除了必要文件

### 15.2 容器启动即退出

#### 常见根因

- 主进程执行失败
- 启动命令写错
- 构建产物不存在
- 运行依赖没装全

#### 核心思路

容器退出通常不是“Docker 死了”，而是容器里的主进程结束了。

先看：

```bash
docker logs <container-name>
```

### 15.3 能启动但访问不到

#### 排查顺序

1. 容器里应用是否真的在监听端口
2. 端口映射是否正确
3. 应用是否监听 `0.0.0.0`
4. 宿主机防火墙是否放行
5. 代理配置是否正确

### 15.4 环境变量未生效

#### 排查思路

1. `.env` 文件路径是否传对
2. Compose 的 `env_file` 是否生效
3. 变量名是否和代码读取一致
4. 变量是否在容器内部实际存在

可以进入容器后检查：

```bash
printenv
```

### 15.5 代理层报 502 / 504

这类问题通常说明：

- 代理能收到请求
- 但代理连不到上游应用

要重点排查：

- 上游服务名是否正确
- 上游服务端口是否正确
- 两个服务是否在同一 Docker 网络
- 应用容器是否真的启动成功

## 16. 学习时最容易踩的误区

### 误区 1：把 Docker 学成命令背诵

真正重要的是对象关系：

- Dockerfile → 生成 image
- image → 运行成 container
- Compose → 管理多个 service

### 误区 2：把容器当虚拟机

容器不是让你长期 SSH 进去手工改环境的地方。

### 误区 3：本地能跑就以为上线没问题

上线还要考虑：

- 端口开放
- 域名解析
- 反向代理
- HTTPS
- 重启策略
- 日志查看

### 误区 4：看到报错就从最复杂层开始怀疑

正确顺序永远是：

- 先看应用本身
- 再看容器
- 再看端口
- 再看代理
- 最后看公网访问

## 附录 A：术语表

- **Image**：镜像，静态运行模板
- **Container**：容器，镜像的运行实例
- **Layer**：镜像层，支持缓存与复用
- **Build Context**：构建上下文，`docker build` 时发送给 Docker 的目录内容
- **Volume**：卷，用于持久化数据
- **Bind Mount**：宿主机目录直接挂入容器
- **Network**：容器网络
- **Port Mapping**：宿主机端口到容器端口的映射
- **Entrypoint / CMD**：容器默认启动命令
- **Reverse Proxy**：反向代理，对外接收请求并转发给上游服务
