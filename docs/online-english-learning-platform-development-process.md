# 在线英语学习平台 Web 端完整开发流程规划

> 适用对象：免费在线英语学习平台 Web 端  
> 核心功能范围：单词记忆、阅读练习、笔友写信练习  
> 文档用途：需求分析、系统设计、开发实施、测试部署、后期迭代维护参考  
> 重要说明：本文是开发流程与规划建议，不等同于当前项目已全部实现的功能说明；实际论文或项目验收中，应以真实代码、真实测试结果和实际部署情况为准。

---

## 1. 结论概述

一个符合高质量用户体验、有效学习成果、技术可扩展性和稳定性的在线英语学习平台，应以“学习目标驱动、核心功能优先、模块化架构支撑、自动化流程保障”为主线进行开发。

本网站完全免费，因此不需要围绕付费课程、商城或商业转化展开设计，而应重点服务于用户的英语学习过程。平台核心功能可以围绕三个学习场景展开：

1. **单词记忆**：帮助用户积累词汇量，并通过复习机制巩固记忆。
2. **阅读练习**：帮助用户在语境中理解单词、句子和文章内容。
3. **笔友写信练习**：帮助用户进行英文输出训练，提高表达能力。

在开发流程上，应避免一开始就追求功能数量，而是先完成一个稳定可用的最小可用版本，再通过迭代方式逐步完善学习记录、反馈机制、数据统计、内容管理、自动化测试和部署流程。这样既能控制项目范围，又能保证项目从需求分析到后期维护的全生命周期保持清晰、高效和可控。

## 2. 项目开发总体原则

### 2.1 学习目标优先

英语学习平台的核心价值不是页面数量多，也不是功能看起来复杂，而是帮助用户完成真实学习任务。因此，项目设计应始终围绕以下问题展开：

- 用户能否快速开始学习？
- 用户能否知道自己今天应该学什么？
- 用户能否获得及时反馈？
- 用户能否看到自己的学习进展？
- 用户是否愿意持续使用？

如果某项功能不能直接服务于学习过程，或者会明显增加开发复杂度，就不应在早期版本中优先实现。

### 2.2 核心功能优先

首个版本应聚焦单词记忆、阅读练习和笔友写信练习三个模块，先保证主要学习流程可用。排行榜、社区互动、复杂推荐算法、实时聊天、付费课程等功能可以作为后续扩展方向，不应影响核心学习功能的稳定开发。

### 2.3 模块化设计

平台应按照业务能力进行模块拆分，例如单词模块、阅读模块、写作模块、学习记录模块、公共组件模块和接口服务模块。模块之间保持清晰边界，可以减少代码耦合，方便后期维护和功能扩展。

### 2.4 组件复用

Web 前端中大量界面元素具有相似结构，例如按钮、卡片、标签、弹窗、进度条、表单输入框和空状态提示等。通过建立统一组件库，可以减少重复开发，保持界面风格一致，并提高后续新增页面的效率。

### 2.5 自动化保障

为了减少人工重复操作，应尽早引入标准化工具和流程，包括代码格式化、类型检查、自动化测试、自动构建和 CI/CD 部署。这样可以在每次修改后及时发现问题，降低后期维护风险。

## 3. 第一阶段：项目立项与需求分析

### 3.1 明确平台定位

项目启动时，应首先明确平台定位：

```text
一个完全免费的在线英语学习平台，面向有英语学习需求的用户，
通过单词记忆、阅读练习和笔友写信练习，帮助用户完成词汇积累、
语言输入和语言输出训练。
```

该定位决定了项目不应过度偏向商业化，也不应在早期引入过多与学习目标无关的功能。

### 3.2 明确目标用户

平台可以面向以下几类用户：

| 用户类型 | 学习目标 | 主要需求 |
|---|---|---|
| 英语基础学习者 | 积累基础词汇、建立学习习惯 | 简单易用的单词学习和复习功能 |
| 阅读能力提升者 | 通过文章理解词汇和句子 | 分级阅读材料、重点词提示、阅读题目 |
| 写作练习用户 | 提升英文表达能力 | 写信主题、写作提示、历史记录 |
| 自主学习用户 | 制定每日学习计划 | 学习记录、进度反馈、持续学习提醒 |

### 3.3 拆分学习场景

平台的学习流程可以抽象为：

```text
选择学习任务 -> 完成练习 -> 获得反馈 -> 保存记录 -> 下次继续学习
```

对应到核心模块：

```text
词汇输入：单词记忆
语境输入：阅读练习
语言输出：笔友写信练习
学习反馈：学习记录与进度统计
```

### 3.4 确定 MVP 范围

为了避免项目失控，首个版本应采用 MVP 思路。建议 MVP 包含：

- 首页展示平台定位和三个学习入口；
- 单词模块支持新词学习、主动回忆练习、即时反馈、自动复习排程、最大间隔兜底、复习队列优先级、题型难度自适应、列表检索筛选和中断保存；
- 阅读模块支持冷启动选材、主题偏好、分级文章、重点词辅助、辅助状态恢复、阅读高亮、阅读策略模式、多题型练习、解析反馈、内容纠错、有效阅读记录和泛读/精读转换；冷启动无任何用户信号时，首篇默认采用中低难度、通用兴趣主题和短篇材料；所有搜索、筛选、生词本、高亮笔记等可能为空的区域都应提供友好空状态引导；长期数据架构应预留学习档案导出、CEFR/Lexile 等能力标准映射字段，以及基于能力雷达生成“能力叙事”的空间；
- 写信模块支持主题展示、信件编辑和写作记录保存；
- 学习记录模块支持展示已学习单词数、阅读完成数和写作次数；
- 页面具备基础响应式布局；
- 核心流程能够稳定运行。

暂不建议在 MVP 中实现：

- 复杂社交系统；
- 付费课程或商城；
- 实时聊天；
- 大规模社区；
- 复杂个性化推荐；
- 过度复杂的数据看板；
- 与核心学习目标关系较弱的装饰性功能。

## 4. 第二阶段：需求规格与范围控制

### 4.1 功能需求

#### 4.1.1 单词记忆模块

单词记忆模块是平台的基础模块，目标是帮助用户完成词汇积累和持续复习。该模块不应只停留在“查看单词详情、手动标记掌握状态”的信息展示层面，而应引入主动回忆和间隔重复思想，形成“学习—测验—反馈—排程—复习—诊断”的闭环。

主要功能包括：

- 单词列表展示，并区分新词、待复习词和困难词；
- 单词释义、音标、例句、发音入口和记忆提示展示；
- 主动回忆练习，例如卡片翻转、选择题、拼写题或听写；
- 根据答题结果自动生成下次复习时间和待复习队列；
- 记录掌握度、困难度、复习次数、正确/错误次数、最近复习时间、反应时间等细粒度数据；
- 用户作答后提供即时反馈，答错时展示正确答案和记忆提示；
- 错词自动进入困难词或高优先级复习队列；
- 展示今日目标、待复习数、完成度、困难词数量等诊断信息；
- 预留学习行为日志、自定义词库、多端同步和离线学习扩展空间；
- 明确复习队列优先级，避免错词过多挤占正常到期复习；
- 设置错词强化保护机制，连续错误时降低题型难度或暂停强化；
- 预留词库元数据，如词性、难度、话题标签、来源和数据质量状态；
- 支持每日新词上限、每日复习上限等学习负荷配置；
- 明确发音和例句朗读的失败降级策略；
- 保存练习会话状态，采用逐题实时提交，支持中途退出后的续接或已完成结果保留；
- 预留 `easinessFactor`、`stability`、`confusableWith` 等字段，支持后续升级间隔重复算法和易混淆词诊断；
- 练习交互支持基础键盘操作和必要无障碍语义；
- 设置最大复习间隔和低频抽样复查，避免高掌握词形成假性掌握；
- 单词列表支持基础搜索、筛选和排序，便于大词库管理；
- 听力练习后续可扩展最小配对辨别和语境听力选义；
- 跟读评测预留发音评分字段和录音隐私策略；
- 音频资源需记录来源和授权信息，音频不可用时不阻断文字学习；
- 拼写题和听写题应具备基础答案标准化与容错策略，避免大小写、空格、非关键标点导致误判。

如果项目周期有限，早期也应至少完成“一种主动回忆练习 + 基础答案容错 + 基础复习排程 + 最大间隔保护 + 队列优先级 + 答题反馈 + 掌握度记录 + 基础搜索筛选 + 逐题提交 + 中断保存”，不建议退回到单纯查看和手动标记模式。

#### 4.1.2 阅读练习模块

阅读练习模块主要帮助用户在真实语境中理解英语。该模块不应只是“文章 + 题目”的静态组合，而应形成“分级选材—阅读辅助—多题型训练—解析反馈—阅读记录—模块联动”的学习闭环。

主要功能包括：

- 阅读材料列表，并支持按难度、主题、标签和完成状态筛选；搜索无结果、筛选无匹配文章等空状态需给出放宽条件或继续探索的引导；
- 文章元数据展示，包括标题、主题、难度、预计阅读时间、目标词和来源；预计阅读时间先使用通用平均速度估算，后续可基于用户个人有效阅读速度动态校准展示值；
- 阅读辅助工具，包括重点词释义、音标、发音、例句、长难句提示和段落定位；
- 生词本联动，用户可将阅读中遇到的生词加入单词记忆模块；
- 阅读理解题覆盖主旨、细节、词义猜测，后续扩展推断、结构和观点态度题；
- 答题反馈包含正确答案、错误原因、简要解析和原文依据；
- 阅读记录保存总停留时长、有效阅读时长、阅读速度、正确率、生词数量、答题过程数据、自我评估及基于正确率对比的即时反馈、重读次数、最后阅读位置和阅读能力估算值；
- 支持断点续读、重读文章、泛读模式和泛读转精读；
- 支持新用户冷启动选材，诊断阅读应轻量且可跳过；若无历史数据且跳过诊断、自评和主题偏好，默认推荐中低难度（如 CEFR A2-B1 或内部等级初级）、通用兴趣主题和短篇材料，并结合主题偏好、正确率、有效阅读速度、查词行为和答题过程数据动态调整难度；阅读能力估算值应采用平滑更新，避免单次异常表现造成大幅波动，且客观数据权重应高于主观自评，自评主要作为信心指数或认知偏差诊断信号；
- 支持合并到诊断/首次阅读流程中的主题口味测试、阅读后自我评估即时反馈、句子级听读同步高亮和阅读技能雷达等后续增强能力；
- 显性训练略读、扫读、预测、总结、猜词等阅读策略，并为略读、扫读等策略提供轻量交互模式；策略锚点默认以小图标或下划线提示，hover 或点击后展开 tooltip/浮层说明；策略反馈 MVP 优先采用“比较条 + 箭头 + 文字说明”的轻量形式；
- 预留 AI 辅助内容生成字段和人工审核流程，避免内容扩展完全依赖人工；内容运营上以常青材料为主体，时效性主题阅读作为补充和引流，避免素材库快速老化；
- 预留与写作模块、学习记录模块和后续推荐能力的联动空间；
- 阅读数据默认私有，支持后续删除、匿名化、多端同步和离线提交规划；离线同步冲突优先智能合并阅读位置、生词新增、高亮和辅助展开状态，并用实体 ID、版本号、操作类型和时间戳处理新增/修改/删除冲突，无法合并时再请求用户确认；删除记录不再用于个性化推荐；
- 阅读激励应支持轻量容错，补签卡需绑定真实学习行为、有效期和持有上限，避免连续记录因偶发中断而造成挫败或被滥用；补签卡获取建议明确为满足以下任一条件即可：连续 3 天完成阅读，或完成 1 次精读；当用户连续答错、频繁查词、中断回归或重读低分文章时，应优先提供策略提示和情感化鼓励，保护学习心流；沉默用户唤醒可结合历史主题偏好低频推荐内容，并提供关闭入口。

早期版本应至少完成“冷启动选材 + 主题偏好 + 分级文章 + 重点词辅助 + 阅读策略提示/模式 + 三类基础题型 + 解析反馈/纠错入口 + 有效阅读记录 + 断点续读 + 阅读后自评”，不宜只实现文章展示和二元对错判断。

#### 4.1.3 笔友写信练习模块

笔友写信练习模块属于输出型训练，目标是帮助用户练习英文表达。主要功能包括：

- 写信主题列表；
- 写作任务说明；
- 英文信件编辑器；
- 写作提示，例如称呼、开头、正文、结尾；
- 信件保存；
- 历史写作记录查看。

该模块可以模拟笔友交流场景，但早期不一定需要实现真实用户之间的通信。对于免费平台来说，先提供“主题驱动的写作练习”更容易控制范围，也更适合保证稳定性。

#### 4.1.4 学习记录模块

学习记录模块用于向用户反馈学习成果。基础指标可以包括：

- 已学习单词数量；
- 已掌握单词数量；
- 已完成阅读数量；
- 已完成写作次数；
- 最近学习时间；
- 连续学习天数；
- 今日任务完成情况。

该模块的重点是清晰反馈，不应在早期堆砌过多复杂图表。

### 4.2 非功能需求

非功能需求决定平台是否稳定、好用和易维护。

| 类型 | 要求 |
|---|---|
| 易用性 | 页面入口清晰，用户能快速进入学习任务 |
| 响应式设计 | 适配常见桌面端和平板浏览器，兼顾移动端浏览 |
| 稳定性 | 核心学习流程不能频繁出错，数据保存应可靠 |
| 可扩展性 | 模块边界清晰，后续可以增加听力、口语、学习报告等功能 |
| 性能 | 页面加载速度较快，学习过程不卡顿 |
| 可维护性 | 代码结构清晰，命名规范，逻辑易于定位 |
| 安全性 | 用户学习记录和写作内容应妥善处理 |
| 可测试性 | 核心逻辑和关键流程可以通过自动化测试验证 |

### 4.3 需求优先级控制

可以使用“四级优先级”控制范围：

| 优先级 | 含义 | 示例 |
|---|---|---|
| 必须有 | 没有就无法形成核心学习平台 | 单词学习、阅读练习、写信练习 |
| 应该有 | 对体验和学习效果有明显帮助 | 学习记录、错词记录、基础统计 |
| 可以有 | 有价值但不影响首版上线 | 成就徽章、主题切换、更多图表 |
| 暂不做 | 容易扩大范围或偏离目标 | 商城、付费系统、大型社区、复杂社交 |

## 5. 第三阶段：产品与用户体验设计

### 5.1 信息架构设计

平台页面结构应简洁清晰。推荐的基础页面包括：

```text
首页
├── 单词记忆
│   ├── 单词列表
│   ├── 单词学习
│   └── 单词复习
├── 阅读练习
│   ├── 阅读列表
│   └── 阅读详情与答题
├── 笔友写信
│   ├── 写信主题
│   ├── 写信编辑
│   └── 历史信件
└── 学习记录
    ├── 今日学习
    └── 总体进度
```

首页应突出三个核心入口，让用户进入网站后能够立即知道可以做什么。

### 5.2 核心学习路径设计

可以设计一个“今日学习任务”流程：

```text
进入首页
  -> 查看今日任务
  -> 学习 10 个单词
  -> 完成 1 篇阅读
  -> 完成 1 次写信练习
  -> 查看今日完成情况
```

这种设计可以降低用户选择成本，使用户更容易坚持学习。

### 5.3 交互反馈设计

学习平台应提供及时反馈：

- 单词记忆后显示掌握状态变化；
- 阅读答题后显示正确与错误；
- 写信保存后提示保存成功；
- 学习记录页面展示完成情况；
- 出错时提供明确错误说明，而不是只显示系统异常。

反馈的目的不是制造复杂动画，而是让用户明确知道“我完成了什么”和“下一步可以做什么”。

### 5.4 页面体验原则

为了提高用户体验，页面设计应遵循：

- 操作路径尽量短；
- 主按钮位置统一；
- 表单字段尽量少；
- 不频繁弹窗打断学习；
- 颜色和字体保持一致；
- 学习进度清晰可见；
- 移动端浏览时主要内容不被遮挡；
- 空数据、加载中、错误状态都有明确提示。

## 6. 第四阶段：技术架构设计

> 第四阶段详细落地文档：[`docs/phase-4-technical-architecture-design.md`](phase-4-technical-architecture-design.md)。本节保留开发流程规划中的概要说明，后续技术评审、接口设计、数据库设计、部署与运维细节以第四阶段文档为准。

### 6.1 前端架构

Web 端可以采用组件化前端框架进行开发，例如 React、Vue 或类似技术栈。对于 TypeScript 项目，建议使用类型定义约束数据结构，减少后期维护成本。

前端目录可以按以下方式组织：

```text
src/
├── components/       # 通用组件
├── features/         # 业务模块
│   ├── vocabulary/   # 单词记忆
│   ├── reading/      # 阅读练习
│   ├── writing/      # 笔友写信
│   └── progress/     # 学习记录
├── services/         # 接口请求与数据服务
├── types/            # 公共类型定义
├── utils/            # 工具函数
├── styles/           # 全局样式
└── App.tsx           # 应用入口
```

对于当前规模的项目，不建议一开始就引入过重的架构。更合理的做法是保证目录清晰、模块边界明确、公共逻辑可复用。

### 6.2 后端与数据设计

如果平台需要保存用户学习记录，后端应至少提供以下能力：

- 用户身份或匿名学习档案；
- 单词数据读取；
- 阅读材料读取；
- 写作记录保存；
- 学习进度保存；
- 数据统计接口。

对于免费平台，可以先采用较轻量的后端方案。如果后续用户量增加，再逐步扩展缓存、搜索、队列和监控等能力。

### 6.3 数据模型设计

核心数据对象可以包括：

| 数据对象 | 主要字段 |
|---|---|
| Word | id、单词、释义、音标、例句、难度、分类 |
| WordProgress | 用户标识、单词 id、掌握状态、复习次数、最近学习时间 |
| ReadingArticle | id、标题、内容、难度、重点词、题目 |
| ReadingRecord | 用户标识、文章 id、答题结果、完成时间 |
| LetterTopic | id、主题、提示、难度 |
| LetterRecord | 用户标识、主题 id、信件内容、创建时间 |
| LearningSummary | 用户标识、学习天数、单词数、阅读数、写作数 |

数据模型设计时应避免把所有信息混在一个对象中。单词、阅读、写作和学习记录应各自独立，这样后续扩展时更容易维护。

### 6.4 模块边界设计

推荐将平台拆分为以下模块：

| 模块 | 职责 |
|---|---|
| 用户模块 | 用户信息、学习档案、登录状态 |
| 单词模块 | 单词展示、记忆状态、复习逻辑 |
| 阅读模块 | 文章展示、阅读题目、答题反馈 |
| 写作模块 | 写信主题、信件编辑、历史记录 |
| 统计模块 | 学习数据汇总与展示 |
| 组件模块 | 按钮、卡片、弹窗、表单、进度条 |
| 服务模块 | 接口请求、错误处理、数据转换 |

模块化设计的目标是：新增功能时尽量只影响相关模块，不让一个小修改牵动整个系统。

## 7. 第五阶段：组件库建设与复用

### 7.1 通用组件规划

平台可以建立轻量级组件库，包含：

| 组件 | 用途 |
|---|---|
| Button | 统一按钮样式和状态 |
| Card | 单词卡片、文章卡片、写作主题卡片 |
| Modal | 确认提示和重要信息展示 |
| ProgressBar | 学习进度展示 |
| Tag | 难度、分类、状态标签 |
| Input / Textarea | 表单和写作输入 |
| Navbar | 页面导航 |
| EmptyState | 无数据状态提示 |
| Loading | 加载状态 |
| ErrorMessage | 错误提示 |

### 7.2 组件复用策略

组件设计应遵循以下原则：

- 只封装重复出现的 UI 和交互；
- 组件参数保持简单；
- 不为了复用而过度抽象；
- 样式统一由组件内部或统一样式规范控制；
- 业务逻辑不要过多塞进通用组件。

例如，单词卡片、文章卡片、写作主题卡片可以复用同一个基础 `Card` 组件，但具体内容仍由各业务模块控制。

### 7.3 避免重复开发

组件库可以减少以下重复任务：

- 重复编写按钮样式；
- 重复处理加载和错误状态；
- 重复设计卡片布局；
- 重复实现弹窗和提示；
- 重复编写表单输入样式；
- 重复处理空数据页面。

## 8. 第六阶段：开发实施流程

### 8.1 迭代式开发

第六阶段采用“后端完全先行、组件库先冻结、AI Studio Build 一次性生成前端页面、生成后自动化检查与回溯式修复”的实施方式。前端不再按照单个业务模块逐页开发，也不得在接口未冻结前基于 Mock 数据推进业务页面。所有业务页面必须在后端接口交付、`src/types/api.ts` 冻结、组件库与 AI 上下文包准备完成后，由 Google AI Studio Build 一次性生成。

#### 第一轮：基础框架与全局基础设施

**前置依赖：**

- 已确认 EnglishLover Web 端的页面范围、路由结构和基础技术栈；
- 已确认组件库建设范围和设计令牌命名规则；
- 后端接口仍处于设计与开发阶段，前端不得开发依赖真实接口的业务页面；
- 本轮仅允许建设框架、路由空壳、基础组件和类型占位，不允许引入 Mock 数据驱动业务页面。

**本轮目标：**

建立前端工程的基础骨架，为后续组件库沉淀、AI Studio Build 一次性生成页面和真实接口接入提供稳定基础。

**前端任务：**

- 初始化 Web 端项目结构，明确 `src/pages/`、`src/features/`、`src/components/`、`src/types/`、`src/lib/`、`src/styles/` 等目录职责；
- 配置 Tailwind CSS、全局样式入口和设计令牌映射，统一颜色、间距、圆角、字号、阴影和响应式断点；
- 实现 P0 基础组件，包括 `Button`、`Card`、`Input`、`Textarea`、`Select`、`Modal`、`Loading`、`Badge`、`Tabs`、`Toast` 等；
- 实现全局布局与导航组件，包括页面外框、侧边栏或顶部导航、主内容区域、移动端导航占位等；
- 建立空壳路由，仅允许展示占位页面或“功能建设中”提示，不允许接入伪造业务数据；
- 建立 `src/types/api.ts` 类型占位文件，明确该文件后续必须由后端契约自动生成并冻结，前端不得手写真实接口类型；
- 建立基础代码规范检查，如 ESLint、TypeScript、Prettier、Tailwind class 约束等；
- 本阶段不使用 Google AI Studio Build 生成业务页面，仅为后续 AI 一次性生成准备工程边界。

**后端任务：**

- 并行完成接口文档冻结前的设计工作；
- 明确接口命名、错误码、枚举字典、分页结构、认证方式和响应包裹格式；
- 开始接口编码和契约测试用例编写。

**风险点：**

- 若第一轮组件边界过弱，后续 AI Studio 可能生成重复 UI 或硬编码样式；
- 若设计令牌命名不稳定，后续页面生成和回溯式修复成本会增加；
- 若前端提前开发依赖接口的业务页面，会破坏“后端完全先行”的流程约束。

**交付标准：**

- 项目可正常启动、类型检查通过、基础构建通过；
- P0 基础组件可在组件示例页中独立展示；
- 全局布局、导航和空壳路由可访问；
- `src/types/api.ts` 仅保留占位说明，不包含手写业务接口定义；
- 未出现 Mock 数据驱动的业务页面；
- 后端接口设计进入冻结前评审状态。

#### 第二轮：反馈与状态组件 + 后端接口交付验收

**前置依赖：**

- 第一轮基础框架、设计令牌、P0 基础组件和空壳路由已完成；
- 后端接口设计已基本稳定，进入最终开发、契约测试和交付验收阶段；
- 前端仍不得开发依赖接口的业务页面。

**本轮目标：**

完成所有反馈、状态和数据展示类组件，并形成 AI 可消费的组件 API 摘要。同时，后端必须完成全部接口开发、契约测试和 `src/types/api.ts` 自动生成，为第三轮 AI Studio 一次性生成页面解除关键阻塞。

**前端任务：**

- 完成反馈与状态组件，包括 `EmptyState`、`ErrorState`、`SaveStatus`、`FeedbackPanel`、`ProgressIndicator`、`Skeleton`、`RetryPanel`、`InlineAlert`、`ConfirmDialog`、`StepIndicator` 等；
- 完成数据展示组件，包括 `StatCard`、`DataList`、`Timeline`、`ProgressChartContainer`、`SummaryPanel`、`TagList` 等；
- 为每个组件编写 AI 可消费 API 摘要，建议维护在 `src/docs/ai-context/COMPONENT_CATALOG.md` 或同类目录中；
- 每个组件摘要必须精简、稳定、可直接供 AI Studio 使用，仅包含组件名、文件路径、Props 签名、一句话使用场景、一句话反模式；
- 组件负责人负责编写摘要，前端架构负责人审核摘要是否准确，测试负责人补充必要的状态覆盖说明；
- 建立组件示例页或 Story 文档，仅允许使用组件库示例数据，不得伪造业务接口数据；
- 建立初版组件合规检查规则，例如禁止页面层自定义 Button、Card、Modal，禁止硬编码颜色和内联样式。

**后端任务：**

- 完成全部业务接口开发；
- 完成接口契约测试，契约测试通过率必须达到 100%；
- 部署稳定测试环境，保证前端可通过固定测试环境访问真实接口；
- 自动化生成并冻结 `src/types/api.ts`；
- 输出《接口交付测试报告》，内容至少包括接口清单、契约测试结果、错误码清单、枚举字段说明、测试环境地址、`src/types/api.ts` 生成方式与版本号、已知限制和后续变更流程；
- 冻结接口后，任何字段、枚举、错误码、响应结构变更必须走变更评审，不得直接修改。

**风险点：**

- 后端接口交付延迟会直接阻塞第三轮 AI Studio 一次性生成；
- 若 `api.ts` 未由后端契约自动生成，前端类型可能与真实接口不一致；
- 若组件 API 摘要不准确，AI Studio 可能误用组件或生成重复 UI。

**交付标准：**

- 所有反馈、状态和数据展示组件完成并通过组件级检查；
- 每个组件均有 AI 可消费 API 摘要；
- 后端完成全部接口开发、契约测试 100% 通过，并交付《接口交付测试报告》；
- `src/types/api.ts` 已自动生成、冻结并提交到前端仓库；
- 前端未开始任何依赖接口的业务页面开发；
- 第二轮结束标志为：后端正式交付，前端状态组件齐全且文档化。

#### 第三轮：业务组件冻结 + 准备 AI Studio 上下文 + 一次性生成全部页面

第三轮是第六阶段的关键轮次，也是整个流程的关键路径。只有在后端接口完全交付、`src/types/api.ts` 冻结、组件库文档齐全后，才允许进入本轮。前端页面不按模块拆分开发，而是通过 Google AI Studio Build 一次性生成全部业务页面，再通过自动化合规检查和回溯式修复达到质量标准。

##### 阶段 3A：业务组件冻结

**前置依赖：**

- 第二轮已完成；
- 后端已交付冻结版 `src/types/api.ts`；
- P0 基础组件、反馈状态组件、数据展示组件均已完成并有 API 摘要。

**本阶段目标：**

基于冻结的真实接口类型完成所有业务组件，并确保业务组件成为 AI Studio 生成页面时唯一可用的业务 UI 组合单元。

**前端任务：**

- 基于 `src/types/api.ts` 完成所有业务组件，包括 `TaskActionCard`、`PracticeQuestion`、`VocabularyReviewPanel`、`ReadingArticlePanel`、`ReadingQuestionPanel`、`LetterEditor`、`SafetyNotice`、`LearningProgressSummary`、`ResultSummary`、`TodayTaskList` 等；
- 所有业务组件的 Props 类型必须直接引用 `src/types/api.ts` 中的接口、响应类型、枚举或字段类型；
- 严禁在业务组件中重复定义接口字段、枚举值或响应结构；
- 对存在组合关系的组件补充使用约束，例如 `LetterEditor` 必须与 `SaveStatus` 组合使用，阅读题结果必须通过 `FeedbackPanel` 展示，空列表必须使用 `EmptyState`，错误恢复必须使用 `ErrorState` 或 `RetryPanel`，安全提醒必须使用 `SafetyNotice`；
- 更新业务组件的 AI 可消费 API 摘要；
- 组件负责人负责编写业务组件摘要，前端架构负责人审核类型来源和组件边界，后端负责人确认字段含义与 `api.ts` 一致。

**后端任务：**

- 不新增接口开发；
- 仅响应接口解释、字段含义确认和交付快照核验；
- 若发现 `api.ts` 与《接口交付测试报告》不一致，必须先修正后端契约并重新冻结，不允许前端临时兼容。

**交付标准：**

- 所有业务组件完成；
- 所有业务组件 Props 类型均来自 `src/types/api.ts`；
- 业务组件 API 摘要已更新并通过审核；
- 不存在重复定义接口类型、枚举或业务字段的情况；
- 组件库达到可供 AI Studio Build 使用的冻结状态。

##### 阶段 3B：构筑 AI Studio 生成上下文包

**前置依赖：**

- 阶段 3A 已完成；
- 组件库 API 摘要、设计令牌和接口类型均已稳定；
- 已明确 Google AI Studio Build 的上下文窗口限制，不能直接上传整个组件库源码。

**本阶段目标：**

构建精炼但完整的 AI Studio 输入包，让 AI 在有限上下文内准确理解组件复用规则、接口类型、页面结构和禁止事项。

**前端任务：**

准备并审核以下上下文文件，统一存放在 `src/docs/ai-context/` 或项目约定目录下，并形成版本快照：

1. `SYSTEM_INSTRUCTION.md`

   - 由前端架构负责人编写，项目负责人审核；
   - 内容用于设置 Google AI Studio Build 的 System Instruction；
   - 必须明确不可违背规则：只能使用 `src/components/` 中已有组件，禁止重复实现 Button、Card、Input、Modal、Loading、EmptyState、ErrorState 等组件，禁止硬编码颜色、字号、间距和阴影，禁止内联样式，所有接口字段必须来自 `src/types/api.ts`，所有页面必须使用统一加载态、空态、错误态、保存态和反馈态，不得生成 Mock 数据，不得自行创造接口字段，不得绕过组件库直接写复杂 UI 结构。

2. `COMPONENT_CATALOG.md`

   - 由组件负责人编写，前端架构负责人审核；
   - 覆盖所有 P0/P1 组件；
   - 每个组件仅包含名称、文件路径、Props 签名、一句话适用场景、一句话反模式；
   - 不上传完整组件源码，避免浪费上下文窗口并降低 AI 误读风险。

3. `API_TYPES_REFERENCE.md`

   - 由前端负责人从冻结版 `src/types/api.ts` 中提取；
   - 后端负责人审核字段含义；
   - 只保留页面消费所需的核心接口响应类型、请求类型、枚举和错误结构；
   - 不直接上传全量原始类型文件，避免上下文膨胀。

4. `DESIGN_TOKENS_QUICKREF.md`

   - 由设计或前端负责人编写；
   - 列出颜色、间距、字号、圆角、阴影、断点等 CSS 变量名和允许使用的 Tailwind class 范围；
   - 明确禁止使用的写法，例如任意 HEX 色值、任意像素值、内联 style。

5. `PAGE_COMPONENT_MAP.md`

   - 由前端架构负责人编写，产品或 UX 负责人审核；
   - 以表格列出目标页面与强制组件映射：`TodayPage`、`VocabularyPractice`、`ReadingPractice`、`PenPalEditor`、`ResultsPage`、`ErrorPage`；
   - 每个页面必须列出强制使用的组件、页面区域与组件对应关系、哪些部分绝不可自行实现、必须覆盖的加载、空、错、保存和反馈状态。

6. `API_FIELD_EXAMPLES.md`

   - 由后端负责人从接口交付快照中提取真实响应样例；
   - 前端负责人审核是否覆盖页面消费字段；
   - 每个接口提供一个真实响应样例，用于展示字段名、嵌套结构、枚举值和空值形态；
   - 样例仅用于理解字段结构，不得作为前端业务页面的 Mock 数据源。

**后端任务：**

- 提供真实接口响应样例和字段解释；
- 确认 `API_FIELD_EXAMPLES.md` 与测试环境真实响应一致；
- 确认所有样例不包含敏感数据。

**风险点：**

- 上下文包过大将降低 AI Studio Build 的生成稳定性；
- 上下文包过少会导致 AI 自行补全字段、组件或样式；
- 若 `PAGE_COMPONENT_MAP.md` 不明确，AI 可能按页面结构自行实现 UI，破坏组件库强制复用要求。

**交付标准：**

- AI Studio 输入包六类文件全部完成；
- 每份文件均有明确负责人和审核人；
- 上下文包总大小控制在 AI Studio Build 可稳定处理的范围内；
- 上下文包完成版本快照，例如 `ai-context-v1.0-m3.zip` 或对应 Git 标签；
- 上下文包不得包含完整组件库源码，不得包含 Mock 数据文件。

##### 阶段 3C：执行 AI Studio 一次性生成

**前置依赖：**

- 阶段 3A、3B 已完成；
- 组件库冻结；
- `src/types/api.ts` 冻结；
- AI Studio 输入包通过审核；
- 自动化合规检查脚本已准备完成。

**本阶段目标：**

使用 Google AI Studio Build 一次性生成全部业务页面代码，并通过自动化检查与回溯式修复，确保生成结果符合组件复用、类型安全和接口契约要求。

**前端任务：**

- 在 Google AI Studio Build 中创建新项目；
- 上传已审核的 AI Studio 输入包，包括 `SYSTEM_INSTRUCTION.md`、`COMPONENT_CATALOG.md`、`API_TYPES_REFERENCE.md`、`DESIGN_TOKENS_QUICKREF.md`、`PAGE_COMPONENT_MAP.md`、`API_FIELD_EXAMPLES.md`；
- 将 `SYSTEM_INSTRUCTION.md` 的内容设置为 System Instruction；
- 提交一次性生成指令，例如：

  ```text
  请根据提供的上下文，一次性生成 EnglishLover Web 端所有页面的完整代码。
  必须严格遵守 System Instruction、组件目录、接口类型、设计令牌和页面-组件映射表。
  生成范围包括 src/pages/ 与 src/features/ 下的页面和页面级组合逻辑。
  禁止生成 Mock 数据，禁止重复实现组件，禁止硬编码样式，禁止创造 api.ts 中不存在的字段。
  ```

- 接收 AI Studio 输出的 `src/pages/` 和 `src/features/` 下全部文件；
- 将生成代码拉回本地仓库，运行自动化合规检查脚本，至少检查：是否出现硬编码颜色、内联样式、自定义按钮/卡片/输入框/弹窗，是否导入了 `src/components/` 以外的自建 UI，是否出现未登记组件，是否使用了 `api.ts` 中不存在的字段名，是否重复定义接口类型或枚举，是否绕过统一加载态/空态/错误态/反馈态，是否引入 Mock 数据，TypeScript 是否编译通过；
- 若检查不通过，将违规代码片段、检查报告和明确修复指令重新提交给 AI Studio，要求其进行回溯式重构；
- 回溯式修复提示必须聚焦违规点，例如“将自定义按钮替换为 `Button` 组件”“移除硬编码颜色，改用设计令牌”“字段 `xxx` 不存在于 `api.ts`，请改为 `yyy`”“错误态必须使用 `ErrorState`，不得自行实现错误卡片”；
- 重复执行“检查—反馈—AI Studio 修复—本地验证”，直至合规检查、类型检查和构建全部通过。

**后端任务：**

- 不参与页面生成；
- 保持测试环境稳定；
- 对字段解释、错误码语义和真实响应异常进行确认；
- 若发现接口契约缺陷，必须重新走接口变更流程，不能让前端通过临时字段兼容。

**风险点：**

- AI Studio 可能生成结构正确但组件复用不合规的代码；
- AI Studio 可能根据样例推断出不存在的字段；
- 自动化检查脚本覆盖不足时，人工审查可能遗漏硬编码样式或重复 UI；
- M3 是关键路径瓶颈，任何接口交付延迟、组件库缺失或上下文包不完整都会阻塞本阶段。

**交付标准：**

- `src/pages/` 和 `src/features/` 下全部目标页面已由 AI Studio Build 一次性生成；
- 所有页面通过组件复用合规检查；
- 所有页面通过设计令牌合规检查；
- 所有页面字段名与 `src/types/api.ts` 完全匹配；
- 不存在 Mock 数据；
- TypeScript 编译零错误；
- 本地构建通过；
- AI Studio 生成记录、修复提示、检查报告和最终生成版本均已归档。

总体依赖关系为：

```text
M1 基础工程
  → M2 后端接口交付 + 状态组件齐备
```

其中，M3 不仅依赖前端组件库成熟度，也依赖后端接口冻结质量和 AI Studio 上下文包质量。M3 之前不得开展依赖接口的业务页面开发；M3 之后的页面修复必须以自动化检查结果、真实接口行为和组件库约束为依据，不得回退到按页面人工重写或使用 Mock 数据推进的旧流程。

## 9. 第七阶段：自动化测试设计

### 9.1 测试目标

自动化测试的目标不是追求测试数量，而是保证核心学习流程稳定。对于该平台，应重点覆盖：

- 单词记忆状态是否正确更新；
- 阅读答题结果是否正确计算；
- 写信内容是否能够正常保存；
- 学习记录是否能够正确统计；
- 关键页面是否能够正常渲染。

### 9.2 单元测试

单元测试适合验证独立函数和基础逻辑，例如：

- 学习进度计算；
- 单词掌握状态更新；
- 阅读题目评分；
- 写作内容校验；
- 日期和连续学习天数计算。

### 9.3 组件测试

组件测试适合验证页面组件的显示和交互，例如：

- 单词卡片是否显示释义和例句；
- 阅读题目选择后是否更新状态；
- 写信输入框是否能正常输入；
- 提交按钮在内容为空时是否禁用；
- 加载、错误和空状态是否正确显示，空状态应包含友好文案和下一步引导；阅读高亮、状态提示等视觉反馈不应只依赖颜色。

### 9.4 端到端测试

端到端测试模拟真实用户流程，建议覆盖最关键路径：

```text
进入首页
  -> 进入单词记忆
  -> 完成一次单词学习
  -> 进入阅读练习
  -> 完成一篇阅读题目
  -> 进入笔友写信
  -> 保存一封练习信件
  -> 查看学习记录
```

端到端测试不需要覆盖所有细节，但必须覆盖最重要的学习路径。

## 10. 第八阶段：CI/CD 与自动化部署

### 10.1 CI/CD 目标

CI/CD 的核心价值是减少重复人工操作，让代码提交后自动完成检查、测试、构建和部署。这样可以降低人为疏漏，提升项目稳定性。

### 10.2 推荐流程

一次标准 CI/CD 流程可以设计为：

```text
提交代码
  -> 安装依赖
  -> 代码格式检查
  -> 类型检查
  -> 自动化测试
  -> 项目构建
  -> 部署到测试环境
  -> 人工确认
  -> 发布到正式环境
```

对于个人毕业设计或小型项目，可以先实现：

- 自动安装依赖；
- 自动类型检查；
- 自动构建；
- 部署到测试环境或静态托管平台。

后续再逐步增加自动化测试、预览环境和正式发布审批。

### 10.3 部署注意事项

部署时应注意：

- 不提交真实密钥；
- 环境变量单独配置；
- 构建产物和源代码分离；
- 保留回滚方案；
- 记录部署步骤；
- 上线后进行基本功能检查。

## 11. 第九阶段：标准化工具与协作流程

### 11.1 代码规范

项目应统一：

- 文件命名规则；
- 组件命名规则；
- 类型命名规则；
- 目录结构；
- 代码格式；
- 接口返回格式；
- 错误处理方式；
- 提交信息格式。

统一规范可以减少沟通成本，也能降低后期维护难度。

### 11.2 分支管理

可以采用简单分支策略：

| 分支 | 用途 |
|---|---|
| main | 稳定版本 |
| develop | 开发版本 |
| feature/xxx | 新功能开发 |
| fix/xxx | 问题修复 |

对于个人项目，也可以简化为 `main + feature` 模式，但应避免直接在稳定版本上做大量未验证修改。

### 11.3 文档管理

项目文档应至少包括：

- 需求说明文档；
- 系统设计文档；
- 页面结构说明；
- 数据结构说明；
- 接口说明；
- 测试说明；
- 部署说明；
- 迭代记录。

文档不需要过度复杂，但必须能说明系统为什么这样设计、如何运行、如何维护。

## 12. 第十阶段：后期迭代与维护

### 12.1 维护内容

项目上线后，需要持续维护：

- 修复页面错误；
- 优化加载速度；
- 更新学习内容；
- 改进交互体验；
- 增加测试覆盖；
- 检查依赖安全性；
- 优化部署流程；
- 根据反馈调整学习任务。

### 12.2 迭代原则

后续迭代仍应围绕核心学习目标展开。可以优先考虑：

- 更合理的单词复习计划；
- 更丰富的阅读材料；
- 更清晰的写作反馈；
- 更直观的学习统计；
- 更好的移动端体验；
- 更稳定的数据保存方式。

不建议为了功能数量而盲目增加无关模块。

### 12.3 反馈闭环

平台应建立简单反馈闭环：

```text
收集用户反馈
  -> 判断是否符合核心学习目标
  -> 评估开发成本
  -> 排入迭代计划
  -> 开发与测试
  -> 发布
  -> 继续观察效果
```

这样可以避免项目偏离用户真实需求。

## 13. 重复性任务规避策略

为了提高开发效率，应系统性减少重复工作。

| 重复任务 | 规避方式 |
|---|---|
| 重复写 UI 样式 | 建立通用组件库 |
| 重复写请求逻辑 | 封装接口服务层 |
| 重复定义数据结构 | 统一 TypeScript 类型或接口模型 |
| 重复手动检查格式 | 使用格式化和代码检查工具 |
| 重复手动测试核心流程 | 使用自动化测试 |
| 重复手动构建部署 | 使用 CI/CD |
| 重复编写相似页面 | 抽象页面布局和业务模板 |
| 重复排查问题 | 建立日志、错误提示和文档记录 |

## 14. 项目范围控制方法

项目范围控制可以从以下方面执行：

1. **明确核心目标**  
   平台核心目标是英语学习，不是社交平台、商城平台或内容社区。

2. **区分必要功能与可选功能**  
   单词、阅读、写信是必要功能；排行榜、社区、复杂推荐是可选功能。

3. **使用优先级列表管理需求**  
   每个需求进入开发前，都应标记优先级和验收标准。

4. **每个迭代只解决一个重点问题**  
   避免同时修改多个模块，降低引入新问题的风险。

5. **用学习效果判断功能价值**  
   如果一个功能不能帮助用户更好地记忆、阅读或写作，就不应优先开发。

6. **保留扩展空间，但不提前实现复杂逻辑**  
   架构上可以预留扩展点，但不要过早实现暂时用不到的复杂功能。

## 16. 总结

一个高质量的免费在线英语学习平台，应从需求分析阶段就围绕用户学习目标展开设计。单词记忆、阅读练习和笔友写信练习分别对应词汇积累、语言输入和语言输出，是平台最重要的核心功能。

在技术实现上，平台应通过模块化架构、组件库复用、统一代码规范、自动化测试和 CI/CD 流程减少重复劳动，提高开发效率和系统稳定性。在项目管理上，应采用迭代式推进方式，先完成核心功能，再逐步优化体验和扩展能力。后续迭代还应关注学习档案可携带性、沉浸阅读、无障碍设计、主动内容质量监控、内容健康度运营、能力叙事和情感化回归体验，使平台不仅能完成练习流程，也能长期陪伴用户学习。

只有坚持核心目标清晰、范围控制明确、流程标准化、验证自动化，才能保证项目从初期需求分析到后期迭代维护都能高效、流畅地推进，并避免偏离用户真实需求。
